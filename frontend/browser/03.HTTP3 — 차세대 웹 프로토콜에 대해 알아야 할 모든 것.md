# HTTP/3 — 차세대 웹 프로토콜에 대해 알아야 할 모든 것

> **출처**: https://velog.io/@sehyunny/everthing-you-need-to-know-about-http3 (velog @sehyunny, 황세현 번역) / 원문: https://portswigger.net/daily-swig/http-3-everything-you-need-to-know-about-the-next-generation-web-protocol

`HTTP/2`까지는 그래도 "TCP 위에서 멀티플렉싱한다" 정도로 알고 있었는데, `HTTP/3`는 아예 **TCP를 버리고 UDP로 갔다**는 얘기를 듣고 "엥 그게 가능해?" 싶어서 정리했다. 결론부터 말하면 `QUIC`이라는 프로토콜이 그걸 가능하게 만든다.

### HTTP/3란?

HTTP/3는 웹에서 정보를 주고받는 `Hypertext Transfer Protocol`의 메이저 개정판이다. **`QUIC` 위에서 동작**하는데, QUIC은 단일 연결에서 여러 데이터 스트림을 멀티플렉싱하는 **암호화된 범용 전송 계층 프로토콜**이다. 구글이 처음 만들었고 **UDP 기반**이다.

### HTTP, HTTP/2, HTTP/3는 어떤 관계?

한 줄로 정리하면 이렇다.

| 버전 | 전송 계층 |
|---|---|
| HTTP | TCP 기반 |
| HTTP/2 | TCP 기반 (+ 멀티플렉싱) |
| HTTP/3 | **QUIC(=UDP) 기반** |

> HTTP/3는 "**QUIC 위에서 도는 HTTP/2**"라고 설명하는 게 제일 정확하다.

![HTTP/3를 가장 잘 묘사하는 표현](https://velog.velcdn.com/images%2Fsehyunny%2Fpost%2F6f1e0637-2e93-4940-b353-879271ade71b%2F2.png)

### HTTP/3 프로토콜의 이점은?

#### 1) HTTP/2의 고질병 — Head of Line Blocking 해결

이게 제일 핵심이다. HTTP/2는 TCP 위에서 여러 스트림을 멀티플렉싱하는데, **TCP의 손실 회복 메커니즘이 이 병렬성을 근본적으로 지원하지 못한다.**

- ❌ **HTTP/2 (TCP)** — 패킷 하나 유실되면 **모든 트랜잭션(스트림)이 다 멈춰서** 그 패킷 재전송을 기다린다. (Head of Line Blocking)
- ✅ **HTTP/3 (QUIC)** — 멀티플렉싱을 **기본적으로** 지원해서, 유실된 패킷은 **해당 스트림에만** 영향을 준다. 나머지 스트림은 계속 흐른다.

#### 2) 보안 강화 (거의 전부 암호화)

QUIC은 거의 완전히 암호화돼 있어서 보안이 크게 올라가고, **중간자 공격(MitM)** 노출 가능성이 낮아진다.

#### 3) 핸드셰이크가 빠르다

QUIC은 **`TLS 1.3`을 암호화 핸드셰이크의 구성 요소로** 쓴다. 그래서

- 새 서버와 **단일 왕복(1-RTT) 안에** 연결 가능
- 클라이언트가 **최초 연결 때부터 암호화된 애플리케이션 데이터를 송신**할 수 있고
- 끊긴 연결을 **신속하게 재개**할 수 있다.

### HTTP/3 지원 현황은?

- 2020년 10월 기준 인터넷 표준 **초안(draft)** 단계, 여러 구현체 존재.
- W3Techs 통계: 상위 1000만 사이트 중 거의 8% 지원 → **2022년 3월 기준 약 25%** 까지 올라옴.
- 브라우저: **Chrome(2019.12~)**, **Firefox(2020.1~)** 지원.

![HTTP/3는 더 빠른 로딩과 더 나은 성능을 제공](https://velog.velcdn.com/images%2Fsehyunny%2Fpost%2Fb81dbb28-61f7-4313-b30b-01ab67f2126d%2F3.png)

### 확산되면 뭐가 좋아지나? (단, 과장은 금물)

Cloudflare 쪽은 "HTTP/3가 모두에게 더 나은 인터넷을 줄 것"이라고 했다. 특히 **패킷 유실이 많은 저품질 네트워크**에서 빠른 연결 설정 + 더 나은 성능을 얻는다.

근데 여기서 균형 잡힌 시각이 중요하다. Hasselt 대학 연구원 **Robin Marx**는 좀 더 신중하다.

> "성능 이점은 있지만, **Head of Line Blocking 제거는 막상 웹 페이지 로딩에서 그렇게 중요하지 않다.**"
> HTTP/3와 QUIC은 "**혁명이 아니라 진화(evolution, not revolution)**"이고, **대부분의 이점은 핸드셰이크 설정 시간 단축에 있다.**

즉 "HOL 해결!"이 마케팅 포인트지만, 실제 체감 이득의 핵심은 **빠른 핸드셰이크** 쪽이라는 것.

### 배포(deployment)의 과제

QUIC이 UDP 기반이라 기존 인프라랑 안 맞는 지점이 많다.

- **로드 밸런서** — QUIC의 **연결 ID(connection ID)·연결 마이그레이션** 기능을 처리하도록 손봐야 함.
- **미들박스(심층 패킷 검사 장비, DPI)** — 일부 네트워크는 **의도적으로 QUIC을 차단**한다. 방화벽이 접속 설정이나 QUIC 전송 헤더를 추적 못 하기 때문.
- **브라우저 지원** — 아직 완전 주류는 아님. Chrome은 최근 95%에서 활성화, Firefox·Safari는 도입 초기.
- **암호화된 헤더** — 패킷 번호·ACK·옵션이 전부 QUIC으로 암호화 → 브라우저가 **자체 서명 인증서랑은 제대로 안 맞는다.**
- **서버 구현** — 직접 QUIC 켜는 건 복잡함. 현실적으로 **CDN이나 외부 서비스 제공자 쓸 때만** 쉽게 활용 가능.

### 언제 주류가 될까?

QUIC은 TCP가 아니라 UDP 기반이라, **HTTP/2를 그냥 QUIC으로 업그레이드할 수 없다.** Marx는 이렇게 본다.

> "앞으로 몇 년간 HTTP/3는 주로 **CDN이나 외부 서비스 제공자를 쓸 때** 얻게 될 것이다."
> "QUIC은 매우 강력하고 TCP보다 **진화·조정이 훨씬 쉬워서** 많은 성능 향상이 나오겠지만, 매우 복잡하고 커뮤니티 전반이 QUIC 동작 방식을 받아들이기까진 시간이 걸린다."

### 📌 정리
`HTTP/3 = QUIC(UDP 기반) 위의 HTTP/2 — 핵심은 HOL 제거가 아니라 빠른 핸드셰이크`

#### 핵심
> HTTP/2까지는 TCP 위였고, 그래서 **패킷 하나 유실되면 멀티플렉싱한 모든 스트림이 멈추는 Head of Line Blocking**이 고질병이었다. HTTP/3는 전송 계층을 **TCP → QUIC(UDP 기반)** 으로 갈아엎어서, 유실 패킷이 **해당 스트림에만** 영향을 주게 만들었다. 추가로 TLS 1.3을 핸드셰이크에 내장해 **1-RTT 연결 + 최초부터 암호화 데이터 전송**이 가능하다. 다만 전문가(Robin Marx)는 "혁명 아닌 진화"라며 **실체감 이득은 HOL 제거보다 핸드셰이크 단축**이라고 본다. UDP 기반이라 로드밸런서·방화벽·서버 구현 같은 배포 장벽이 있어서, 당분간은 **CDN 통해서** 주로 쓰게 된다.

#### ⭐️ 앞으로 기억할 것
- 1. **전송 계층이 바뀐 게 본질**
  HTTP/2 = TCP, HTTP/3 = QUIC(UDP). "QUIC 위의 HTTP/2"로 외우면 됨.
- 2. **TCP의 Head of Line Blocking**
  TCP 손실 회복이 멀티플렉싱 병렬성을 지원 못 해서, 패킷 1개 유실 → 전체 스트림 정지. QUIC은 스트림별 독립이라 해당 스트림만 영향.
- 3. **TLS 1.3 내장 → 빠른 핸드셰이크**
  1-RTT로 새 서버 연결, 최초 연결부터 암호화 데이터 송신, 끊긴 연결 빠른 재개. ← 실제 체감 이득의 핵심.
- 4. **"혁명이 아니라 진화"**
  HOL 제거가 마케팅 포인트지만 웹 로딩에선 생각보다 안 중요. 진짜 이득은 핸드셰이크 시간 단축.
- 5. **UDP라서 배포가 어렵다**
  로드밸런서(connection ID/migration), DPI 방화벽의 의도적 차단, 자체 서명 인증서 이슈, 복잡한 서버 구현 → 그래서 당분간 **CDN 의존**.

📌 결론
▎ HTTP/3는 TCP를 버리고 QUIC(UDP)으로 가서 TCP의 구조적 한계(HOL blocking)를 풀었다.
▎ 단 실제 이득의 무게중심은 HOL 제거보다 **TLS 1.3 기반 빠른 핸드셰이크** 쪽이다.
▎ UDP 기반이라 인프라 호환 장벽이 커서, 현실적으로는 CDN/외부 서비스 통해 점진 확산된다.
▎ ⭐️ 실무 적용 포인트
▎ - 직접 서버에 QUIC 켜기보다 **Cloudflare·CDN의 HTTP/3 옵션을 켜는 게** 가장 쉬운 도입 경로.
▎ - 이득이 가장 큰 케이스: **모바일·패킷 유실 많은 저품질 네트워크** 사용자.
▎ - 사내 방화벽/프록시가 UDP·QUIC을 막고 있으면 HTTP/3로 안 떨어지고 TCP로 폴백되니, 효과 안 보일 때 네트워크 장비부터 의심.
