# 프런트엔드 개발자가 알아야 할 접근성 필수 사항

> **원문**: https://martijnhols.nl/blog/accessibility-essentials-every-front-end-developer-should-know
> **번역 출처**: https://velog.io/@eunbinn/accessibility-essentials-every-front-end-developer-should-know

![](https://velog.velcdn.com/images/chromeheartz/post/d5573515-fd3e-4e6c-a615-ada4f57cb4bf/image.png)

접근성(a11y) 하면 뭔가 거창하고 따로 챙겨야 할 작업처럼 느껴지는데, 기본적인 것만 제대로 적용해도 충분히 큰 차이를 만들 수 있다는 걸 잘 정리한 글을 읽었다. 7가지 핵심 항목으로 나눠서 설명해줘서 이해하기 좋았음.

---

### 1. 시맨틱 HTML

접근성의 시작은 시맨틱 HTML이다. `<button>`, `<a>` 같은 요소들은 기본적으로 접근성을 내장하고 있다.

❌ `<div>`에 `onClick`을 붙이는 건 안 된다. 키보드 탐색, 스크린 리더 등 접근성 기능이 아예 없다.

✅ 버튼은 `<button>`, 링크는 `<a>` — 역할에 맞는 요소를 쓰는 것만으로도 많은 게 해결된다.

`<select>`, `<input>`, `<textarea>` 같은 네이티브 폼 요소도 마찬가지다. 직접 커스텀 컴포넌트를 만들기보다 `react-select` 같은 검증된 라이브러리를 쓰는 게 낫다.

---

### 2. 폼

**모든 폼 필드는 `<form>` 안에, `onSubmit` 핸들러와 제출 버튼과 함께** 있어야 한다.

#### 라벨 연결

```html
<label for="email">이메일:</label> <input type="email" id="email" />
```

`for` / `htmlFor` 속성으로 명시적으로 연결해야 한다. 암시적 연결(`<label>` 안에 `<input>` 넣기)은 모든 스크린 리더가 지원하지 않는다.

React에서 ID 충돌 방지하려면 `useId` 훅 쓰면 된다:

```tsx
const id = useId();
// <label htmlFor={id}> ... <input id={id} />
```

#### 플레이스홀더는 라벨이 아니다

- 입력 시작하면 사라져서 뭘 입력해야 하는지 잊어버림
- 대비가 낮아 읽기 어려움
- 라벨 대신 쓰면 절대 안 됨

---

### 3. 키보드 내비게이션

Tab 키로 앱을 탐색하고, Enter 키로 동작을 실행할 수 있어야 한다.

#### 포커스 표시기 절대 없애지 말 것

```css
/* ❌ 이렇게 없애면 안 됨 */
:focus {
  outline: none;
}

/* ✅ 필요할 때만 표시되게 */
:focus-visible {
  outline: 2px solid blue;
}
```

`:focus`는 마우스 클릭에도 반응하는데, `:focus-visible`은 키보드로 포커스됐을 때만 표시된다. 브라우저가 알아서 판단해준다.

![포커스 표시기를 사용하여 폼 필드를 순차적으로 이동하기](https://martijnhols.nl/_next/static/media/accessibility-focus-indicator.ae7695c1.gif)

### 4. 모달

모달을 접근 가능하게 만드는 건 까다롭다. 가능하면 **`<dialog>` 요소**를 쓰는 게 제일 낫다. 대부분의 브라우저에서 잘 지원되고, 키보드 탐색 포함한 접근성 문제를 대부분 해결해준다.

직접 구현해야 한다면 세 가지를 챙겨야 한다:

#### 포커스 관리

1. 모달이 열리면 포커스를 모달 안으로 이동
2. Tab 키가 모달 밖으로 나가지 않도록 포커스 트랩
3. 모달 닫히면 트리거 버튼으로 포커스 반환

`react-focus-lock` 라이브러리가 이걸 해결해준다.

#### 배경 콘텐츠 비활성화

```html
<!-- 모달 뒤 콘텐츠에 inert 추가 -->
<div inert>모달 뒤 콘텐츠</div>
```

`inert` 속성을 붙이면 보조 기술에서 숨겨지고, 포커스도 안 간다.

#### Escape 키로 닫기

사용자가 Escape 키로 모달을 닫을 수 있어야 한다.

---

### 5. 이미지 대체 텍스트

모든 `<img>`에 `alt` 속성은 필수다.

```html
<!-- 의미 있는 이미지 -->
<img src="chart.png" alt="2024년 월별 매출 차트" />

<!-- 장식용 이미지 — alt를 빈 문자열로 -->
<img src="divider.svg" alt="" />
```

`alt`를 아예 빠뜨리면 스크린 리더가 파일명을 읽어버린다. 장식용이라면 빈 문자열로라도 넣어야 한다.

**작성 팁**: "시력이 좋지 않은 사람에게 이미지를 설명한다"고 생각하면 뭘 써야 할지 나온다. 이미지 안에 텍스트가 있으면 그것도 반드시 포함.

---

### 6. 스타일링

#### 클릭 가능 영역

버튼과 링크는 누르기 쉬운 크기여야 한다. 패딩으로 영역을 키우고, 시각적으로 너무 커 보이면 음수 마진으로 균형 맞추면 된다.

![모달 닫기 버튼의 클릭 가능한 영역 비교](https://martijnhols.nl/_next/static/media/accessibility-closebuttonclickablearea.b93d2236.jpg)

#### 애니메이션 줄이기

움직임에 민감한 사용자를 위해 `prefers-reduced-motion` 미디어 쿼리를 챙긴다.

```css
@media (prefers-reduced-motion: reduce) {
  .modal {
    animation: none;
  }
}
```

#### 상대 단위 사용

브라우저 기본 글꼴 크기 설정을 존중해야 한다. `px` 대신 `rem`, `em`을 쓰면 사용자가 글꼴 크기를 바꿔도 레이아웃이 깨지지 않는다.

### 7. ARIA 속성

ARIA는 **최후의 수단**이다. 시맨틱 HTML로 해결되면 ARIA는 필요 없고, 잘못 쓰면 오히려 해롭다.

#### aria-label

눈에 보이는 텍스트 없이 아이콘만 있는 버튼에 사용.

```tsx
<button aria-label="검색">
  <SearchIcon />
</button>
```

상호작용하는 요소에만 써야 한다.

#### aria-hidden

스크린 리더에서 숨겨야 하는 장식용/중복 요소에 사용.

```tsx
<div>
  React <ReactLogo aria-hidden />
</div>
```

이 외에도 `aria-live`, `aria-expanded`, `aria-describedby` 등이 있는데, 복잡도가 높아 웬만하면 시맨틱 HTML로 먼저 해결하는 게 낫다.

---

### 📌 정리

`접근성은 나중에 챙기는 게 아니라, 처음부터 개발 과정의 일부다`

#### 핵심

> 접근성의 80%는 시맨틱 HTML, 폼 라벨, 키보드 탐색, 포커스 관리 — 이 네 가지에서 나온다.
> ARIA는 최후의 수단. 시맨틱 HTML이 먼저다.

#### ⭐️ 앞으로 기억할 것

- 1. **`<div onClick>` 대신 `<button>`**
     키보드 탐색, 스크린 리더 모두 네이티브 요소가 해결해준다.

- 2. **라벨은 반드시 `htmlFor`로 명시적 연결**
     플레이스홀더는 라벨 대체 불가. `useId`로 ID 충돌 방지.

```tsx
const id = useId();
<label htmlFor={id}>이름</label>
<input id={id} />
```

- 3. **`:focus` 지우지 말고 `:focus-visible`로 교체**

```css
:focus-visible {
  outline: 2px solid blue;
}
```

- 4. **모달 = `<dialog>` 먼저 고려**
     직접 구현 시 포커스 트랩 + `inert` + Escape 닫기 세 가지 필수.

- 5. **`alt=""` 빈 문자열도 명시적으로**
     `alt` 자체를 빠뜨리면 파일명을 읽는다.

- 6. **`px` 대신 `rem`/`em`**
     브라우저 글꼴 크기 설정 존중.

> **📌 결론**
> ▎ 접근성은 "보조 기술 사용자만을 위한 것"이 아니라 모든 사용자의 UX를 개선한다
> ▎ 시맨틱 HTML만 잘 써도 절반은 해결된다
> ▎ ⭐️ 완성 후 실제로 Tab 키만으로 앱을 한 번 탐색해보자 — 문제가 바로 보인다
