# 프런트엔드 아키텍처 시각화하기

> **원문**: https://frontendatscale.com/issues/17
> **번역 출처**: https://junghan92.medium.com/%EB%B2%88%EC%97%AD-%ED%94%84%EB%9F%B0%ED%8A%B8%EC%97%94%EB%93%9C-%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98-%EC%8B%9C%EA%B0%81%ED%99%94%ED%95%98%EA%B8%B0-3ae9a3dd8e67

아키텍처 다이어그램, 그리는 것 자체는 쉽다. 박스 그리고 화살표 잇고 라벨 붙이면 끝. 근데 *이해되게* 그리는 건 완전히 다른 얘기다. 보는 사람마다(PM, 디자이너, 백엔드, 신입) 궁금한 게 다르니까. 이 글은 **C4 모델**을 프론트엔드에 맞게 변형해서, 같은 시스템을 줌인/줌아웃하며 보여주는 법을 다룬다. 앞에서 정리한 "[프런트엔드 아키텍처란 무엇인가]"가 *결정*에 대한 글이었다면, 이건 그 결정을 *어떻게 그려서 소통하느냐*에 대한 글.

### 다이어그램은 엔지니어의 핵심 역량

다이어그램을 그리는 능력은 소프트웨어 엔지니어의 필수 스킬이다. 박스·화살표·라벨로 그리는 것 자체는 단순하지만, **다양한 청중이 이해하게** 만드는 게 어렵다. 이해관계자마다 우선순위가 다르기 때문.

### 코드의 지도 (Code as Maps)

Simon Brown이 만든 **C4 모델**은 코드를 여러 상세 수준의 "지도"로 그리는 방법이다.

> 구글 맵에서 줌인/줌아웃하는 것처럼, 같은 시스템을 다른 배율로 본다. 전국 지도와 동네 골목 지도가 둘 다 필요한 것처럼.

기술/비기술 청중 모두에게 통하도록 **4개 레벨**로 나눈다:

| 레벨 | 이름 | 보여주는 것 | 주 청중 |
|---|---|---|---|
| 1 | System Context | 시스템 + 사용자/외부 시스템과의 상호작용 | 비기술 이해관계자 |
| 2 | Containers | 시스템을 앱·데이터 저장소로 분해 | 기술 청중 |
| 3 | Components | 컨테이너의 기본 구성요소 | 기술 청중 |
| 4 | Code | UML·ER·의존성 그래프 수준의 구현 디테일 | 개발자 |

- **레벨 1 (System Context)**: 가장 높은 곳에서 본 전경. 시스템이 누구와, 무엇과 상호작용하는지. 비개발자에게 설명할 때 유용.
- **레벨 2 (Containers)**: 시스템을 애플리케이션과 데이터 저장소 단위로 쪼갠다.
- **레벨 3 (Components)**: 컨테이너가 어떤 기본 빌딩 블록으로 분해되는지.
- **레벨 4 (Code)**: 코드 레벨. UML, ER 다이어그램, 의존성 그래프 등.

### 프론트엔드를 위한 C4 변형

C4는 백엔드 색이 강해서, 저자는 프론트엔드에 맞게 레벨 3·4를 비튼다.

#### 레벨 3: 컴포넌트 → **모듈(Modules)**

"컴포넌트"라는 단어를 그대로 쓰면 **UI 컴포넌트(버튼, 폼)와 헷갈린다.** 그래서 레벨 3에서는 "컴포넌트" 대신 **모듈**이라 부른다.

모듈은 **앱의 UI 화면 단위**로 조직한다:

```
/dashboard
/settings
/login
```

→ 라우트/화면 기준으로 큰 덩어리를 나누는 것.

#### 레벨 4: 코드 → **UI 분해(UI Decomposition)**

전통적 UML 대신, 모듈의 UI를 **3가지 범주**로 분해한다:

- **screens** — 모듈 안의 구별되는 뷰(화면)
- **features** — 중첩 폴더로 둘 만큼 복잡한 UI 섹션
- **components** — 버튼·폼 같은 표준 UI 요소

> 즉 레벨 4에서는 "클래스 다이어그램"이 아니라 "이 화면이 어떤 screen / feature / component로 쪼개지는가"를 그린다. 프론트엔드의 실제 폴더 구조와 자연스럽게 맞물린다.

---

### 📌 정리
`C4 모델을 프론트엔드로 — 컴포넌트는 "모듈", 코드는 "UI 분해"로`

#### 핵심
> 좋은 아키텍처 다이어그램은 "그리기 쉬움"이 아니라 "이해시키기"가 관건이다. Simon Brown의 C4 모델은 코드를 4개 배율(Context→Container→Component→Code)의 지도로 그려 청중별로 소통한다. 프론트엔드에 적용할 땐 레벨 3을 UI 컴포넌트와 혼동되지 않게 **"모듈"(화면/라우트 단위)**로, 레벨 4를 UML 대신 **screens / features / components의 "UI 분해"**로 변형하면 실제 코드 구조와 잘 맞는다.

#### ⭐️ 앞으로 기억할 것
- 1. **다이어그램은 줌 레벨이 다른 "지도"다** — 한 장에 다 담지 말고 청중에 맞춰 배율을 바꿔라
- 2. **C4 4레벨**: Context(전경) → Container(앱/저장소) → Component(빌딩블록) → Code(구현)
- 3. **프론트 레벨 3 = "모듈"** (`/dashboard`, `/settings` 같은 화면/라우트 단위) — UI 컴포넌트와 단어 충돌 회피
- 4. **프론트 레벨 4 = "UI 분해"** = screens / features / components 3분류
  ```
  screens   → 모듈 안의 개별 화면
  features  → 중첩 폴더가 필요한 복잡 섹션
  components→ 버튼·폼 등 표준 UI
  ```
- 5. **Diagrams as Code** — Structurizr(DSL) / Mermaid / PlantUML로 코드처럼 다이어그램 관리 → 버전관리·갱신 용이

📌 결론
▎ 아키텍처 소통의 핵심은 "한 장의 완벽한 그림"이 아니라 "배율별 지도 세트".
▎ C4를 프론트엔드 어휘(모듈 / screens·features·components)로 번역해 쓰면 코드와 그림이 일치한다.
▎ ⭐️ 실무 적용 포인트
▎ - 온보딩 문서에 → 레벨 1~2(Context/Container) 한 장씩 그려두면 신입이 전체 그림을 빨리 잡는다.
▎ - 폴더 구조 설계 시 → screens/features/components 3분류를 레벨 4 분해와 맞춰 일관성 유지.
▎ - 다이어그램이 금방 낡는 게 문제면 → Structurizr/Mermaid로 "코드로 그리는" 방식 도입.
