# 코드베이스를 수직으로 쌓아라 (The Vertical Codebase)

> 원문: [The Vertical Codebase](https://tkdodo.eu/blog/the-vertical-codebase) — TkDodo

TkDodo(React Query 메인테이너) 글. 코드베이스를 어떻게 "제대로" 구조화할지에 대한 그의 의견인데, 핵심 주장은 명확하다. **`components / hooks / types / utils` 같은 "기술 레이어별(horizontal) 분리"를 버리고, "기능별(vertical) 분리"로 가라.** 2년 전 자기 트윗을 인용하며 시작하는데, 지금도 100% 동의한다고.

> components / hooks / types / utils 분리는 많은 코드베이스에서 보이지만 내가 제일 싫어하는 방식이다. 도메인이 아니라 **타입으로** 묶는다. `useTheme`이 `useTodo` 옆에는 살지만 정작 `ThemeProvider` 옆엔 없다… 왜?

![Code colocation](/blog/_astro/GEg89DDXsAA8DC.DbrOSQsh.jpeg)

### horizontal(수평) 분리가 왜 망하는가

처음 시작할 땐 편하다. 근데 딱 거기까지다. 어차피 대부분은 `components`가 되고, 나머지 폴더는 파일 몇 개뿐. 시간이 지나면 악몽이 되고, 한번 자리잡으면 떼어내기가 정말 어렵다.

대표 사례가 Sentry 코드베이스. 이 구조로 10년+ 굴린 결과 **top-level `components` 디렉터리에 200개 넘는 파일**이 쌓였다. 공통점? "컴포넌트라는 것" 그 하나뿐. `analyticsArea`부터 `workflowEngine`까지 다 거기 있다. 아니 사실은 **아무것도 못 찾는다** — 어디서 찾기 시작해야 할지 모르니까.

> 스타트업이 망할 거면(NGMI) 구조는 중요하지 않다. 아마 그래서 대부분이 이걸 문제로 안 느낀다 — 몇 년 뒤 모습을 상상 못 하니까. 근데 일단 "성공"하는 위치까지 가면, 코드베이스가 규모와 성장을 버티게 만드는 게 좋은 생각이다. 그 출발이 horizontal 구조를 버리는 것.

### "근데 AI는 신경 안 쓰잖아?" 🤷

에이전트가 작성·리뷰·수정을 다 자동으로 하니 코드를 다신 안 본다는 사람들이 있다. "다음 모델이 주말에 다 갈아엎고 새로 짤 거야"라고 굳게 믿는 캠프라면 이 글은 당신 거 아니다. (근데 코드 안 읽는 사람은 블로그도 안 읽을 테니, 애초에 빈 벤다이어그램이긴 하다 ㅋㅋ)

TkDodo 주장: **에이전트도 사람과 똑같은 게 필요하다 — 경계(boundaries), 제약(constraints), 빠른 피드백 루프.** 탐색하기 쉬운 구조, 잘 잡힌 lint/TS 설정, 빠르고 믿을 수 있는 테스트. 그래서 에이전트가 **새 코드베이스엔 강하고, 수년간 유기적으로 자란 코드베이스엔 약한 것.** 내가 직접 탐색 안 해도 신경 써야 하는 이유.

### Code colocation — 함께 바뀌는 코드는 함께 둔다

중요한 건 [인지 부하(Cognitive Load)](https://github.com/zakirullin/cognitive-load)다. 코드베이스를 불필요하게 이리저리 점프하는 게 거기 더해진다. **같이 바뀌는 코드는 같이 살아야 한다.** 예를 들어 컴포넌트의 `props`를 별도 파일로 빼는 사람은 거의 없다:

```tsx
// src/components/widget.tsx
export type WidgetProps = { id: string }

export function Widget(props: WidgetProps) {}
```

`props`는 컴포넌트의 일부(public interface)니까 같이 두는 게 당연하다. 좋다. 근데 여기에 데이터 페칭을 추가하면?

```tsx
export type WidgetProps = { id: string }

export function Widget(props: WidgetProps) {
  const { data } = useSuspenseQuery(widgetQueryOptions(props.id))
}
```

**`widgetQueryOptions`는 어디 사나?** util이니까 `src/utils/widget.ts`? 🤨 코드 양이 적을 땐 괜찮지만 스케일이 안 된다. `components`도 `hooks`도 아니란 이유만으로 모은 `utils` 디렉터리는 꽤 자의적이다.

Sentry에선 `utils/analytics`가 `components/analyticsArea`와 논리적으로 묶여 있고, `utils/feedback`은 `components/feedback`에서만 쓴다. `profiling` 관련 코드는 `components/profiling` / `types/profiling` / `utils/profiling` / `views/profiling`로 쪼개져 있다. *말이 되게 해봐라(make it make sense).*

#### 낮은 결합(coupling), 높은 응집(cohesion)

우리가 원하는 건 [low coupling & high cohesion](https://www.geeksforgeeks.org/software-engineering/software-engineering-coupling-and-cohesion/). 의존 관계가 명확하고 모듈이 최대한 집중된 구조. horizontal 분리는 정반대를 만든다 — 모듈 간 경계 없음(어떤 컴포넌트든 어떤 util이든 import 가능) + 느슨하게 관련된 코드가 같이 삶.

### 수직으로 정렬하라 (Align Vertically)

컴포넌트가 유행했을 때 js/css/html에 뭘 했는지 기억하나? **논리적으로 말이 되는 덩어리로 섞었다.** 버튼의 스타일과 버튼의 js가 갑자기 같이 살게 됐다. 혁명적. 🎉

![Separation of concerns](/blog/_astro/separation_of_concerns.CHn-okft.jpeg)

또 하나: 대부분 회사엔 제품을 위에서 아래까지 소유하는 **feature 팀(product engineer)** 이 있다. "백엔드 팀 / 프론트 팀 / UX 팀"으로 안 나누고, dashboards 팀·replays 팀·billing 팀이 있고 각 팀에 프론트·백엔드·디자이너가 다 섞여 자기 도메인을 온전히 소유한다. **코드베이스도 그렇게 조직하는 게 맞다** — 각 팀이 "자기 코드"를 위한 전용 공간을 갖게.

js와 css를 안 나누고, 프론트와 백엔드 엔지니어를 안 나누는데, 왜 components와 hooks/utils는 나누나?

### 올바른 vertical 찾기

모든 코드는 뭔가를 한다(아니면 존재하면 안 됨). 그 "하는 일"을 알아내서 논리적으로 비슷한 코드끼리 묶는다.

**`widget` 관련 코드는 전부 `src/widgets/`로.** 컴포넌트든 훅이든 타입이든 util이든 상수든 상관없다. **기술적으로 뭐냐가 아니라 무슨 일을 하느냐**가 유일한 기준. 이게 팀 구조나 `codeowners` 설정과도 잘 맞아떨어진다(profiling 팀 코드는 `src/profiling/`).

#### Domains? Features? → "Verticals"
- "도메인으로 묶어라"는 좀 제한적 — 도메인은 비즈니스 규칙에 묶이고 그게 바뀔 수 있음. 모든 코드가 비즈니스 문제를 푸는 것도 아니고.
- "feature로 묶어라"도 비슷한 문제 — 모든 게 feature는 아님.
- 그래서 TkDodo는 **기술 레이어가 아닌 기능으로 묶은 코드 = Verticals** 라는 용어를 선호한다.

올바른 vertical을 찾는 게 어려운 부분이다. 정밀과학이 아니다("components는 여기, types는 저기"처럼 딱 떨어지지 않음). **좋은 그룹핑이 뭔지 실제로 생각해야 한다.** 시작점은 보통 **라우트/페이지** — `/dashboard` 페이지가 있으면 그 그룹부터. dashboard가 widget을 가지면 거기에. widget이 충분히 크거나 여러 라우트에서 쓰이면(`/explore`에서도 widget을 만든다면) 자체 vertical로 승격.

### 그럼 공유 코드는? (Shared Code)

이 구조 제안하면 항상 첫 번째로 나오는 질문. 모든 코드가 단일 vertical에 넣을 만큼 고립돼 있진 않잖아?

답: 보통 **공유 코드도 자기만의 vertical로 만든다.** Sentry의 `PageFilters`는 여러 페이지에서 쓰이는데, "feature"라 부르긴 뭣해도 그 자체로 완벽히 유효한 vertical/도메인이다. (그래서 "feature"란 용어를 피한 것.) 지금은 컴포넌트는 `components/pageFilters`, 타입은 `types/core`, util은 `utils/withPageFilters`에 흩어져 있다 😭 → 발견도 어렵고, 소유권도 불명확하고, 작은 변경도 먼저 파헤쳐야 함.

#### 디자인 시스템
도메인 특화가 아닌 공유 코드는 보통 **디자인 시스템**에서 와야 한다. 없으면 top-level `/design-system` 디렉터리를 만들어 그 자체를 vertical로 두면 된다. 스코프가 명확하니까 — 일관성을 위한 재사용 에셋·컴포넌트·패턴.

### 경계 (Boundaries)

코드를 한 vertical로 모으면 응집은 바로 올라가지만, **결합이 자동으로 줄진 않는다.** 한 feature용 유틸을 만들었는데 나중에 예상 못 한 곳에서 재사용되고 있어서 작은 변경도 위험해지는 일, 한두 번이 아니다.

고치려면 경계가 필요하다. **vertical의 어느 부분이 다른 vertical에서 쓰여도 되는 "public"이고, 어느 부분이 "private"인지 선언**하는 방법 = public interface.

#### 모노레포
가장 좋은 방법은 모노레포. 각 vertical이 자체 패키지가 되고, 자체 의존성을 갖고, public interface는 `package.json`의 [`exports`](https://nodejs.org/api/packages.html#exports) 필드로 정의. 이제 vertical 간 의존이 **명시적**이 된다. [`pnpm workspaces`](https://pnpm.io/workspaces)가 이 용도로 좋다.

"이 vertical은 저 vertical에 의존 못 함" 같은 규칙도 만들 수 있다. Nx는 기본 제공. 모노레포 없이도 가능 — Sentry는 [`eslint-plugin-boundaries`](https://github.com/javierbrea/eslint-plugin-boundaries)로 어디가 뭘 의존할 수 있는지 강제한다(예: 디자인 시스템의 private utils에 deep import 하면 eslint 에러).

### 함정은? (Where's the catch?)

공짜 점심은 없다:
- **각 코드에 올바른 vertical을 고르는 게 어렵다.**
- **"private" 코드 때문에 여러 팀이 같은 걸 처음부터 다시 구현할 위험.**

둘 다 팀 간 소통이 더 필요하고, 그게 소프트웨어 엔지니어의 진짜 어려운 부분이다(늘 그래왔다). 쉽다고 한 적 없다. 그래도 조언은 변함없다: **코드베이스를 vertical로 쌓아라. 나머지는 따라온다.**

### 📌 정리
`타입(components/hooks/utils)이 아니라 기능(verticals)으로 묶어라`

#### 핵심
> 코드베이스를 `components / hooks / types / utils`처럼 **기술 레이어(horizontal)** 로 나누면, 같이 바뀌는 코드가 흩어지고(낮은 응집), 아무 데서나 import 가능해지고(높은 결합), 스케일에서 무너진다. 대신 **"무슨 일을 하느냐"로 묶는 vertical 구조**(`src/widgets/`, `src/profiling/` 안에 컴포넌트·훅·타입·util 다 포함)로 가라. 공유 코드는 자체 vertical 또는 design-system으로. 경계는 모노레포 패키지나 eslint-plugin-boundaries로 강제. 이건 사람뿐 아니라 AI 에이전트에게도 좋은 구조다.

#### ⭐️ 앞으로 기억할 것
- 1. **분류 기준은 "타입"이 아니라 "기능"**
  `useTheme`은 `useTodo` 옆이 아니라 `ThemeProvider` 옆에 살아야 한다. 기술적으로 뭐냐(component/hook/util)가 아니라 무슨 일을 하느냐.
  ```text
  ❌ src/components/widget.tsx
     src/hooks/useWidget.ts
     src/utils/widget.ts
  ✅ src/widgets/  ← 컴포넌트·훅·타입·util 전부 여기
  ```
- 2. **함께 바뀌는 코드는 함께 둔다 (colocation)**
  props를 별도 파일로 안 빼듯, `widgetQueryOptions`도 widget 옆에. 점프 = 인지 부하.
- 3. **vertical은 라우트/페이지에서 출발**
  `/dashboard` → dashboard vertical. 여러 곳에서 쓰이면 자체 vertical로 승격(`/explore`도 widget 쓰면 → `src/widgets/`).
- 4. **"공유 코드는?" → 그것도 vertical로**
  PageFilters처럼 여러 페이지가 쓰는 건 자체 vertical. 도메인 무관 공유는 `/design-system`.
- 5. **응집을 올렸으면 결합도 막아라 (boundaries)**
  모으는 것만으론 결합이 안 줄어든다. public/private interface 선언 필요.
  → 모노레포 + `package.json` exports / pnpm workspaces
  → 또는 `eslint-plugin-boundaries`, Nx project dependency rules
- 6. **AI 시대에도(오히려 더) 구조가 중요**
  에이전트도 경계·제약·빠른 피드백이 필요. 새 코드베이스엔 강하고 유기적으로 자란 코드베이스엔 약한 이유.
- 7. **함정 인지**
  올바른 vertical 고르기가 어렵고, private 코드로 인한 중복 구현 위험. 둘 다 결국 팀 간 소통 문제.

📌 결론
▎ horizontal(components/hooks/utils) = 처음엔 편하지만 스케일에서 무조건 망한다(Sentry 200개 파일).
▎ vertical = "무슨 일을 하느냐"로 묶기. 같이 바뀌는 코드가 같이 살아 응집↑, 경계가 생겨 결합↓.
▎ 공유 코드/디자인 시스템도 그냥 또 하나의 vertical일 뿐.
▎ 정답이 딱 떨어지지 않아서 어렵지만, 그 "생각하는 것"이 본질이다.
▎ ⭐️ 실무 적용 포인트
▎ → 새 프로젝트면 처음부터 `src/<feature>/` 단위로. 타입별 폴더 만들지 말 것.
▎ → 기존 horizontal 코드베이스면, 자주 같이 바뀌는 묶음부터 점진적으로 vertical로 이동.
▎ → 경계는 말로 하지 말고 eslint-plugin-boundaries 같은 도구로 강제(deep import 차단).
▎ → 우리 dyimin-homepage의 `src/features/web/<feature>/` 구조가 이미 이 vertical 철학과 동일.
