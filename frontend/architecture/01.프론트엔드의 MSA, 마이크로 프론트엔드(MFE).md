# 프론트엔드의 MSA, 마이크로 프론트엔드(MFE)

> 출처: [프론트엔드계의 MSA, MFE에 대해 알아보자](https://velog.io/@zerone001/MFE) — @zerone001 (Velog, 한국어 원문)

앱이 커질수록 모놀리식 프론트엔드는 작은 수정에도 전체 빌드/배포가 필요하고, 여러 팀이 한 코드베이스를 만지면서 머지 충돌이 늘고, 기술 스택도 하나로 강제된다. MFE는 이걸 "독립적으로 개발·배포·운영 가능한 작은 단위로 쪼개는" 방식으로 푼다. 통합을 "언제(빌드타임/런타임)" × "어디서(서버/브라우저/엣지)" 두 축으로 보면 머릿속이 깔끔해진다.

### 모놀리식 프론트엔드의 문제

- 작은 변경에도 전체 리빌드/재배포가 필요 (배포 의존성)
- 여러 팀이 한 레포를 만지면서 머지 충돌 증가
- 기술 스택이 하나로 강제됨
- 빌드·테스트 시간이 계속 늘어남

> 마이크로 프론트엔드 = 애플리케이션을 독립적으로 개발·배포·운영 가능한 작은 단위로 분할하는 아키텍처

### 예시로 보기

온라인 교육 플랫폼을 떠올려보자. 결제 / 강의 / 커뮤니티 / 공통 UI 팀이 각자 다른 프레임워크(Vue, React, Svelte)를 써도 되고, 독립적으로 배포한다. 그리고 컨테이너(셸) 애플리케이션이 이걸 하나로 통합해 사용자에게 보여준다.

![MFE 구조 예시](https://velog.velcdn.com/images/zerone001/post/d2e5216f-1ad7-445f-a994-e1cd18e11288/image.png)

### "그냥 코드만 분리하면 안 되나?"

안 된다. 코드만 나눠봤자:
- 결국 전부 함께 빌드된다
- 일부만 고쳐도 전체 재배포가 필요하다
- 기술 스택 통일이 여전히 강제된다

진짜 MFE는 **제대로 된 통합을 통해 각 조각이 독립적인 라이프사이클(개발·배포·장애 격리)을 갖는 것**이 핵심이다.

---

## 통합 방식: "언제" 합치는가

### 빌드 타임 통합

배포 전에 미리 합쳐서 하나의 번들로 만드는 방식.

**방법 1 — 의존성 관리(npm 패키지)**
각 앱을 npm 패키지로 만들어 빌드 타임에 함께 번들링.

```json
{
  "dependencies": {
    "@app/aplication1": "^2.2.0",
    "@app/aplication2": "^1.9.0",
    "@app/aplication3": "^1.1.0"
  }
}
```

**방법 2 — 모노레포 기반**
Nx, Turborepo 같은 도구로 독립 패키지를 관리하다가 최종 빌드 때 하나의 번들로 합친다.

| 구분 | 내용 |
|------|------|
| ✅ 장점 | 코드 일관성, 강력한 초기 로딩 성능, 타입 안정성 |
| ❌ 단점 | 빌드 시간은 그대로, 라이브러리 버전 통일 필요 |

### 런타임 통합

독립적으로 배포하고, 사용자가 접속하는 순간 실시간으로 조립.

**Module Federation (Webpack 5)**
호스트(컨테이너) 앱이 기능을 노출(expose)하는 리모트 앱(독립 MFE)들을 런타임에 통합한다.

| 구분 | 내용 |
|------|------|
| ✅ 장점 | 기술 스택 자유, 코드 충돌 없음, 장애 격리 |
| ❌ 단점 | (초기 설정 복잡, 런타임 의존) |

> 장애 격리 예시: 상품 팀이 새로 올린 기능이 오작동해도 다른 앱엔 영향이 없다. 서비스는 계속 돌아간다.

**싱글톤 패턴**
앱마다 다른 라이브러리 버전을 쓰더라도, 런타임에는 단일 버전을 강제해서 버전 충돌을 막는다. (개발 시점엔 각자 버전을 쓰되 런타임 충돌 방지)

**Next.js Multi-zone**
경로별로 다른 앱으로 라우팅하는 간단한 방식.

```javascript
// next.config.js
module.exports = {
  async rewrites() {
    return [
      {
        source: "/test/:path*",
        destination: "http://localhost:3001/:path*",
      },
    ];
  },
};
```

사용자가 `/test`로 들어오면 해당 앱으로 라우팅. 구현은 간단하지만 **풀 페이지 리로드(hard navigation)** 가 발생한다.

### 어떤 기준으로 고를까

- **소규모 팀 / 초기 로딩 속도 우선** → 빌드 타임 통합
- **대규모 팀 / 배포 분리 우선** → 런타임 통합

## 통합 위치: "어디서" 합치는가

### SSR (서버 사이드 렌더링)
서버가 모든 MFE를 가져와 조립해 완성된 HTML을 브라우저로 전송.

- ✅ 빠른 초기 로딩, SEO 최적화
- ❌ 불필요한 서버 부하, 조립이 서버에 집중

### CSR (클라이언트 사이드 렌더링)
브라우저가 필요한 앱을 동적으로 로드·조립. Module Federation으로 런타임 모듈 활용.

- ✅ 동적이고 인터랙티브한 UX
- ❌ 느린 초기 로딩, SEO 최적화 어려움

### 엣지 컴퓨팅
독립 앱들을 엣지 네트워크(Cloudflare Workers 등)에서 결합, 전 세계 사용자 근처에서 실행.

- ✅ 전 세계 빠른 응답, 서버 부하 분산, 클라이언트 의존 감소
- ❌ 엣지 인프라 필요, 비용 증가

### 📌 정리
`MFE는 "언제(빌드/런타임) × 어디서(서버/브라우저/엣지)" 두 축으로 본다`

#### 핵심
> MFE는 프론트엔드판 MSA로, 앱을 독립 개발·배포·운영 가능한 단위로 쪼개 모놀리식의 배포 의존성·머지 충돌·기술 스택 강제 문제를 푼다. 단순 코드 분리로는 부족하고 "제대로 된 통합"이 필요하다. 통합은 **시점(빌드 타임 vs 런타임)** 과 **위치(SSR / CSR / 엣지)** 두 축으로 결정하며, 정답은 팀 규모와 우선순위(초기 로딩 vs 배포 독립성)에 달렸다.

#### ⭐️ 앞으로 기억할 것
- 1. 코드 분리 ≠ MFE
  레포만 나눠도 함께 빌드되면 의미 없다. 독립 배포 + 장애 격리 + 라이프사이클 분리가 핵심.
- 2. 통합 시점 2가지
  - 빌드 타임: npm 패키지 / 모노레포(Nx, Turborepo) → 일관성·성능·타입 안정성, 대신 빌드시간 그대로·버전 통일 필요.
  - 런타임: Module Federation(Webpack5) → 기술 자유·충돌 없음·장애 격리.
- 3. 라이브러리 버전 충돌은 싱글톤으로
  Module Federation의 singleton 옵션으로 런타임에 단일 버전 강제. (React 중복 로딩 같은 문제 방지)
- 4. Next.js Multi-zone은 간단하지만 hard navigation
  `rewrites`로 경로별 다른 앱 연결. SPA 부드러운 전환은 안 되고 풀 리로드 발생 → 트레이드오프 인지.
  ```js
  async rewrites() { return [{ source: "/test/:path*", destination: "http://localhost:3001/:path*" }]; }
  ```
- 5. 통합 위치는 SEO/초기로딩/비용으로 결정
  SEO·초기로딩 중요 → SSR / 엣지. 인터랙티브 우선 → CSR. 글로벌 응답속도 → 엣지(비용↑).

📌 결론
▎ MFE는 대규모 프론트엔드에서 독립 스택·독립 배포·낮은 결합도를 준다.
▎ 대신 새로운 복잡성(코드 중복, UX 불일치, 리소스 증가)이 따라온다.
▎ 도입은 "유행"이 아니라 "우리 프로젝트의 실제 문제를 푸는가 + 팀 역량·조직 규모"로 판단.
▎ ⭐️ 실무 적용 포인트
  - 먼저 "두 축"으로 후보를 좁힌다: 빌드/런타임 → 그다음 SSR/CSR/엣지.
  - 소규모면 빌드 타임(모노레포)부터, 팀이 커지고 배포 분리가 절실해지면 런타임(Module Federation).
  - 공통 라이브러리는 singleton으로 묶어 중복 로딩·버전 충돌 예방.
