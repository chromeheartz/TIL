# useState 지옥 탈출하기 — useReducer

> 원문: [A cure for React useState hell?](https://www.builder.io/blog/use-reducer) — Builder.io (Steve Sewell)
> 번역/정리 출처: [번역글 — eunbinn](https://velog.io/@eunbinn/a-cure-for-react-useState-hell)

컴포넌트 하나에 `useState`가 대여섯 개씩 쌓이기 시작하면 슬슬 불안해진다. 읽기도 힘들고, 상태끼리 모순되는 걸 막을 방법도 없고, 입력 검증을 어디다 둬야 할지도 애매하다. 이 글은 그 "useState 지옥"을 `useReducer`로 어떻게 정리하는지 단계적으로 보여준다. 솔직히 useReducer는 Redux 냄새 때문에 괜히 무겁게 느껴졌는데, 읽고 나니 "그냥 상태 변환을 한 함수에 모으는 도구"라는 게 와닿았다.

![thumbnail](https://velog.velcdn.com/images/eunbinn/post/ea6ac663-30d1-41ae-9919-3a711f6076f8/image.jpeg)

### 문제: useState 지옥

캘린더 이벤트 편집 폼. 상태가 늘어날수록 `useState`도 줄줄이 늘어난다.

```jsx
import { useState } from "react";

function EditCalendarEvent() {
  const [startDate, setStartDate] = useState();
  const [endDate, setEndDate] = useState();
  const [title, setTitle] = useState("");
  const [description, setDescription] = useState("");
  const [location, setLocation] = useState();
  const [attendees, setAttendees] = useState([]);

  return (
    <>
      <input value={title} onChange={(e) => setTitle(e.target.value)} />
      {/* ... */}
    </>
  );
}
```

뭐가 문제냐면:
- 한눈에 읽기 힘들다
- 안전장치(검증)가 없다
- 모순된 상태를 막을 수 없다 (종료일이 시작일보다 앞서는 경우 등)
- 입력 길이 같은 제약을 걸 곳이 없다

### useState는 더 강력한 걸로 대체할 수 있다

`useReducer` 등장. 가장 단순한 형태는 "이전 상태 + 새 값"을 병합하는 reducer다.

```jsx
import { useReducer } from "react";

function EditCalendarEvent() {
  const [event, updateEvent] = useReducer(
    (prev, next) => {
      return { ...prev, ...next };
    },
    { title: "", description: "", attendees: [] }
  );

  return (
    <>
      <input
        value={event.title}
        onChange={(e) => updateEvent({ title: e.target.value })}
      />
      {/* ... */}
    </>
  );
}
```

> useReducer의 핵심 이점: **상태 변환을 제어할 수 있는 함수(reducer)를 끼워넣을 수 있다**는 것. 모든 상태 갱신이 이 한 함수를 거쳐간다.

### 예제: reducer 안에서 검증하기

상태가 바뀔 때마다 reducer가 검증/보정을 한다. 검증 로직이 한 곳에 모인다.

```jsx
import { useReducer } from "react";

function EditCalendarEvent() {
  const [event, updateEvent] = useReducer(
    (prev, next) => {
      const newEvent = { ...prev, ...next };
      // 시작 날짜가 종료 날짜 이후가 될 수 없음을 보장
      if (newEvent.startDate > newEvent.endDate) {
        newEvent.endDate = newEvent.startDate;
      }
      // 제목이 100자를 넘을 수 없음을 보장
      if (newEvent.title.length > 100) {
        newEvent.title = newEvent.title.substring(0, 100);
      }
      return newEvent;
    },
    { title: "", description: "", attendees: [] }
  );

  return (
    <>
      <input
        value={event.title}
        onChange={(e) => updateEvent({ title: e.target.value })}
      />
      {/* ... */}
    </>
  );
}
```

### useReducer는 useState가 되는 곳이면 어디든 된다

작은 컴포넌트도 마찬가지. 흔한 카운터:

```jsx
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);

  return <button onClick={() => setCount(count + 1)}>Count is {count}</button>;
}
```

useReducer로 바꾸면 검증(최대 10)을 자연스럽게 끼워넣을 수 있다:

```jsx
import { useReducer } from "react";

function Counter() {
  const [count, setCount] = useReducer((prev, next) => Math.min(next, 10), 0);

  return <button onClick={() => setCount(count + 1)}>Count is {count}</button>;
}
```

### (선택) Redux 스타일 패턴

더 복잡해지면 `action.type`으로 분기하는 익숙한 형태로.

```jsx
import { useReducer } from "react";

function EditCalendarEvent() {
  const [event, updateEvent] = useReducer(
    (state, action) => {
      const newEvent = { ...state };

      switch (action.type) {
        case "updateTitle":
          newEvent.title = action.title;
          break;
        // 다른 action들...
      }
      return newEvent;
    },
    { title: "", description: "", attendees: [] }
  );

  return (
    <>
      <input
        value={event.title}
        onChange={(e) => updateEvent({ type: "updateTitle", title: "Hello" })}
      />
      {/* ... */}
    </>
  );
}
```

> 단, 이건 여러 패턴 중 하나일 뿐. 진짜 복잡한 상태라면 Mobx / Zustand / XState 같은 라이브러리가 더 나을 수 있다.

### reducer 공유하기 (Context와 함께)

dispatch는 리렌더 간에 동일성이 유지돼서, 깊은 트리에 prop으로 일일이 내리지 않고 Context로 뿌리기 좋다.

```jsx
const TodosDispatch = React.createContext(null);

function TodosApp() {
  // 참고: `dispatch`(updateTodos)는 리렌더 간에 변하지 않는다
  const [todos, updateTodos] = useReducer(todosReducer);

  return (
    <TodosDispatch.Provider value={updateTodos}>
      <DeepTree todos={todos} />
    </TodosDispatch.Provider>
  );
}
```

```jsx
function DeepChild(props) {
  // action을 보내고 싶으면 context에서 dispatch를 꺼내 쓰면 된다
  const updateTodos = useContext(TodosDispatch);

  function handleClick() {
    updateTodos({ type: "add", text: "hello" });
  }

  return <button onClick={handleClick}>Add todo</button>;
}
```

### 흔한 함정: 상태를 직접 변경(mutate)하기

state는 불변으로 다뤄야 한다. 기존 객체를 건드리면 안 된다.

```jsx
// ❌ Wrong: 기존 객체를 그대로 변경
function reducer(state, action) {
  switch (action.type) {
    case "incremented_age": {
      state.age++;          // 🚩 기존 객체 변경
      return state;
    }
    case "changed_name": {
      state.name = action.nextName; // 🚩 기존 객체 변경
      return state;
    }
    // ...
  }
}
```

```jsx
// ✅ Correct: 항상 새 객체를 만들어 반환
function reducer(state, action) {
  switch (action.type) {
    case "incremented_age": {
      return { ...state, age: state.age + 1 };
    }
    case "changed_name": {
      return { ...state, name: action.nextName };
    }
    // ...
  }
}
```

### (선택) 해법: Immer

중첩 객체라 스프레드가 번거롭다면 [Immer]의 `use-immer`를 쓰면 "변경하는 것처럼" 쓰면서도 내부적으로 불변성을 지켜준다.

```jsx
import { useImmerReducer } from "use-immer";

function reducer(draft, action) {
  switch (action.type) {
    case "increment":
      draft.count++;   // draft는 마음껏 변경해도 됨
      break;
    case "decrement":
      draft.count--;
      break;
  }
}

function Counter() {
  const [state, dispatch] = useImmerReducer(reducer, initialState);
  return (
    <>
      Count: {state.count}
      <button onClick={() => dispatch({ type: "increment" })}>+</button>
      <button onClick={() => dispatch({ type: "decrement" })}>-</button>
    </>
  );
}
```

### 📌 정리
`상태가 모순될 수 있으면 useReducer로 변환을 한 곳에 가둬라`

#### 핵심
> useState 여러 개로 흩어진 상태는 검증·일관성 제어가 어렵다. useReducer는 모든 상태 갱신을 하나의 reducer 함수로 통과시켜, 검증/보정/모순 방지 로직을 한 곳에 모은다. Redux의 무거운 패턴이 아니라 `(prev, next) => ({...prev, ...next})` 수준으로 가볍게 시작할 수 있고, 필요해지면 action.type 분기·Context 공유·Immer로 확장하면 된다.

#### ⭐️ 앞으로 기억할 것
- 1. "상태끼리 관계/제약"이 생기면 useReducer 신호
  startDate ≤ endDate, title ≤ 100자 같은 불변식(invariant)을 reducer 한 곳에서 강제할 수 있다.
  ```js
  const [event, updateEvent] = useReducer((prev, next) => {
    const e = { ...prev, ...next };
    if (e.startDate > e.endDate) e.endDate = e.startDate;
    return e;
  }, initial);
  ```
- 2. 꼭 action.type 안 써도 된다
  가장 단순한 merge reducy `(prev, next) => ({...prev, ...next})`만으로도 useState 여러 개를 하나로 합칠 수 있다. switch는 복잡해질 때만.
- 3. reducer에서 절대 state를 mutate하지 말 것
  `state.age++` ❌ → `return {...state, age: state.age+1}` ✅. 안 그러면 리렌더가 깨진다.
  ```js
  // 중첩이 깊으면 Immer로:
  function reducer(draft, action) { draft.count++; } // use-immer
  ```
- 4. dispatch는 안정적 → Context로 뿌리기 좋다
  dispatch(updateXxx)는 리렌더 간 동일성이 유지돼서 prop drilling 없이 Context로 깊은 자식까지 전달 가능.
- 5. 단순하게 시작, 필요할 때만 복잡성 추가
  useState → useReducer → Immer → (정말 복잡하면) Zustand/Jotai/XState 순으로. 처음부터 무거운 거 꺼내지 말 것.

📌 결론
▎ useState 여러 개 = 검증·일관성 사각지대. useReducer로 변환 로직을 한 함수에 모은다.
▎ useReducer는 useState가 되는 곳이면 어디서나 된다 — 카운터 같은 작은 것도 OK.
▎ 직접 mutate 금지, 새 객체 반환. 번거로우면 Immer.
▎ ⭐️ 실무 적용 포인트
  - 폼처럼 필드 간 제약이 있는 상태는 처음부터 useReducer 고려.
  - useState가 4~5개 넘어가면 "이거 하나로 묶을 수 있나?" 자문.
  - Context + useReducer 조합이면 작은 전역상태는 외부 라이브러리 없이도 충분.
