# 새 리액트 문서가 권장하는 9가지 코드 스타일

> **원문**: https://blog.testdouble.com/posts/2023-10-16-react-docs-recommendations/
> **번역 출처**: https://velog.io/@eunbinn/react-docs-recommendations

![](https://velog.velcdn.com/images/chromeheartz/post/6ea1b4f6-6d72-42f9-9195-aecf5dcf3ff2/image.png)

리액트는 원래 "의견이 없는" 프레임워크로 유명했는데, 새로운 공식 문서(react.dev)에 생각보다 꽤 많은 권장 사항이 담겨 있었다. 이를 번역하신 분의 글을 보고 정리해보았다. 
팀 코드 리뷰에서 "왜 이렇게 써야 해?"라고 물어볼 때 공식 문서 레퍼런스로 던져줄 수 있는 9가지

---

### 1. 리스트 키는 인덱스 말고 안정적인 식별자 쓰기

```jsx
// ❌ 인덱스 키
items.map((item, index) => <li key={index}>…</li>)

// ✅ 고유 id
items.map((item) => <li key={item.id}>…</li>)
```

리액트는 `key`로 리스트 요소를 추적한다. 배열 인덱스를 쓰면 항목이 추가·삭제·정렬될 때 리액트가 잘못된 요소를 같은 것으로 인식해서 버그가 생긴다. ESLint 룰로는 `react/no-array-index-key`.

---

### 2. 컴포넌트는 모듈 최상위에 정의하기 (중첩 금지)

```jsx
// ❌ 함수 안에서 컴포넌트 정의
function ParentComponent() {
  function ChildComponent() { … }
  return <div><ChildComponent /></div>;
}

// ✅ 파일 최상위에 분리
function ChildComponent() { … }

function ParentComponent() {
  return <div><ChildComponent /></div>;
}
```

컴포넌트를 다른 컴포넌트 안에서 정의하면, 부모가 렌더링될 때마다 자식 컴포넌트가 "새로운" 컴포넌트로 취급된다. 결과적으로 매 렌더마다 자식을 언마운트/마운트하는 성능 문제가 생긴다. ESLint 룰: `react/no-unstable-nested-components`.

---

### 3. 상태는 최소 표현만 저장하기

```jsx
// ❌ 파생 가능한 값도 state로 관리
const [allItems, setAllItems] = useState([]);
const [urgentItems, setUrgentItems] = useState([]);

function handleSomeEvent(newItems) {
  setAllItems(newItems);
  setUrgentItems(newItems.filter(item => item.priority === 'urgent'));
  // 두 state가 언제든 어긋날 수 있음
}

// ✅ 하나만 state로, 나머지는 파생
const [allItems, setAllItems] = useState([]);
const urgentItems = allItems.filter(item => item.priority === 'urgent');

function handleSomeEvent(newItems) {
  setAllItems(newItems); // 하나만 업데이트하면 됨
}
```

state가 여러 개이면 서로 sync가 어긋나는 순간이 반드시 온다. 계산해서 얻을 수 있는 값은 state에 넣지 말고 렌더링 중에 계산하는 게 맞다.

---

### 4. 최적화(메모이제이션)는 문제가 생긴 다음에 하기

```jsx
// ❌ 문제도 없는데 먼저 최적화
const urgentItems = useMemo(
  () => allItems.filter(item => item.priority === 'urgent'),
  [allItems]
);

// ✅ 일단 심플하게
const urgentItems = allItems.filter(item => item.priority === 'urgent');
```

`useMemo`, `useCallback`, `React.memo`는 코드를 복잡하게 만든다. 실제로 느리다고 측정되기 전까지는 쓰지 않는 것이 가독성 면에서 낫다. 최적화는 "관찰된 문제"에 대한 대응이지, 선제적으로 거는 게 아니다.

---

### 5. 다른 훅을 호출할 때만 `use` 접두사 붙이기

```jsx
// ❌ 훅을 안 쓰는데 use 붙임 → hooks 규칙 적용받아서 제한만 생김
function useDateColumnConfig() {
  return {
    dataType: 'date',
    formatter: prettyFormatDate,
    editorComponent: DateEditor,
  };
}

// ✅ 그냥 일반 함수로
function getDateColumnConfig() {
  return {
    dataType: 'date',
    formatter: prettyFormatDate,
    editorComponent: DateEditor,
  };
}

// ✅ 진짜 훅을 쓸 때만 use 붙임
function useNameColumnConfig() {
  const { t } = useTranslation(); // 다른 훅 호출 → use 타당
  return {
    dataType: 'string',
    title: t('columns.name'),
  };
}
```

`use`로 시작하는 함수는 리액트가 훅으로 취급해서 조건문·반복문 안에서 호출 못 하는 등 규칙이 적용된다. 다른 훅을 안 쓰는 함수라면 `use`를 빼면 훨씬 유연하게 쓸 수 있다.

---

### 6. prop 변경에 따른 state 조정 — Effect 말고 렌더링 중에

```jsx
// ❌ items가 바뀌면 effect에서 selection 초기화
function List({ items }) {
  const [selection, setSelection] = useState(null);

  useEffect(() => {
    setSelection(null);
  }, [items]);
}

// ✅ 렌더링 중에 직접 처리
function List({ items }) {
  const [prevItems, setPrevItems] = useState(items);
  const [selection, setSelection] = useState(null);

  if (items !== prevItems) {
    setPrevItems(items);
    setSelection(null); // 렌더 도중 바로 처리
  }
}
```

Effect는 리액트의 "탈출구"다. 외부 시스템과 동기화할 때 쓰는 것이지, 단순히 prop 변화에 state를 맞추는 용도로 쓰면 불필요한 렌더 사이클이 생긴다.

---

### 7. 데이터 페칭은 `useEffect` 말고 라이브러리

```jsx
// ❌ useEffect로 직접 fetch
const [items, setItems] = useState();
useEffect(() => {
  api.loadItems().then(newItems => setItems(newItems));
}, []);
// race condition, 로딩/에러 상태, 캐싱 전부 직접 해결해야 함

// ✅ React Query 등 라이브러리
import { useQuery } from '@tanstack/react-query';

const { data: items } = useQuery(['items'], () => api.loadItems());
```

`useEffect`로 데이터를 가져오면 race condition, cleanup, 에러 처리, 로딩 상태, 캐싱 등을 보일러플레이트로 다 만들어야 한다. 리액트 공식 문서도 TanStack Query, SWR 같은 라이브러리를 쓰라고 직접 권장한다.

---

### 8. 이벤트 응답 로직은 Effect 말고 이벤트 핸들러에

```jsx
// ❌ 저장 완료 후 처리를 effect에서
const [savedData, setSavedData] = useState(null);
const [validationErrors, setValidationErrors] = useState(null);

useEffect(() => {
  if (savedData) {
    setValidationErrors(null);
  }
}, [savedData]);

async function saveData() {
  const response = await api.save(data);
  setSavedData(response.data);
}

// ✅ 이벤트 핸들러 안에서 바로 처리
async function saveData() {
  const response = await api.save(data);
  setSavedData(response.data);
  setValidationErrors(null); // 여기서 바로
}
```

이벤트에 응답하는 코드는 이벤트 핸들러에 있어야 한다. Effect에 넣으면 state 변경이 간접적으로 연결되어 디버깅하기 어렵고, 의도치 않게 여러 번 실행될 수 있다.

---

### 9. Effect 의존성 배열에서 "지우지 말고" 함수에서 없애기

```jsx
// ❌ 경고 무시하고 의존성 배열에서 그냥 제거
useEffect(() => {
  doSomethingWith(someValue);
}, []); // someValue를 쓰는데 의존성 배열에 없음 → lint 경고 무시

// ✅ 의존성 자체가 필요 없도록 effect 내부를 리팩터
useEffect(() => {
  doSomethingWithoutNeedingValue();
}, []);
```

의존성 배열에서 값을 빼는 것은 "이 effect는 이 값에 반응하지 않아도 된다"는 거짓말이다. 나중에 찾기 힘든 버그의 원인이 된다. ESLint 룰 `react-hooks/exhaustive-deps`를 disable 하는 것도 마찬가지. 의존성이 문제라면 effect 로직 자체를 바꿔서 그 의존성이 필요 없게 만들어야 한다.

---

### 📌 정리

`리액트 공식 문서에는 "왜 이렇게 써야 해?"에 답하는 근거들이 이미 있다`

#### 핵심

> 리액트 문서는 이제 단순한 API 레퍼런스가 아니다. 코드 스타일과 설계 의사결정에 대한 명확한 입장을 가지고 있고, 9가지 권장 사항 대부분이 "복잡성 줄이기"와 "예측 가능성 높이기"로 귀결된다.

#### ⭐️ 앞으로 기억할 것

- 1. **state는 최소로, 파생값은 렌더링 중 계산**
  state끼리 sync 맞추려다 버그 나는 패턴이 제일 흔하다. 계산 가능하면 그냥 변수로.
  ```jsx
  const urgentItems = allItems.filter(item => item.priority === 'urgent');
  ```

- 2. **`use` 접두사는 진짜 훅을 호출할 때만**
  팀에서 커스텀 훅인 척하는 일반 함수 많이 봤는데, hooks 규칙 적용받아서 손해만 본다.

- 3. **Effect는 외부 시스템 동기화 전용**
  이벤트 응답, prop→state 반영, 데이터 페칭 모두 Effect의 올바른 용도가 아니다. Effect를 쓰려고 할 때마다 "외부 시스템이랑 sync 맞추는 건가?"를 한 번 물어보자.

- 4. **의존성 배열 lint 경고는 절대 `// eslint-disable`로 끄지 말기**
  경고가 불편하면 effect 구조를 바꾸는 게 정답이다.

- 5. **데이터 페칭은 TanStack Query / SWR**
  공식 문서가 직접 권장하므로 팀 설득할 때 레퍼런스로 쓸 수 있다.

> **📌 결론**
> ▎ 리액트 문서는 "하지 말라는 것"과 "왜"를 명확히 설명하는 코드 스타일 가이드이기도 하다
> ▎ 9가지 중 절반 이상이 Effect 오남용에 관한 것 — Effect를 쓰기 전에 한 번 더 생각하는 습관이 필요
> ▎ ⭐️ 코드 리뷰에서 "공식 문서에서 이렇게 권장해"라고 말할 수 있는 근거가 생겼다
