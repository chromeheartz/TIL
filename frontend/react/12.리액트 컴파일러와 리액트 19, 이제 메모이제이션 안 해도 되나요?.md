# 리액트 컴파일러와 리액트 19, 이제 메모이제이션 안 해도 되나요?

> **원문**: https://www.developerway.com/posts/react-compiler-soon
> **번역 출처**: https://velog.io/@eunbinn/react-compiler-soon

![react compiler thumbnail](https://velog.velcdn.com/images/eunbinn/post/5f2d6176-2d09-43e4-891c-82e5edb9bcb5/image.png)

리액트 19 얘기가 나오면서 "이제 `useMemo`, `useCallback` 다 필요 없어지는 거 아냐?"라는 말이 계속 들렸다. 솔직히 나도 헷갈렸어서, 컴파일러가 정확히 뭘 바꾸는 건지 정리해보려고 이 글을 읽었다. 결론부터 말하면 **리액트 19 ≠ 리액트 컴파일러**다. 둘은 같이 발표됐을 뿐 별개의 릴리즈다.

### 리액트 19는 리액트 컴파일러가 아니다

이게 핵심 오해 포인트다. 리액트 팀이 한 블로그 글에서 둘 다 발표했지만, 사실 별개의 릴리즈다.

- 리액트 19는 새 기능들과 함께 곧 나온다.
- 컴파일러(예전 코드명 `React Forget`)는 아직 개발 시간이 더 필요하다.
- 컴파일러 여정은 2021년에 시작됐고, 아직 갈 길이 남았다.

리액트 팀 멤버들이 "19랑 컴파일러는 별개"라고 트위터에서 직접 정정했을 정도다. 즉, **리액트 19 쓴다고 자동으로 컴파일러가 따라오는 게 아니다.**

### 리액트 컴파일러로 바뀌는 것들

컴파일러는 Babel 플러그인 형태로 동작한다. 핵심은 우리가 손대지 않아도 **모든 리액트 코드를 자동으로 메모이제이션** 해준다는 것.

예를 들어 이런 코드를 짜면:

```javascript
const Component = () => {
  const onSubmit = () => {};
  const onMount = () => {};

  useEffect(() => {
    onMount();
  }, [onMount]);

  return <Form onSubmit={onSubmit} />;
};
```

컴파일러는 실질적으로 이런 동작을 하는 코드처럼 만들어준다:

```javascript
const FormMemo = React.memo(Form);

const Component = () => {
  // 우리가 직접 useCallback을 안 써도 컴파일러가 알아서 감싼다
  const onSubmit = useCallback(() => {}, []);
  const onMount = useCallback(() => {}, []);

  useEffect(() => {
    onMount();
  }, [onMount]);

  return <FormMemo onSubmit={onSubmit} />;
};
```

> ⚠️ 주의: 진짜로 위 코드처럼 변환되는 건 아니다. "실질적인 효과"가 이렇다는 의미다.

이게 적용되면 우리가 평소에 알고 가르치던 것들이 꽤 많이 바뀐다.

#### 부모가 리렌더링되면 자식도 리렌더링된다 → 이제 사실이 아니다

지금은 prop이 안 바뀌어도 부모가 리렌더링되면 자식도 같이 리렌더링된다. 그래서 흔히 "부모 리렌더링 → 자식 리렌더링은 미신(myth)"이라고 설명하면서 "사실은 무조건 리렌더 된다"고 정정하곤 했다.

근데 컴파일러가 들어오면 **모든 게 메모이제이션** 되니까, 그 "미신"이 오히려 현실이 된다. prop이 안 바뀌면 자식은 진짜로 리렌더링 안 한다.

#### 더 이상 "성능을 위한 합성(composition)"은 필요 없다

지금은 무거운 컴포넌트의 불필요한 리렌더링을 막으려고 state를 아래로 내리는 트릭을 자주 쓴다.

❌ 컴파일러 전 — `isOpen`이 바뀔 때마다 `VerySlowComponent`까지 리렌더링됨:

```javascript
const Component = () => {
  const [isOpen, setIsOpen] = useState(false);

  return (
    <>
      <Button onClick={() => setIsOpen(true)}>open dialog</Button>
      {isOpen && <ModalDialog />}
      <VerySlowComponent />
    </>
  );
};
```

✅ 합성으로 state를 한 단계 아래로 격리 — 무거운 컴포넌트를 보호:

```javascript
const ButtonWithDialog = () => {
  const [isOpen, setIsOpen] = useState(false);

  return (
    <>
      <Button onClick={() => setIsOpen(true)}>open dialog</Button>
      {isOpen && <ModalDialog />}
    </>
  );
};

const Component = () => {
  return (
    <>
      <ButtonWithDialog />
      <VerySlowComponent />
    </>
  );
};
```

컴파일러가 들어오면 이런 "성능 목적의 합성"은 필요 없어진다. 다만 **관심사 분리(separation of concerns)** 차원에서의 합성은 여전히 가치 있다. 즉, 패턴 자체가 사라지는 게 아니라 "성능 때문에 억지로 쓰던 이유"가 사라지는 거다.

#### 더 이상 `useMemo` / `useCallback`을 도배하지 않아도 된다

이 두 훅은 사실상 프로덕션 코드에서 사라질 거다. 따라서:

- 복잡한 의존성 배열(dependency chain) 관리에서 해방
- `children` 같은 무심코 넘긴 prop 때문에 메모이제이션이 깨지는 문제도 사라짐

솔직히 이게 제일 반갑다. 의존성 배열 빼먹어서 버그 나는 거, 너무 흔했으니까.

#### diffing(비교)과 reconciliation(재조정)도 설명이 달라진다

지금 우리가 가르치는 내용은 이렇다. `<Child />`를 렌더링하면 `type`과 `props`를 가진 엘리먼트 객체가 만들어진다. 부모가 리렌더링되면 **새 객체**가 생기고, 얕은 비교(shallow comparison)에 걸려서 자식도 리렌더링된다.

컴파일러가 들어오면, prop이 안 바뀐 경우 `<Child />`는 **메모이즈된 객체**를 반환한다. 실질적으로 이런 코드와 같다:

```javascript
const Parent = () => {
  const child = useMemo(() => <Child />, []);
  return child;
};
```

### 그래도 안 바뀌는 것들

컴파일러가 만능은 아니다. 이건 여전히 유효하다:

- 컴포넌트 안에서 컴포넌트를 정의하는 건 여전히 안티패턴
- `key` 속성은 여전히 엘리먼트 식별에 필수
- Context는 여전히 까다로운 주제
- 데이터 페칭, 에러 핸들링 관련 논의도 그대로

### 📌 정리
`리액트 19와 컴파일러는 별개, 컴파일러는 자동 메모이제이션이다`

#### 핵심
> 리액트 19와 리액트 컴파일러는 **별개의 릴리즈**다. 컴파일러는 Babel 플러그인으로 동작하며 `useMemo`/`useCallback`/`React.memo`를 우리가 손대지 않아도 자동으로 적용해준다. 그 결과 "성능을 위한 합성", "의존성 배열 관리", "수동 메모이제이션" 같은 기존 노하우의 상당수가 필요 없어진다. 단, `key`·컴포넌트 안 컴포넌트 금지·Context·데이터 페칭 같은 기본기는 그대로 유효하다.

#### ⭐️ 앞으로 기억할 것
- 1. 리액트 19 쓴다고 컴파일러가 따라오는 게 아니다
  둘은 같이 발표됐을 뿐 별개. 컴파일러는 더 늦게, 단계적으로 온다.

- 2. 컴파일러 = 자동 메모이제이션
  내가 안 써도 알아서 감싸준다. "효과"만 동일하지 실제 변환 코드가 위 예시랑 똑같진 않다.
  ```javascript
  // 내가 짠 코드
  const onSubmit = () => {};
  // 컴파일러가 만드는 효과
  const onSubmit = useCallback(() => {}, []);
  ```

- 3. "부모 리렌더 → 자식 리렌더" 미신이 현실이 된다
  지금은 prop 안 바뀌어도 자식이 리렌더됐지만, 컴파일러 후엔 prop 안 바뀌면 진짜 안 한다.

- 4. 성능용 합성은 졸업, 관심사 분리용 합성은 유지
  state 내리는 트릭을 "성능 때문에" 쓸 이유는 없어진다. 구조적 이유로는 여전히 OK.

- 5. `useMemo`/`useCallback` 도배 끝
  의존성 배열 지옥, `children` 때문에 메모 깨지는 문제에서 해방.

📌 결론
▎ 리액트 19 ≠ 리액트 컴파일러 — 헷갈리지 말 것.
▎ 컴파일러는 "수동 최적화" 시대를 끝내는 자동 메모이제이션 도구다.
▎ 그래도 `key`, 안티패턴, Context, 데이터 페칭 같은 기본기는 변하지 않는다.
▎ ⭐️ 실무 적용 포인트: 지금 당장 `useMemo`/`useCallback`을 다 걷어내진 말 것. 컴파일러는 아직 안 왔고, 19와 별개로 단계적으로 도입된다. 다만 새 코드에서 "성능 때문에 억지로 쪼개는" 설계는 점점 비중을 줄여도 된다.
