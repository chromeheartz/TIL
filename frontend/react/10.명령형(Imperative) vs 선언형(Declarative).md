# 명령형(Imperative) vs 선언형(Declarative)

> 원문: [Imperative vs Declarative Programming](https://fireship.dev/c/react/imperative-vs-declarative) — Fireship

React 배우다 보면 무조건 듣는 말 — "React는 **선언형(declarative)** 이다." 근데 그게 대체 무슨 뜻인지 파고들면 "명령형은 **how**, 선언형은 **what**" 같은 정의가 나오는데, 이게 이미 차이를 아는 사람한테나 와닿지 모르는 사람한텐 하나도 안 와닿는다(그래서 이 글을 읽는 거고). "declarative"가 그냥 [good](https://twitter.com/jergason/status/699379381127368704)이라는 의미로 남용되는 것도 혼란을 더한다. 직관은 있는데 설명은 못 하는, 일종의 *vibe driven development*. Fireship이 비유 + 코드로 이걸 풀어준다.

### 비유 1 — 레드 랍스터

남편이랑 데이트하러 식당에 가서 직원에게:

- **명령형 (HOW):** "저기 'Gone Fishin' 사인 아래 빈 테이블이 보이네요. 저랑 남편이 저기로 걸어가서 앉을게요." → 자리를 *어떻게* 잡을지 단계를 나열.
- **선언형 (WHAT):** "2인 테이블 주세요." → *무엇을* 원하는지만.

### 비유 2 — 길 찾기

"월마트 옆인데, 너네 집 어떻게 가?"

- **명령형:** "주차장 북쪽 출구로 나와서 좌회전, I-15 북쪽으로… 12번가 출구에서 우회전… 298번지야." (단계 나열)
- **선언형:** "내 주소는 298 West Immutable Alley, Eden, Utah 84310야." (목적지만)

내가 어떻게 가든 중요한 건 내가 모는 차다 — *명령형 수동 변속기* vs *선언형 자동 변속기*.

> **중요 포인트: 많은 선언형 접근은 그 아래에 명령형 추상화 레이어를 깔고 있다.**
> - "2인 테이블요" = 직원이 자리까지 가는 명령형 단계를 다 안다고 가정
> - 주소를 안다 = GPS가 가는 길의 명령형 단계를 안다고 가정
> - 자동 변속기 = 기어 변속 위에 추상화 레이어가 있음

### 언어로 보면

| 명령형 | 선언형 | 둘 다 |
|---|---|---|
| C, C++, Java | HTML, SQL | JavaScript, C#, Python |

```sql
SELECT * FROM Users WHERE Country='Mexico';
```
```html
<article>
  <header>
    <h1>Declarative Programming</h1>
    <p>Sprinkle Declarative in your verbiage to sound smart</p>
  </header>
</article>
```

둘 다 한눈에 "무슨 일이 일어나는지" 안다. **무엇(what)** 에 관심이 있지 **어떻게(how)** 엔 관심 없다. 멕시코 유저를 어떻게 SELECT하는지, 브라우저가 `article`을 어떻게 파싱·렌더하는지는 추상화돼 숨겨져 있다. 내 *what*은 "멕시코 유저" 또는 "새 헤더+문단"일 뿐.

### 명령형 JS 예시 (HOW에 집중)

```javascript
// 1. double — 배열 각 원소 2배
function double(arr) {
  let results = [];
  for (let i = 0; i < arr.length; i++) {
    results.push(arr[i] * 2);
  }
  return results;
}

// 2. add — 배열 전체 합
function add(arr) {
  let result = 0;
  for (let i = 0; i < arr.length; i++) {
    result += arr[i];
  }
  return result;
}

// 3. jQuery — 버튼 클릭 시 highlight 토글 + 텍스트 변경
$("#btn").click(function () {
  $(this).toggleClass("highlight");
  $(this).text() === "Add Highlight"
    ? $(this).text("Remove Highlight")
    : $(this).text("Add Highlight");
});
```

이 셋의 공통점 = **무엇이 이들을 명령형으로 만드는가:**
1. **HOW를 서술한다** — 배열을 명시적으로 순회하거나 단계를 직접 깔아놓음.
2. **상태(state)를 변이(mutate)시킨다** — `results`, `result`, DOM을 직접 바꿈. (선언형/함수형 사고에 익숙하지 않으면 잘 안 보이는 포인트)
3. **한눈에 안 읽힌다** — 인터프리터처럼 코드를 한 줄씩 따라가야 이해됨(+가변 데이터라 맥락도 같이 추적해야 함).

### 선언형 JS 예시 (위 3가지 문제를 다 고침)

```javascript
// 1. map 활용
function double(arr) {
  return arr.map((item) => item * 2);
}

// 2. reduce 활용
function add(arr) {
  return arr.reduce((prev, current) => prev + current, 0);
}
```
```jsx
// 3. React 컴포넌트
<Btn
  onToggleHighlight={handleToggle}
  highlight={highlight}>
  {buttonText}
</Btn>
```

`map`/`reduce`를 쓰는 게 **"선언형 해법은 대개 명령형 구현 위의 추상화"** 라는 앞 얘기와 연결된다. 우리는 무엇을 원하는지만 서술하고(map/reduce가 어떻게 구현됐는지 모르고 관심도 없음), **상태를 직접 변이하지 않는다**(변이는 map/reduce 안에 추상화됨). 익숙해지면 더 읽기 쉽다.

3번은 살짝 치트로 React를 썼지만 — **세 가지 명령형 문제(how 서술/상태 변이/가독성)가 다 해결됐다.** React의 진짜 아름다움은 **state에 기반해 UI가 어떤 모습이어야 하는지 서술하는 것만으로 선언형 UI를 만들 수 있다**는 것.

### 선언형의 또 다른 이점: 맥락 독립성

덜 언급되는 이점 — 선언형 코드는 **context-independent**할 수 있다. "최종 목표가 무엇인가"에 집중하므로(목표 달성 단계가 아니라), **같은 코드를 다른 프로그램에서도 그대로 쓸 수 있다.** 위 두 함수와 컴포넌트는 program-agnostic. 명령형은 정의상 현재 상태의 맥락에 의존하는 경우가 많아 이게 어렵다.

#### 함수형 프로그래밍은 선언형의 부분집합
글에서 깊이 안 들어간 것 하나 — **함수형 프로그래밍은 선언형의 subset이다.** JS의 `.map` / `.reduce` / `.filter`부터 익히고 올라가라. 코드베이스 개선의 가장 낮은 곳에 달린 열매가 "더 함수형으로 만들기"일 확률이 높다.

#### 웹에서 모은 다른 정의들
- "개발자의 멘탈 모델에 맞는 언어로 프로그래밍하는 행위(기계의 동작 모델이 아니라)."
- "선언(declaration)으로 프로그래밍하는 것."
- 명령형 언어는 컴퓨터 내부 상태의 명시적 조작을 지정하고, 절차형 언어는 따라야 할 명시적 단계 시퀀스를 지정한다 — 선언형은 이와 대조.
- "제어 흐름(control flow)을 서술하지 않고 계산의 로직을 표현하는 패러다임."
- "**실행 중 코드를 추적(trace)할 수 있는가**로 선을 긋는다. 정규식(Regex)은 100% 선언형 — 패턴이 실행되는 동안 추적 불가능하니까."

### 📌 정리
`명령형 = HOW(단계 나열·상태 변이), 선언형 = WHAT(목표 서술·추상화)`

#### 핵심
> 명령형은 "어떻게"를 단계별로 지시하며 상태를 직접 변이하고, 선언형은 "무엇"을 원하는지 서술하고 그 구현(how)을 추상화 레이어에 맡긴다. JS의 `for` 루프 + `push`/`+=`(명령형) vs `map`/`reduce`(선언형), DOM 직접 조작(명령형) vs React 컴포넌트(선언형)가 대표 대비. React가 선언형이라는 건 **state에 기반해 UI 모습만 서술하면 되는 것.** 선언형은 상태 변이를 안 하고, 한눈에 읽히고, 맥락 독립적이라 재사용이 쉽다. 함수형은 선언형의 부분집합.

#### ⭐️ 앞으로 기억할 것
- 1. **명령형 3대 특징을 보면 식별된다**
  ① HOW를 단계로 서술 ② 상태를 변이 ③ 한 줄씩 추적해야 이해됨. 이 셋이 보이면 명령형.
- 2. **선언형 = "구현 위의 추상화"**
  `map`/`reduce`가 어떻게 동작하는지 몰라도 됨 — 그 아래 명령형 루프가 숨어있을 뿐. "2인 테이블요"처럼.
  ```javascript
  // ❌ 명령형: 어떻게(루프+push+변이)
  let r = []; for (let i=0;i<arr.length;i++) r.push(arr[i]*2);
  // ✅ 선언형: 무엇(각 원소 2배)
  arr.map(x => x * 2);
  ```
- 3. **선언형은 상태를 변이하지 않는다**
  새 값을 반환할 뿐 기존 걸 안 바꿈 → 맥락 추적 부담↓, 예측 가능성↑.
- 4. **React가 선언형이라는 의미**
  DOM을 직접 토글(`$(this).toggleClass`)하지 말고, **state에 따라 UI가 어떻게 보일지만 서술**(`highlight={highlight}`). React가 실제 DOM 조작(how)을 대신함.
- 5. **선언형 코드는 맥락 독립적 → 재사용 쉬움**
  목표만 서술하니 program-agnostic. 명령형은 현재 상태 맥락에 의존해 재사용이 어렵다.
- 6. **시작점은 `.map` / `.reduce` / `.filter`**
  함수형은 선언형의 부분집합. 코드베이스를 더 선언형으로 만드는 가장 쉬운 첫걸음.

📌 결론
▎ 명령형은 "어떻게 도달하나"의 단계, 선언형은 "무엇을 원하나"의 목표.
▎ 선언형의 실질 이점 3가지: 상태 변이 없음 / 한눈에 읽힘 / 맥락 독립(재사용).
▎ React의 선언형 = state → UI 매핑만 서술하면 DOM 조작은 React가 처리.
▎ ⭐️ 실무 적용 포인트
▎ → 루프+push+변이 패턴을 보면 map/reduce/filter로 바꿀 수 있나 먼저 자문.
▎ → React에서 ref로 DOM 직접 만지는 코드(명령형)는 state 기반 선언형으로 옮길 수 있는지 점검.
▎ → "이 코드, 다른 프로그램에 그대로 떼서 쓸 수 있나?"가 선언형/재사용성의 리트머스.
