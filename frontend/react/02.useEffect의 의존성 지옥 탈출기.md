 # React의 useEffectEvent — `useEffect` 의존성 지옥 탈출기

  > **원문**: https://blog.logrocket.com/react-has-finally-solved-its-biggest-problem-useeffectevent/
  > **번역 출처**: https://velog.io/@superlipbalm/react-has-finally-solved-its-biggest-problem-useeffectevent


![](https://velog.velcdn.com/images/chromeheartz/post/0b6f4783-6f91-4b18-a95b-9866ce378682/image.png)

  React를 쓰면서 `useEffect` 때문에 머리 아팠던 적, 한 번쯤은 다
  있을 거다. 의존성 배열 넣어야 하나 말아야 하나, 넣으면 무한루프,
   빼면 lint 경고... 이 글은 그 근본 원인인 "오래된 클로저" 문제를
   `useEffectEvent`가 어떻게 해결하는지 다룬다.

  ### 왜 `useEffect`가 문제인가

  React 버그의 대표 원인은 단연 `useEffect`다. 실제로 Cloudflare가
   자사 대시보드에 의존성 배열에 객체를 잘못 넣어 **무한 루프를
  일으켜 전체 대시보드를 다운**시킨 사고가 있었다. 단순한 실수처럼
   보이지만, 이런 실수를 유발하는 구조 자체가 문제다.

  ### 오래된 클로저(Stale Closure) 문제

  `useEffect` 안에서 상태값을 참조할 때 발생하는 가장 클래식한
  버그다.

  ```javascript
  function MyUserInfo() {
    const [userName, setUserName] = useState("Bob");
    const [loginMessage, setLoginMessage] = useState("");

    useEffect(() => {
      let loggedInTime = 0;
      const interval = setInterval(() => {
        loggedInTime++;
        setLoginMessage(
          `${userName} has been logged in for ${loggedInTime}
  seconds` // ← 이름이 바뀌어도 "Bob" 고정
        );
      }, 1000);
      return () => clearInterval(interval);
    }, []); // 빈 의존성 배열 → 처음 캡처한 userName이 갱신되지
  않음

    // ...
  }
  ```

  `userName`이 바뀌어도 `setInterval` 안의 콜백은 처음 마운트 때
  캡처한 "Bob"을 계속 참조한다. 이게 오래된 클로저다.

  #### ❌ 의존성에 추가하면?

  ```javascript
  useEffect(() => {
    // ...
  }, [userName]); // userName 바뀔 때마다 interval이 리셋됨
  ```

  `userName`을 의존성에 넣으면 이름 바꿀 때마다 타이머가 처음부터
  다시 시작된다. 원하는 동작이 아니다.

  ### useRef로 임시방편

  ```javascript
  const nameRef = useRef(userName);
  nameRef.current = userName; // 매 렌더마다 최신값으로 갱신

  useEffect(() => {
    let loggedInTime = 0;
    const interval = setInterval(() => {
      loggedInTime++;
      setLoginMessage(
        `${nameRef.current} has been logged in for ${loggedInTime}
   seconds` // ref로 최신값 접근
      );
    }, 1000);
    return () => clearInterval(interval);
  }, []);
  ```

  동작은 한다. 근데 솔직히 보기 안 좋다. 상태값 하나 쓰려고 ref
  따로 선언하고 동기화까지 해야 하는 게 번거롭다.

  ### `useEffectEvent` — 진짜 해결책

  ```javascript
  const getName = useEffectEvent(() => userName); // 항상 최신
  userName 반환

  useEffect(() => {
    let loggedInTime = 0;
    const interval = setInterval(() => {
      loggedInTime++;
      setLoginMessage(
        `${getName()} has been logged in for ${loggedInTime}
  seconds`
      );
    }, 1000);
    return () => clearInterval(interval);
  }, []); // 의존성 배열 깔끔하게 유지
  ```

  `useEffectEvent`로 감싼 함수는 **항상 최신 state/prop을
  참조**하면서, 동시에 **의존성 배열에 포함시킬 필요가 없다**.
  이게 핵심이다.

  더 정리하면 이렇게 쓸 수 있다:

  ```javascript
  const onTick = useEffectEvent((tick: number) =>
    setLoginMessage(`${userName} has been logged in for ${tick}
  seconds`)
  );

  useEffect(() => {
    let ticks = 0;
    const interval = setInterval(() => onTick(++ticks), 1000);
    return () => clearInterval(interval);
  }, []);
  ```

  `onTick` 자체가 "이벤트 핸들러 역할"을 하면서 내부에서 상태를
  다룬다. `useEffect`는 순수하게 부수효과 실행 로직만 갖게 된다.

  ### 커스텀 훅으로 추상화

  `useEffectEvent`의 진가는 커스텀 훅 만들 때 드러난다.

  ```javascript
  function useInterval(onTick: (tick: number) => void) {
    const onTickEvent = useEffectEvent(onTick); // 콜백을 이벤트로
   변환

    useEffect(() => {
      let ticks = 0;
      const interval = setInterval(() => onTickEvent(++ticks),
  1000);
      return () => clearInterval(interval);
    }, []); // onTick이 바뀌어도 의존성 배열 건드릴 필요 없음
  }
  ```

  기존에는 커스텀 훅에 콜백을 넘기면 "이 함수가 바뀔 때 effect를
  재실행해야 하나?" 문제가 항상 따라붙었다. `useEffectEvent`로
  콜백을 감싸면 그 걱정이 사라진다.

  #### timeout 동적 변경까지

  ```javascript
  function useInterval(onTick: (tick: number) => void, timeout:
  number = 1000) {
    const onTickEvent = useEffectEvent(onTick);
    const getTimeout = useEffectEvent(() => timeout); // timeout도
   이벤트로

    useEffect(() => {
      let ticks = 0;
      let mounted = true;
      function tick() {
        if (mounted) {
          onTickEvent(++ticks);
          setTimeout(tick, getTimeout()); // 매번 최신 timeout
  값으로
        }
      }
      setTimeout(tick, getTimeout());
      return () => {
        mounted = false;
      };
    }, []);
  }
  ```

  `timeout`이 런타임에 바뀌어도 `getTimeout()`이 항상 최신값을
  물고 온다. 이전엔 이걸 구현하려면 ref 두 개 써야 했다.

  ---

  ### 📌 정리
  `useEffectEvent — 클로저와 의존성 배열의 갈등을 끊어내는 도구`

  #### 핵심

  > `useEffectEvent`로 감싼 함수는 항상 최신 상태를 참조하면서도
  `useEffect`의 의존성 배열에 포함할 필요가 없다. "렌더링 흐름에
  속하지 않는 로직"을 분리하는 것이 이 훅의 철학이다.

  #### ⭐️ 앞으로 기억할 것

  - 1. **"이 값이 바뀌어도 effect를 재실행하면 안 되는데, 최신값은
   써야 해" — 이게 `useEffectEvent` 신호다**
    `useEffect` 의존성에 넣기도 뭐하고 빼기도 찜찜한 상태값이
  생기면 `useEffectEvent`를 떠올리자.
    ```javascript
    // userName이 바뀌어도 interval은 재시작하면 안 될 때
    const getUser = useEffectEvent(() => userName);
    ```

  - 2. **ref 동기화 패턴의 공식 대체재**
    `const ref = useRef(x); ref.current = x;` 이 패턴이 보이면
  `useEffectEvent`로 교체할 수 있는지 확인한다.
    ```javascript
    // Before
    const cbRef = useRef(cb);
    cbRef.current = cb;
    useEffect(() => { ...; cbRef.current(); }, []);

    // After
    const onCb = useEffectEvent(cb);
    useEffect(() => { ...; onCb(); }, []);
    ```

  - 3. **커스텀 훅에 콜백 넘길 때 의존성 문제 해결**
    훅 내부에서 `useEffectEvent`로 콜백을 감싸면 호출부에서
  `useCallback` 안 써도 된다.
    ```javascript
    function usePolling(onData: (data: any) => void) {
      const onDataEvent = useEffectEvent(onData); // 의존성 걱정
  없이 최신 콜백 사용
      useEffect(() => {
        const id = setInterval(async () => {
          const data = await fetch('/api/data').then(r =>
  r.json());
          onDataEvent(data);
        }, 5000);
        return () => clearInterval(id);
      }, []);
    }
    ```

  > **📌 결론**
  > ▎ `useEffect`의 의존성 배열 문제는 "오래된 클로저"라는 JS 근본
   특성에서 비롯된다
  > ▎ `useRef` 동기화는 동작하지만 의도가 불명확하고
  보일러플레이트가 많다
  > ▎ `useEffectEvent`는 "이 로직은 렌더링 흐름이 아닌 이벤트
  타이밍에 속한다"는 의미를 코드에 명시적으로 표현한다
  > ▎ ⭐️ 아직 실험적 API이지만 패턴 자체(ref-sync 대체)는 지금도
  유효하므로 동작 방식을 익혀두면 React 컴파일러 시대에도 바로
  적용 가능하다
