# 리액트 애플리케이션의 보안

> **원문**: [Security in React Applications](https://certificates.dev/blog/security-in-react-applications) — Aurora Scharff
> **번역 출처**: [[번역] 리액트 애플리케이션의 보안](https://ricki-lee.medium.com/%EB%B2%88%EC%97%AD-%EB%A6%AC%EC%95%A1%ED%8A%B8-%EC%95%A0%ED%94%8C%EB%A6%AC%EC%BC%80%EC%9D%B4%EC%85%98%EC%9D%98-%EB%B3%B4%EC%95%88-04187bc2b431)

리액트를 쓰면 XSS는 알아서 막아준다고 막연히 믿고 있었다. 근데 정확히 어디까지 막아주고 어디서부터 내 책임인지는 한 번도 정리해본 적이 없다. 토큰은 그냥 `localStorage`에 넣어왔고. 이 글을 읽고 그 경계선을 확실히 그어보기로 했다.

### 리액트가 기본으로 막아주는 것 — JSX 자동 이스케이프

리액트는 JSX 표현식 안의 값을 DOM에 넣기 전에 자동으로 이스케이프한다. `<`, `>`, `&` 같은 특수문자를 HTML 엔티티로 바꿔버리기 때문에, 사용자 입력이 그대로 들어와도 태그가 아니라 **문자열**로 렌더링된다.

```jsx
function Comment({ text }) {
  return <p>{text}</p>;
}

<Comment text="<script>alert('XSS')</script>" />
// 화면엔 <script>alert('XSS')</script> 라는 "글자"가 찍힌다
// 스크립트는 실행되지 않는다
```

중요한 건 이게 children뿐 아니라 **props로 넘기는 값에도 똑같이 적용**된다는 것. 즉 JSX 표현식을 통과하는 값은 기본적으로 안전하다.

그래서 뒤집어 말하면, **JSX 표현식을 우회하는 지점부터가 위험 구간**이다.

#### dangerouslySetInnerHTML

이름에 `dangerously`가 붙은 건 농담이 아니다. 이 prop은 리액트의 자동 이스케이프를 완전히 건너뛰고 엘리먼트의 `innerHTML`을 직접 세팅한다.

```jsx
// ❌ 넘어온 html을 그대로 믿는다
function UnsafeComponent({ html }) {
  return <div dangerouslySetInnerHTML={{ __html: html }} />;
}

// html = '<img src="" onerror="alert(document.cookie)">' 라면
// onerror가 실행되고 쿠키가 털린다
```

여기서 눈여겨볼 건 공격 벡터가 `<script>` 태그가 아니라는 점이다. `<img>`의 `onerror` 같은 **인라인 이벤트 핸들러**로도 충분하다. 그래서 "script 태그만 걸러내면 되겠지" 하는 직접 필터링은 반드시 뚫린다.

CMS에서 내려온 HTML을 렌더링해야 하는 상황처럼 정말로 HTML을 그려야 할 때가 있다. 그럴 땐 반드시 **먼저 sanitize**한다.

```jsx
// ✅ 위험한 태그·이벤트 핸들러를 털어내고 넣는다
import DOMPurify from 'dompurify';

function BlogPost({ userContent }) {
  const cleanHTML = DOMPurify.sanitize(userContent);
  return <div dangerouslySetInnerHTML={{ __html: cleanHTML }} />;
}
```

`DOMPurify`가 하는 일이 딱 그거다. `<script>` 같은 태그와 `onerror`, `onclick` 같은 인라인 핸들러를 제거하는 것. 직접 정규식으로 막아보려는 시도는 하지 말자. 우회법이 무한하다.

### 안전한 인증과 CSRF

솔직히 이 부분이 제일 뜨끔했다.

#### 토큰 저장

인증 토큰을 `localStorage`나 `sessionStorage`에 넣으면 **그 페이지에서 실행되는 모든 자바스크립트가 읽을 수 있다.** 내 코드든, 서드파티 스크립트든, XSS로 주입된 코드든 전부.

```js
// ❌ XSS에 취약 — 페이지의 어떤 스크립트든 이걸 읽는다
localStorage.setItem('token', authToken);

// ❌ 같은 문제 — sessionStorage도 자바스크립트가 접근 가능하다
sessionStorage.setItem('token', authToken);
```

즉 XSS가 한 번 뚫리는 순간 토큰이 통째로 털린다. 위의 sanitize를 아무리 잘 해도 이건 **방어**의 문제가 아니라 **피해 범위**의 문제다.

#### 안전한 쿠키 속성

대신 서버가 `HttpOnly` 쿠키로 내려주게 한다.

```
Set-Cookie: token=...; HttpOnly; Secure; SameSite=Strict
```

| 속성 | 하는 일 |
| --- | --- |
| `HttpOnly` | 자바스크립트에서 `document.cookie`로 접근 불가 — XSS가 나도 토큰만은 못 읽는다 |
| `Secure` | HTTPS 연결에서만 전송 — 평문 네트워크에서 가로채기 방지 |
| `SameSite=Strict` | 크로스 사이트 요청에 쿠키를 붙이지 않음 — CSRF 방어 |

#### CSRF 방어

쿠키 인증을 켜면 CSRF가 따라온다. 브라우저가 요청에 쿠키를 **자동으로** 붙여주기 때문이다. 악성 사이트가 내 서버로 요청을 쏘면 사용자 인증이 딸려간다. 편의가 곧 취약점이다.

`SameSite=Strict`가 1차 방어선이고, 그 위에 CSRF 토큰을 얹는다. 쿠키는 자동으로 붙지만 **커스텀 헤더는 공격자가 크로스 오리진으로 붙일 수 없다**는 게 핵심 원리다.

```js
async function updateProfile(data) {
  const response = await fetch('/api/profile', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-CSRF-Token': getCsrfToken(), // 공격자는 이 헤더를 위조할 수 없다
    },
    credentials: 'include', // 쿠키를 함께 보낸다
    body: JSON.stringify(data),
  });
  return response.json();
}
```

### 서버 측 입력 검증

이건 머리로는 알고 있었는데, 문장으로 못 박아두는 게 좋겠다.

> 프론트엔드 검증은 UX를 위한 것이지 보안을 위한 것이 아니다.

공격자는 내 리액트 앱을 안 쓴다. `curl`로 API를 때린다. 클라이언트 검증은 전부 우회 가능하다.

서버에서 지켜야 할 순서는 세 단계다. **인가 확인 → 스키마 검증 → 파라미터화된 쿼리.**

```ts
'use server';
import { z } from 'zod';

const paymentSchema = z.object({
  email: z.string().email(),
  amount: z.number().positive(),
});

async function submitPayment(formData: FormData) {
  // 1. 이 유저가 이 작업을 할 권한이 있는가? (인증 ≠ 인가)
  const session = await getSession();
  if (!session?.user) {
    return { error: 'Unauthorized' };
  }

  // 2. 입력이 스키마에 맞는가?
  const result = paymentSchema.safeParse({
    email: formData.get('email'),
    amount: Number(formData.get('amount')),
  });

  if (!result.success) {
    return { error: result.error.flatten() };
  }

  // 3. 쿼리는 파라미터 바인딩으로 — 문자열 조합 금지
  await db.query(
    'INSERT INTO payments (user_id, email, amount) VALUES ($1, $2, $3)',
    [session.user.id, result.data.email, result.data.amount]
  );
}
```

1번을 빼먹기 쉽다. "로그인했으니까 통과"가 아니라 "이 사람이 이 리소스를 만질 수 있는가"까지 봐야 한다. 그리고 `user_id`를 클라이언트가 보낸 값이 아니라 **세션에서 꺼내 쓴다**는 것도 포인트다. 폼에서 온 `user_id`를 믿으면 남의 결제를 만들 수 있다.

3번, 쿼리를 문자열로 이어붙이는 순간 SQL 인젝션이다. 파라미터화된 쿼리는 값을 값으로만 취급하게 만든다.

### 콘텐츠 보안 정책 (CSP)

앞의 것들이 다 뚫렸다고 가정하고 까는 층이다. CSP는 페이지가 **어떤 출처의 리소스를 로드/실행할 수 있는지**를 브라우저에 선언해두는 것이다. 주입된 스크립트가 있어도 브라우저가 실행을 거부한다.

HTTP 헤더로 내려준다.

```
Content-Security-Policy:
  default-src 'self';
  script-src 'self';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  object-src 'none';
  frame-ancestors 'none'
```

`script-src 'self'`면 같은 출처의 스크립트만 실행된다. 인젝션된 인라인 `<script>`는 죽는다. `object-src 'none'`은 플래시류 플러그인을, `frame-ancestors 'none'`은 클릭재킹(내 페이지를 남의 iframe에 끼워넣는 것)을 막는다.

#### 인라인 스크립트를 위한 논스(nonce)

문제는 인라인 스크립트가 꼭 필요한 경우다. 이때 `'unsafe-inline'`을 열어버리면 CSP를 켠 의미가 사라진다. 대신 **요청마다 새로 만든 논스**를 쓴다.

```
// ❌ 모든 인라인 스크립트 허용 — CSP를 끈 것과 다름없다
Content-Security-Policy: script-src 'self' 'unsafe-inline'

// ✅ 이 요청에서 발급한 논스를 가진 스크립트만
Content-Security-Policy: script-src 'nonce-a1b2c3d4' 'strict-dynamic'
```

```html
<script nonce="a1b2c3d4" src="/static/js/main.js"></script>
```

논스는 **요청마다 새로 생성**해야 한다. 고정값이면 공격자가 그 값을 복사해서 자기 스크립트에 붙이면 그만이다. 그럴 거면 논스가 아니다.

`'strict-dynamic'`은 논스로 신뢰받은 스크립트가 동적으로 만든 스크립트까지 신뢰하겠다는 뜻이다. 번들러가 청크를 동적 로드하는 요즘 앱에선 사실상 필수.

#### 정책 테스트하기

CSP를 처음부터 강제로 켜면 멀쩡한 리소스까지 막혀서 앱이 깨진다. `Content-Security-Policy-Report-Only` 헤더로 먼저 켜면 **차단은 안 하고 위반만 리포트**해준다. 콘솔에서 뭐가 걸리는지 다 보고 나서 진짜 헤더로 전환하는 게 순서다.

### 마치며

리액트의 XSS 방어는 **기본값**이지 **보장**이 아니다. 우회 경로는 결국 내가 직접 만든다. 그리고 어느 한 겹도 단독으로 완전하지 않다. sanitize가 뚫려도 `HttpOnly`가 토큰을 지키고, 그것도 뚫려도 CSP가 스크립트 실행을 막는다. 겹겹이 쌓는 게 요점이다.

---

### 📌 정리

`리액트는 XSS의 일부만 막아준다. 나머지는 전부 내 몫이다.`

#### 핵심

> 리액트의 자동 이스케이프는 JSX 표현식을 통과하는 값에만 적용된다.
> `dangerouslySetInnerHTML`로 우회하는 순간, 토큰을 `localStorage`에 두는 순간, 검증을 클라이언트에만 두는 순간 방어는 사라진다.
> 보안은 한 겹이 아니라 여러 겹이다. sanitize → HttpOnly 쿠키 → 서버 검증 → CSP.

#### ⭐️ 앞으로 기억할 것

- 1. JSX를 벗어나는 지점만 감시하면 된다

리액트가 못 막는 곳은 정확히 `dangerouslySetInnerHTML`, `href={userInput}` 같은 URL 주입, 그리고 DOM 직접 조작이다. 코드 리뷰에서 이 셋만 찾아도 XSS 표면 대부분을 커버한다.

```jsx
<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(html) }} />
```

- 2. 공격은 `<script>`로만 오지 않는다

`<img onerror>` 하나면 충분하다. 그래서 직접 필터링은 답이 없고 검증된 sanitizer를 쓴다.

- 3. `localStorage`에 토큰을 넣는 건 XSS 한 방에 전부 잃겠다는 선언이다

`HttpOnly` 쿠키는 XSS가 나도 토큰만은 못 읽게 만든다. 방어 실패를 전제로 피해를 줄이는 설계.

```
Set-Cookie: token=...; HttpOnly; Secure; SameSite=Strict
```

- 4. 쿠키 인증을 켰으면 CSRF가 따라온다

브라우저가 쿠키를 자동으로 붙여주는 게 편의이자 취약점. `SameSite=Strict` + 커스텀 헤더(`X-CSRF-Token`)로 막는다. 공격자는 크로스 오리진에서 커스텀 헤더를 붙일 수 없다는 게 원리.

- 5. "인증했다"와 "권한이 있다"는 다른 질문이다

서버 검증 순서: 인가 확인 → 스키마 검증(Zod) → 파라미터화된 쿼리. 그리고 `user_id`는 폼이 아니라 **세션에서** 꺼낸다.

```ts
const session = await getSession();
if (!session?.user) return { error: 'Unauthorized' };
```

- 6. CSP의 논스는 요청마다 새로 발급해야 한다

고정 논스는 논스가 아니다. 그리고 `'unsafe-inline'`을 넣는 순간 CSP는 장식이 된다.

📌 결론

> **📌 결론**
> ▎ 리액트의 XSS 방어는 기본값이지 보장이 아니다. 우회 경로는 내가 직접 만든다.
> ▎ 토큰 저장 위치는 취향 문제가 아니라 피해 범위 문제다. `localStorage` → `HttpOnly` 쿠키.
> ▎ 클라이언트 검증은 UX, 서버 검증이 보안. 공격자는 내 UI를 쓰지 않는다.
> ▎ CSP는 "이미 뚫렸을 때"를 위한 층이다. 안 뚫릴 거라 믿고 생략하는 게 아니라, 뚫릴 거라 가정하고 까는 것.
>
> ▎ ⭐️ 실무 적용 포인트
> ▎ 코드베이스에서 `dangerouslySetInnerHTML` 전수 grep → sanitize 없는 곳 즉시 처리
> ▎ `localStorage.setItem('token'` 패턴 grep → 백엔드와 쿠키 전환 논의
> ▎ 서버 핸들러마다 "인가 체크가 있는가?" 한 줄 확인
> ▎ CSP는 `Report-Only`로 먼저 켜서 깨지는 곳 관찰 후 전환
