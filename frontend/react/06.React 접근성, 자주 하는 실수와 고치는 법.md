# React 접근성, 자주 하는 실수와 고치는 법

> 원문: [Accessibility in React: Common Mistakes and How to Fix Them](https://certificates.dev/blog/accessibility-in-react-common-mistakes-and-how-to-fix-them) — Aurora Scharff

a11y(accessibility)는 "잘 보이는 사람" 기준으로만 만들지 않는다는 얘기다. 키보드로만 탐색하는 사람, 스크린리더로 페이지를 듣는 사람, 200%로 확대해서 보는 사람, 애니메이션을 끈 사람 — 이 모두에게 동작하게 만드는 것. 기준은 [`WCAG`](https://www.w3.org/WAI/standards-guidelines/wcag/)인데, 충격적이게도 [웹사이트의 96%가 WCAG 위반](https://webaim.org/projects/million/)이 검출된다고 한다. 그만큼 고칠 게 많다는 뜻.

Aurora Scharff가 쓴 글이고, React 컴포넌트 코드에서 거의 매번 틀리는 패턴들 — 시맨틱 누락, 깨진 포커스, 라벨 없는 요소, 조용한 동적 업데이트 — 위주로 정리한다. 대부분 뭘 봐야 할지만 알면 금방 고친다.

### 올바른 HTML 요소를 써라

접근성에서 제일 큰 개선은 **그냥 맞는 요소를 고르는 것**이다. `<button>`은 포커스 가능하고, Enter/Space에 반응하고, 스크린리더에 자기 자신을 알린다. `<div onClick>`은 그 중 아무것도 안 한다.

```jsx
// ❌ 버튼처럼 보이지만 버튼이 아님
<div className="btn" onClick={handleClick}>Submit</div>

// ✅ 이게 버튼이다
<button onClick={handleClick}>Submit</button>
```

`<div>`로 버튼 흉내를 내려면 `role="button"`, `tabIndex={0}`, Enter/Space용 `onKeyDown`, 포커스 스타일까지 전부 직접 붙여야 한다. **`<button>`이 공짜로 주는 걸 재구현하는 짓.** 링크도 똑같다 — 이동은 `<a href>`, 액션은 `<button>`. 그룹 탐색은 `<nav>`, 본문은 `<main>`, 목록은 `<ul>`. 스크린리더는 이 요소들로 섹션 간 점프와 구조 파악을 한다.

#### 문서 구조와 heading 위계

스크린리더 유저는 **heading으로 페이지를 탐색한다.** 페이지의 모든 heading을 목록으로 뽑아 그 사이를 점프할 수 있어서, heading 위계가 곧 목차 역할을 한다.

`<h1>`(페이지 제목) → `<h2>`(주요 섹션) → `<h3>`(하위) 순서를 논리적으로 지킨다. **레벨 건너뛰기 금지(h1 → h3), 폰트 크기 보고 레벨 고르기 금지.** 스타일은 CSS로, 구조는 HTML로.

```jsx
// ❌ 시각적 크기 기준으로 heading 레벨 선택
<h1>My App</h1>
<h4>Welcome back, Jane</h4>
<h2>Recent Orders</h2>
<h4>Order #1234</h4>

// ✅ 논리적 위계
<h1>My App</h1>
<h2>Welcome back, Jane</h2>
<h2>Recent Orders</h2>
<h3>Order #1234</h3>
```

`<header>`, `<nav>`, `<main>`, `<aside>`, `<footer>` 같은 랜드마크도 페이지 레벨에서 같은 역할. 스크린리더가 랜드마크를 목록화해 바로 점프할 수 있다. React라고 룰이 달라지는 거 없다. JSX는 시맨틱을 바꾸지 않고 문법만 바꾼다(`htmlFor`, `className`).

### 상호작용 요소엔 전부 라벨을

모든 인터랙티브 요소는 **접근 가능한 이름(accessible name)** 이 필요하다. 폼 입력은 `<label>`:

```jsx
<label htmlFor="email-input">Email address</label>
<input id="email-input" type="email" />
```

> placeholder는 라벨이 아니다. 입력 시작하면 사라지니까.

아이콘만 있는 버튼은 아이콘 자체를 `aria-hidden="true"`로 숨기고(안 그러면 스크린리더가 SVG를 읽으려 해서 소음), 이름은 버튼에 단다:

```jsx
// aria-label 방식
<button aria-label="Close dialog" onClick={onClose}>
  <XIcon aria-hidden="true" />
</button>

// 시각적으로만 숨긴 텍스트 방식 (Tailwind sr-only)
<button onClick={onClose}>
  <XIcon aria-hidden="true" />
  <span className="sr-only">Close dialog</span>
</button>
```

이미지/링크도 같은 문제. 모든 `<img>`엔 `alt`가 필요하다. 정보 전달용이면 내용을 묘사, 장식용이면 빈 문자열로 스킵:

```jsx
<img src="/product.jpg" alt="Red running shoes, side view" /> {/* 정보 */}
<img src="/divider.svg" alt="" />                            {/* 장식 */}
```

> `alt` 속성이 아예 없는 것과 `alt=""`는 다르다. 속성이 없으면 일부 스크린리더가 **파일명을 읽는다**(거의 쓸모없음). alt 누락은 전체 웹사이트의 절반 이상에서 나타난다.

이미지/아이콘만 든 링크는 스크린리더에 "빈 링크"로 보인다:

```jsx
// ❌ 스크린리더가 그냥 "link"라고만 읽음
<a href="/profile"><UserIcon aria-hidden="true" /></a>

// ✅ aria-label
<a href="/profile" aria-label="Your profile"><UserIcon aria-hidden="true" /></a>
```

이미 DOM 어딘가에 라벨이 있으면 `aria-labelledby`로 가리킨다:

```jsx
<h2 id="cart-heading">Shopping Cart</h2>
<ul aria-labelledby="cart-heading">
  {items.map(item => <li key={item.id}>{item.name}</li>)}
</ul>
```

#### 라벨·에러 연결은 `useId`로

input과 label을 잇는 ID는 문자열 하드코딩 대신 `useId`. SSR과 같은 컴포넌트 다중 인스턴스에서도 안정적이고 유니크한 ID를 만들어준다. 폼 에러도 같은 패턴이다. **빨간 에러 메시지를 옆에 띄우는 것만으론 부족하다** — 프로그래밍적으로 input과 연결 안 되면 스크린리더 유저는 필드로 돌아왔을 때 그걸 못 듣는다. `aria-describedby`로 에러를 잇고, `aria-invalid`로 필드를 표시:

```jsx
function TextField({ label, error }) {
  const id = useId();
  const errorId = `${id}-error`;
  return (
    <div>
      <label htmlFor={id}>{label}</label>
      <input
        id={id}
        aria-invalid={!!error}
        aria-describedby={error ? errorId : undefined}
      />
      <p id={errorId} role="alert">{error || ''}</p>
    </div>
  );
}
```

> 중요: `role="alert"` 컨테이너는 에러가 없어도 DOM에 **계속 남겨둔다.** 스크린리더는 "이미 존재하는" live region의 내용 변화를 감시한다. 그래서 요소 자체를 조건부 렌더링(`{error && <p role="alert">...}`)하면 처음 나타날 때 많은 스크린리더가 announce를 안 한다. **컨테이너는 마운트해두고 내용만 갱신.**

### 포커스를 관리하라

전통적 웹사이트는 새 페이지로 가면 포커스가 맨 위로 리셋된다. **React SPA는 풀 로드 없이 콘텐츠만 갈아끼우니 포커스가 그대로 남는다** — 키보드/스크린리더 유저가 이미 사라진 요소에 포커스된 상태가 될 수 있다. 페이지가 크게 바뀌면 새 콘텐츠로 포커스를 옮긴다:

```jsx
function PageContent({ title, children }) {
  const headingRef = useRef<HTMLHeadingElement>(null);
  useEffect(() => { headingRef.current?.focus(); }, [title]);
  return (
    <main>
      <h1 ref={headingRef} tabIndex={-1}>{title}</h1>
      {children}
    </main>
  );
}
```

> `tabIndex={-1}` = tab 순서엔 안 넣으면서 프로그래밍적으로만 포커스 가능하게.

모달도 같다. 열리면 포커스가 안으로 들어가고, 닫히면 트리거로 돌아와야 한다. 네이티브 `<dialog>`는 포커스 트랩 + Escape 닫기까지 거의 자동 처리:

```jsx
function Modal({ isOpen, onClose, title, children }) {
  const dialogRef = useRef<HTMLDialogElement>(null);
  const titleId = useId();
  useEffect(() => {
    if (isOpen) dialogRef.current?.showModal();
    else dialogRef.current?.close();
  }, [isOpen]);
  return (
    <dialog ref={dialogRef} onClose={onClose} aria-labelledby={titleId}>
      <h2 id={titleId}>{title}</h2>
      {children}
      <button onClick={onClose}>Close</button>
    </dialog>
  );
}
```

`showModal()`이 포커스 트랩 + 백드롭 + Escape를 다 해준다. `onClose`는 어떤 방식으로 닫혀도 발화되니 React state가 동기화된다.

요소 제거 시에도 포커스가 중요하다. 포커스된 요소가 제거되면 포커스가 `<body>`로 튀어 유저가 위치를 완전히 잃는다:

```jsx
function TodoList({ items, onDelete }) {
  const listRef = useRef<HTMLUListElement>(null);
  function handleDelete(id) {
    onDelete(id);
    listRef.current?.focus(); // 삭제 후 리스트로 포커스 회수
  }
  return (
    <ul ref={listRef} tabIndex={-1} aria-label="Todo items">
      {items.map(item => (
        <li key={item.id}>
          {item.text}
          <button aria-label={`Delete ${item.text}`} onClick={() => handleDelete(item.id)}>
            Delete
          </button>
        </li>
      ))}
    </ul>
  );
}
```

접거나 펴는 섹션, 탭 패널 등 콘텐츠가 나타났다 사라지는 모든 UI에 같은 원칙.

> 중첩 모달, 위치 지정 팝오버, 화살표 키 메뉴처럼 복잡한 오버레이는 직접 만들면 금방 지옥이다. 커스텀 select 하나만 해도 화살표 네비게이션, Escape 닫기, 포커스 트랩, `aria-expanded`, `aria-activedescendant` 등이 필요하다. **잘못 구현한 건 없느니만 못하다.** 이럴 때 `Ariakit`, `React Aria`, `Radix`, `Base UI` 같은 접근성 라이브러리 — 스타일 없는 접근성 프리미티브를 준다. 스타일은 내가, 동작은 걔네가.

### 동적 업데이트를 announce 하라

토스트, 검증 에러, 상태 메시지처럼 풀 리로드 없이 일부만 갱신되는 것들 — 보는 사람은 알지만 스크린리더 유저는 브라우저가 알려주지 않으면 모른다. 에러는 `role="alert"`, 덜 급한 업데이트는 `role="status"`. 여기서도 **컨테이너는 DOM에 두고 내용만 갱신**:

```jsx
<div role="alert">{errorMessage}</div>
<div role="status">{statusMessage}</div>
```

> `role="alert"`는 스크린리더를 즉시 끊고 알린다. `role="status"`는 지금 읽던 걸 끝낼 때까지 기다린다. `alert`는 에러/중대 정보에만 아껴 써라.

### ARIA / data 속성으로 스타일링

올바른 ARIA 속성을 쓰면 좋은 점: **그걸로 스타일을 줄 수 있다.** 시각 스타일용 별도 state를 따로 관리하지 말고, 실제 접근성 상태를 스타일이 반영하게:

```jsx
// ❌ 스타일용 isActive state 따로 추적
<li className={isActive ? 'bg-blue-100' : ''}>...</li>

// ✅ 이미 있는 ARIA 속성 기반
<li className="aria-selected:bg-blue-100">...</li>

// 또는 라이브러리가 주는 data 속성
<li className="data-[active-item]:bg-blue-100">...</li>
```

`aria-disabled`, `aria-expanded`, `aria-selected`를 CSS 속성 선택자나 Tailwind 모디파이어로 직접 스타일. `data-focus-visible`로 마우스 포커스 vs 키보드 포커스 구분도 가능. 접근성 라이브러리들이 이런 속성을 다 노출해줘서, 내부 state 관리 없이 스타일만 입히면 되는 게 실무 이점이다.

### 📌 정리
`맞는 HTML 요소 + 라벨 + 포커스 + announce, 이 4개면 8할은 잡는다`

#### 핵심
> React 접근성 실수는 대부분 4가지 — ① 시맨틱 요소 안 씀(div onClick) ② 라벨/alt 누락 ③ SPA 라우팅·조건부 렌더에서 포커스 유실 ④ 동적 업데이트를 스크린리더가 모름. 고치는 법도 정해져 있다: 올바른 태그 쓰고, accessible name 붙이고, 변화 시 포커스를 옮기고, live region(`role="alert"`/`status`)은 마운트한 채 내용만 갱신. 복잡한 위젯은 직접 만들지 말고 Radix/React Aria 같은 라이브러리.

#### ⭐️ 앞으로 기억할 것
- 1. **`<div onClick>` 발견 즉시 `<button>`/`<a>`로**
  포커스·키보드·스크린리더 announce를 공짜로 받는다. div로 흉내내면 role+tabIndex+onKeyDown+포커스스타일 다 직접.
- 2. **placeholder는 라벨이 아니다 / `alt` 없음 ≠ `alt=""`**
  input엔 `<label htmlFor>`, 장식 이미지엔 `alt=""`. 속성 없으면 스크린리더가 파일명을 읽음.
- 3. **아이콘은 `aria-hidden="true"`, 이름은 버튼/링크에**
  `aria-label` 또는 `sr-only` 텍스트. 아이콘만 든 링크는 "빈 링크"로 읽힘.
- 4. **live region은 조건부 렌더링하지 말 것**
  ```jsx
  // ❌ 처음 나타날 때 announce 안 됨
  {error && <p role="alert">{error}</p>}
  // ✅ 컨테이너 유지, 내용만
  <p role="alert">{error || ''}</p>
  ```
- 5. **포커스 회수 3대 타이밍**
  ① SPA 라우트 변경 → 새 `<h1 tabIndex={-1}>`로 `.focus()` ② 모달 open/close → 네이티브 `<dialog>`+`showModal()` ③ 요소 삭제 → 리스트/다음 항목으로 포커스.
- 6. **에러는 `aria-describedby`+`aria-invalid`로 input과 연결**
  ID는 `useId()`로. 빨간 글씨만으론 스크린리더가 못 들음.
- 7. **ID 하드코딩 금지 → `useId()`**
  SSR·다중 인스턴스에서 안정적·유니크.
- 8. **복잡한 위젯(select/메뉴/중첩 모달)은 라이브러리**
  Ariakit / React Aria / Radix / Base UI. "잘못 만든 접근성은 없느니만 못하다."

📌 결론
▎ 접근성의 80%는 "맞는 태그 + 라벨"이라는 기본기에서 나온다 — 화려한 ARIA보다 시맨틱 HTML 먼저.
▎ React 특유의 함정은 포커스(SPA 라우팅·조건부 렌더)와 동적 업데이트(live region) 두 곳에 몰려 있다.
▎ ARIA 속성을 제대로 박으면 스타일링(`aria-selected:`, `data-[...]`)까지 따라오는 보너스.
▎ ⭐️ 실무 적용 포인트 (Quick Audit)
▎ → onClick 달린 div/span 찾아 button/a로 교체
▎ → 모든 input에 label, 모든 img에 alt, 모든 아이콘버튼에 이름
▎ → 폼 에러 ↔ input 연결(aria-describedby/invalid)
▎ → 모달 포커스 진입/복귀/Escape, 삭제 시 포커스 행방 점검
▎ → 토스트/상태 메시지에 role="alert" / "status"
▎ → 전체 Tab 순회 + axe-core 자동 검사. 폼·자주 쓰는 컴포넌트부터.
