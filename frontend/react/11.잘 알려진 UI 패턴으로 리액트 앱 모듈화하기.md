# 잘 알려진 UI 패턴으로 리액트 앱 모듈화하기

> 원문: [Modularizing React Applications with Established UI Patterns](https://martinfowler.com/articles/modularizing-react-apps.html) — Juntao Qiu (martinfowler.com)
> 번역/정리 출처: [번역글 — eunbinn](https://velog.io/@eunbinn/modularizing-react-apps)

리액트로 일하다 보면 `useEffect` 안에 fetch 때려넣고, 받아온 데이터를 거기서 바로 가공하고, 렌더링까지 한 컴포넌트에서 다 처리하게 되는 경우가 많다. 처음엔 빠른데 컴포넌트가 커지면 한 파일 안에서 추상화 레벨이 마구 섞여서 읽기가 괴롭다. 이 글은 "리액트는 그냥 뷰 만드는 라이브러리일 뿐"이라는 전제에서 출발해서, 백엔드에서나 익숙하던 계층 분리(presentation / domain / data)를 프론트에 어떻게 단계적으로 적용하는지 결제 기능 예제로 보여준다. 솔직히 읽으면서 "아 내가 짜던 게 딱 이 안티패턴이었네" 싶었다.

### 출발점: 리액트는 겸손한 뷰 라이브러리다

리액트 공식 설명이 "A JavaScript library for building user interfaces"인 데는 이유가 있다. 데이터 fetching, 응답 가공, 비즈니스 계산을 죄다 컴포넌트(특히 `useEffect`)에 욱여넣지 말라는 것. 흔히 보는 그 패턴:

```tsx
useEffect(() => {
  fetch("https://address.service/api")
    .then((res) => res.json())
    .then((data) => {
      // 서버 응답을 받자마자 로컬 포맷으로 매핑/변환하는 로직이 여기 섞인다
      const addresses = data.map((item) => ({
        street: item.streetName,
        address: item.streetAddress,
        postcode: item.postCode,
      }));

      setAddresses(addresses);
    });
}, []);

// 실제 렌더링은 아래에...
```

현실의 프론트 앱은 라우터, 로컬스토리지, 캐싱, 네트워크 요청, 서드파티 연동, 보안, 로깅, 성능 최적화까지 다 끌어안는다. 이걸 전부 컴포넌트/훅에 밀어넣으면 추상화 레벨이 섞여서 혼란만 커진다. 마틴 파울러의 말을 빌리면, 계층을 나누는 가장 큰 이점은 **뷰 / 모델 / 데이터 세 주제를 서로 독립적으로 생각할 수 있게 해줘서 집중도가 올라간다**는 것이다.

---

### 리액트 앱이 진화하는 5단계

글의 큰 그림. 작은 앱에서 출발해서 계층형으로 가는 흐름이다.

#### 1. 단일 컴포넌트 앱

데이터 순회, 서드파티 컴포넌트 설정, 렌더링이 전부 컴포넌트 하나에. 아주 작은 앱이면 굴러는 가는데 인지 부하가 금방 쌓인다.

![evolution-1](https://martinfowler.com/articles/modularizing-react-apps/evolution-1.png)

#### 2. 다중 컴포넌트 앱

HTML 구조에 맞춰 컴포넌트를 쪼갠다. 컴포넌트 하나에 집중할 수 있게 되지만, 네트워크 요청·데이터 변환·상태 관리는 여전히 여기저기 흩어져 있다.

![evolution-2](https://martinfowler.com/articles/modularizing-react-apps/evolution-2.png)

#### 3. 훅으로 상태 관리 분리

상태 로직과 사이드이펙트를 커스텀 훅으로 빼낸다. 컴포넌트는 렌더링에 집중, 로직은 재사용 가능해진다.

![evolution-3](https://martinfowler.com/articles/modularizing-react-apps/evolution-3.png)

#### 4. 비즈니스 모델의 등장

데이터 매핑, null 체크, fallback 값 같은 걸 뷰와 무관한 도메인 객체로 캡슐화한다. 여기서 상속·다형성 같은 패턴이 자연스럽게 나온다.

![evolution-4](https://martinfowler.com/articles/modularizing-react-apps/evolution-4.png)

#### 5. 계층형 프론트엔드 앱

presentation / domain / data 세 계층으로 분리. 각 계층의 관심사를 독립적으로 생각할 수 있다. 전통적인 Presentation-Domain-Data 패턴 그대로다.

![evolution-5](https://martinfowler.com/articles/modularizing-react-apps/evolution-5.png)

## 📌 실전: 결제 기능 단계별 리팩토링

### 0단계 — 모든 게 한 컴포넌트에 (안티패턴)

리액트 막 배운 사람이 흔히 짜는 형태. 네트워크 초기화 + 데이터 변환 + 렌더링이 한 덩어리.

```tsx
export const Payment = ({ amount }: { amount: number }) => {
  const [paymentMethods, setPaymentMethods] = useState<LocalPaymentMethod[]>([]);

  useEffect(() => {
    const fetchPaymentMethods = async () => {
      const url = "https://online-ordering.com/api/payment-methods";

      const response = await fetch(url);
      const methods: RemotePaymentMethod[] = await response.json();

      if (methods.length > 0) {
        // 서버 포맷 → 로컬 포맷 변환
        const extended: LocalPaymentMethod[] = methods.map((method) => ({
          provider: method.name,
          label: `Pay with ${method.name}`,
        }));
        extended.push({ provider: "cash", label: "Pay in cash" }); // 현금 옵션 추가
        setPaymentMethods(extended);
      } else {
        setPaymentMethods([]);
      }
    };

    fetchPaymentMethods();
  }, []);

  return (
    <div>
      <h3>Payment</h3>
      <div>
        {paymentMethods.map((method) => (
          <label key={method.provider}>
            <input
              type="radio"
              name="payment"
              value={method.provider}
              defaultChecked={method.provider === "cash"}
            />
            <span>{method.label}</span>
          </label>
        ))}
      </div>
      <button>${amount}</button>
    </div>
  );
};
```

이 컴포넌트 하나를 이해하려면 ① 네트워크 요청 초기화 방법 ② 서버 데이터를 로컬 포맷으로 매핑하는 법 ③ 결제수단 하나하나 렌더링하는 법 ④ Payment 자체의 렌더링 로직 — **이 네 개를 동시에 머리에 올려야** 한다. 그게 문제다.

![payment-methods](https://martinfowler.com/articles/modularizing-react-apps/payment-methods.png)

### 1단계 — 뷰가 아닌 코드를 훅으로 추출

상태 + 사이드이펙트를 `usePaymentMethods` 훅으로 빼낸다.

```tsx
const usePaymentMethods = () => {
  const [paymentMethods, setPaymentMethods] = useState<LocalPaymentMethod[]>([]);

  useEffect(() => {
    const fetchPaymentMethods = async () => {
      const url = "https://online-ordering.com/api/payment-methods";

      const response = await fetch(url);
      const methods: RemotePaymentMethod[] = await response.json();

      if (methods.length > 0) {
        const extended: LocalPaymentMethod[] = methods.map((method) => ({
          provider: method.name,
          label: `Pay with ${method.name}`,
        }));
        extended.push({ provider: "cash", label: "Pay in cash" });
        setPaymentMethods(extended);
      } else {
        setPaymentMethods([]);
      }
    };

    fetchPaymentMethods();
  }, []);

  return { paymentMethods };
};

export const Payment = ({ amount }: { amount: number }) => {
  const { paymentMethods } = usePaymentMethods(); // 컴포넌트는 이제 깔끔

  return (
    <div>

      <div>

          </label>
        ))}
      </div>
      <button>${amount}</button>
    </div>
  );
};
```

### 2단계 — 서브 컴포넌트로 뷰 분리

결제수단 순회 렌더링 부분을 순수 함수 컴포넌트로 뽑는다.

```tsx
const PaymentMethods = ({

}) => (

          name="payment"
          value={method.provider}
          defaultChecked={method.provider === "cash"}
        />
        <span>{method.label}</span>
      </label>
    ))}
  </>
);

export const Payment = ({ amount }: { amount: number }) => {
  const { paymentMethods } = usePaymentMethods();

  return (
    <div>
      <h3>Payment</h3>
      <PaymentMethods paymentMethods={paymentMethods} />
      <button>${amount}</button>
    </div>
  );
};
```

### 3단계 — 도메인 모델로 로직 캡슐화

`label` 만드는 규칙(현금이면 "Pay in", 아니면 "Pay with"), 기본 결제수단 판별 같은 **비즈니스 규칙**을 클래스로 옮긴다.

```ts
// src/models/PaymentMethod.ts
class PaymentMethod {
  private remotePaymentMethod: RemotePaymentMethod;

  constructor(remotePaymentMethod: RemotePaymentMethod) {
    this.remotePaymentMethod = remotePaymentMethod;
  }

  get provider() {
    return this.remotePaymentMethod.name;
  }

  get label() {
    if (this.provider === "cash") {
      return `Pay in ${this.provider}`;
    }
    return `Pay with ${this.provider}`;
  }

  get isDefaultMethod() {
    return this.provider === "cash";
  }
}
```

훅은 변환을 별도 함수로 빼고 모델을 쓰도록:

```ts
const payInCash = new PaymentMethod({ name: "cash" });

const convertPaymentMethods = (methods: RemotePaymentMethod[]) => {
  if (methods.length === 0) {
    return [];
  }

  const extended: PaymentMethod[] = methods.map(
    (method) => new PaymentMethod(method)
  );
  extended.push(payInCash);

  return extended;
};

export const usePaymentMethods = () => {
  const [paymentMethods, setPaymentMethods] = useState<PaymentMethod[]>([]);

  useEffect(() => {
    const fetchPaymentMethods = async () => {
      const url = "https://online-ordering.com/api/payment-methods";

      const response = await fetch(url);
      const methods: RemotePaymentMethod[] = await response.json();

      setPaymentMethods(convertPaymentMethods(methods));
    };

    fetchPaymentMethods();
  }, []);

  return { paymentMethods };
};
```

프레젠테이션 컴포넌트는 이제 도메인 객체에만 의존한다. `method.provider === "cash"` 같은 판단도 `method.isDefaultMethod`로 넘어갔다.

```tsx
export const PaymentMethods = ({ options }: { options: PaymentMethod[] }) => (
  <>
    {options.map((method) => (
      <label key={method.provider}>
        <input
          type="radio"
          name="payment"
          value={method.provider}
          defaultChecked={method.isDefaultMethod}
        />
        <span>{method.label}</span>
      </label>
    ))}
  </>
);
```

## 📌 새 요구사항: 자선 기부 (반올림)

결제 금액을 올림해서 차액을 기부하는 체크박스가 추가된다.

![round-up](https://martinfowler.com/articles/modularizing-react-apps/round-up.png)

### 4단계 — 기부 상태를 훅으로

`useRoundUp`이 기부 동의 상태와 총액/팁 계산을 담당.

```ts
// src/hooks/useRoundUp.ts
export const useRoundUp = (amount: number) => {
  const [agreeToDonate, setAgreeToDonate] = useState<boolean>(false);

  const { total, tip } = useMemo(
    () => ({
      total: agreeToDonate ? Math.floor(amount + 1) : amount,
      tip: parseFloat((Math.floor(amount + 1) - amount).toPrecision(10)),
    }),
    [amount, agreeToDonate]
  );

  const updateAgreeToDonate = () => {
    setAgreeToDonate((agreeToDonate) => !agreeToDonate);
  };

  return { total, tip, agreeToDonate, updateAgreeToDonate };
};
```

```tsx
export const Payment = ({ amount }: { amount: number }) => {
  const { paymentMethods } = usePaymentMethods();
  const { total, tip, agreeToDonate, updateAgreeToDonate } = useRoundUp(amount);

  return (
    <div>
      <h3>Payment</h3>
      <PaymentMethods options={paymentMethods} />
      <div>
        <label>
          <input type="checkbox" onChange={updateAgreeToDonate} checked={agreeToDonate} />
          <p>{formatCheckboxLabel(agreeToDonate, tip)}</p>
        </label>
      </div>
      <button>${total}</button>
    </div>
  );
};

const formatCheckboxLabel = (agreeToDonate: boolean, tip: number) => {
  return agreeToDonate
    ? "Thanks for your donation."
    : `I would like to donate $${tip} to charity.`;
};
```

### 산탄총 수술(shotgun surgery) 문제

일본(100엔 단위 올림), 덴마크(10크로네 단위 올림)로 확장하면서 나라별 반올림 규칙이 생긴다. 단순하게 가면 나라별 분기 로직이 **뷰·훅·유틸 여러 파일에 흩어진다**. 작은 변경 하나에 여기저기 다 손대야 하는 전형적인 산탄총 수술 냄새.

![shotgun-surgery](https://martinfowler.com/articles/modularizing-react-apps/shotgun-surgery.png)

### 5단계 — 다형성(전략 패턴)으로 해결

나라별 동작을 전략으로 캡슐화한다. 올림 알고리즘을 함수로 두고, `CountryPayment`가 통화기호 + 알고리즘을 들고 있는 구조.

```ts
// src/models/CountryPayment.ts
export type RoundUpStrategy = (amount: number) => number;

export const roundUpToNearestInteger: RoundUpStrategy = (amount: number) =>
  Math.floor(amount + 1);

export const roundUpToNearestHundred: RoundUpStrategy = (amount: number) =>
  Math.floor(amount / 100 + 1) * 100;

export const roundUpToNearestTen: RoundUpStrategy = (amount: number) =>
  Math.floor(amount / 10 + 1) * 10;

const calculateTipFor =
  (roundUpFn: (amount: number) => number) =>
  (amount: number) =>
    parseFloat((roundUpFn(amount) - amount).toPrecision(10));

export class CountryPayment {
  private readonly _currencySign: string;
  private readonly algorithm: RoundUpStrategy;

  public constructor(currencySign: string, roundUpAlgorithm: RoundUpStrategy) {
    this._currencySign = currencySign;
    this.algorithm = roundUpAlgorithm;
  }

  get currencySign(): string {
    return this._currencySign;
  }

  getRoundUpAmount(amount: number): number {
    return this.algorithm(amount);
  }

  getTip(amount: number): number {
    return calculateTipFor(this.getRoundUpAmount.bind(this))(amount);
  }
}
```

![strategy-pattern](https://martinfowler.com/articles/modularizing-react-apps/strategy-pattern.png)

훅은 전략을 주입받기만 한다:

```ts
// src/hooks/useRoundUp.ts
export const useRoundUp = (amount: number, strategy: CountryPayment) => {
  const [agreeToDonate, setAgreeToDonate] = useState<boolean>(false);

  const { total, tip } = useMemo(
    () => ({
      total: agreeToDonate ? strategy.getRoundUpAmount(amount) : amount,
      tip: strategy.getTip(amount),
    }),
    [agreeToDonate, amount, strategy]
  );

  const updateAgreeToDonate = () => {
    setAgreeToDonate((agreeToDonate) => !agreeToDonate);
  };

  return { total, tip, agreeToDonate, updateAgreeToDonate };
};
```

컴포넌트는 기본 전략만 default prop으로 받고 나머진 그대로:

```tsx
export const Payment = ({
  amount,
  strategy = new CountryPayment("$", roundUpToNearestInteger),
}: {
  amount: number;
  strategy?: CountryPayment;
}) => {
  const { paymentMethods } = usePaymentMethods();
  const { total, tip, agreeToDonate, updateAgreeToDonate } = useRoundUp(amount, strategy);

  return (
    <div>
      <h3>Payment</h3>
      <PaymentMethods options={paymentMethods} />
      <DonationCheckbox
        onChange={updateAgreeToDonate}
        checked={agreeToDonate}
        content={formatCheckboxLabel(agreeToDonate, tip, strategy)}
      />
      <button>{formatButtonLabel(strategy, total)}</button>
    </div>
  );
};
```

```ts
// src/utils.ts
export const formatCheckboxLabel = (
  agreeToDonate: boolean,
  tip: number,
  strategy: CountryPayment
) => {
  return agreeToDonate
    ? "Thanks for your donation."
    : `I would like to donate ${strategy.currencySign}${tip} to charity.`;

  return `${strategy.currencySign}${total}`;

```

> 새 나라가 추가돼도 이제 손댈 곳은 "전략 하나 더 만들기"뿐이다. 산탄총 수술 → 한 곳 수정으로 바뀌었다.

### 6단계 — 네트워크 클라이언트 추출 (data 계층)

fetch + 변환을 훅 밖, 별도 클라이언트로 뺀다. 리액트가 아니어도 재사용 가능해진다.

```ts
// src/clients/PaymentMethodClient.ts
export interface RemotePaymentMethod {
  name: string;
}

export const fetchPaymentMethods = async (): Promise<PaymentMethod[]> => {
  const response = await fetch("https://online-ordering.com/api/payment-methods");
  const methods: RemotePaymentMethod[] = await response.json();

  return convertPaymentMethods(methods);
};
```

훅은 이제 정말 얇아진다:

```ts
// src/hooks/usePaymentMethods.ts
export const usePaymentMethods = () => {
  const [paymentMethods, setPaymentMethods] = useState<PaymentMethod[]>([]);

  useEffect(() => {
    fetchPaymentMethods().then((methods) => setPaymentMethods(methods));
  }, []);

  return { paymentMethods };
};
```

### 이렇게 계층을 나누면 좋은 점

1. **유지보수성** — 버그가 특정 모듈에 갇혀서 수정 시 리스크가 작다.
2. **모듈성** — 재사용·조합이 쉬워진다.
3. **가독성** — 로직 흐름이 명확해서 온보딩이 빨라진다.
4. **확장성** — 모듈당 복잡도가 낮아 기능 추가가 시스템 전체에 파장을 안 준다.
5. **기술 스택 전환** — 도메인 로직이 뷰 프레임워크와 분리돼 있어 뷰 라이브러리 교체가 현실적으로 가능.

```
├── components/   # 순수 프레젠테이션 (PaymentMethods, DonationCheckbox)
├── hooks/        # 상태 관리 (usePaymentMethods, useRoundUp)
├── models/       # 도메인 로직 (PaymentMethod, CountryPayment)
├── clients/      # 데이터 접근 (PaymentMethodClient)
└── utils.ts      # 포맷팅 헬퍼
```

### 📌 정리
`리액트는 뷰일 뿐, 로직은 계층으로 내려라`

#### 핵심
> 프론트엔드도 일반 소프트웨어와 다를 게 없다. 관심사 분리·계층화·다형성 같은 전통적 설계 패턴이 그대로 유효하다. 핵심은 ① 코드의 역할을 분석하고 ② 뷰 / 비-뷰를 가르고 ③ 남은 로직을 책임별로 또 가르는 것. 그러면 뷰 걱정 없이 도메인 로직을 바꿀 수 있고, 도메인 로직 재사용성이 올라가며, 계층 간 결합이 줄어든다.

#### ⭐️ 앞으로 기억할 것
- 1. `useEffect` 안은 "사이드이펙트 트리거"만
  fetch·변환·계산을 다 넣지 말 것. 변환은 클라이언트(data 계층), 계산 규칙은 모델(domain 계층)로.
  ```ts
  // ❌ 훅 안에서 변환까지
  useEffect(() => { fetch(url).then(r=>r.json()).then(d=> setX(d.map(...))) }, []);
  // ✅ 변환은 클라이언트가, 훅은 받아서 상태만
  useEffect(() => { fetchPaymentMethods().then(setPaymentMethods) }, []);
  ```
- 2. 프레젠테이션 컴포넌트는 도메인 객체에만 의존
  `method.provider === "cash"` 같은 판단을 JSX에 두지 말고 `method.isDefaultMethod`로. 뷰는 "어떻게 보여줄지"만 안다.
- 3. 흩어지는 분기(shotgun surgery)는 전략 패턴 신호
  나라별/타입별 if-else가 여러 파일로 번지면 → 동작을 클래스/함수 전략으로 캡슐화하고 주입. 새 케이스는 "전략 하나 추가"로 끝난다.
  ```ts
  export const useRoundUp = (amount: number, strategy: CountryPayment) => { /* strategy.getRoundUpAmount(amount) */ };
  ```
- 4. 계층은 단계적으로 등장한다
  처음부터 5계층 만들지 말 것. 단일 컴포넌트 → 다중 → 훅 → 모델 → 계층형. 복잡도가 올라올 때마다 한 단계씩 추출하는 게 자연스럽다.
- 5. 도메인 로직을 뷰에서 떼면 뷰 교체가 가능해진다
  models/clients가 리액트를 모르면, 나중에 뷰 라이브러리를 바꿔도 도메인은 그대로 산다.

📌 결론
▎ 리액트 컴포넌트/훅에 모든 걸 욱여넣는 건 추상화 레벨을 섞어서 인지 부하를 키운다.
▎ presentation(컴포넌트) / domain(모델) / data(클라이언트) 세 계층으로 책임을 가르면 각각을 독립적으로 사고할 수 있다.
▎ 리팩토링 순서: 훅 추출 → 서브 컴포넌트 추출 → 도메인 모델 캡슐화 → 전략 패턴 → 네트워크 클라이언트 분리.
▎ ⭐️ 실무 적용 포인트
  - 새 PR 짤 때 "이 로직이 뷰냐, 도메인이냐, 데이터냐" 한 번 자문하고 위치 정하기.
  - if-else가 파일을 넘나들기 시작하면 그게 전략 패턴 도입 타이밍.
  - 모델/클라이언트 폴더에는 리액트 import가 없어야 정상.
