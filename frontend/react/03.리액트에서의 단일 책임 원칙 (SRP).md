# 리액트에서의 단일 책임 원칙 (SRP)

> **원문**: https://cekrem.github.io/posts/single-responsibility-principle-in-react
> **번역 출처**: https://velog.io/@eunbinn/single-responsibility-principle-in-react

![](https://velog.velcdn.com/images/chromeheartz/post/7f1832e6-3106-466b-837e-3a0b11721b7c/image.png)

`SRP` 를 React에 어떻게 적용하는지 잘 정리된 글을 발견해서 읽었다. 컴포넌트 분리를 코드로
단계별로 보여줘서 이해하기 좋았음. Uncle Bob의 `Clean Architecture` 에서 나온 원칙이다.

### 📌 SOLID

SOLID는 객체지향 설계의 5가지 원칙이다. 두문자어로, 각 앞글자를 따서 만들었다.

| 원칙  | 이름                  | 한 줄 요약                                         |
| ----- | --------------------- | -------------------------------------------------- |
| **S** | Single Responsibility | 하나의 모듈은 하나의 이유로만 변경되어야 한다      |
| **O** | Open/Closed           | 확장에는 열려 있고, 수정에는 닫혀 있어야 한다      |
| **L** | Liskov Substitution   | 자식 클래스는 부모 클래스를 대체할 수 있어야 한다  |
| **I** | Interface Segregation | 인터페이스는 사용하는 쪽에 맞게 잘게 분리해야 한다 |
| **D** | Dependency Inversion  | 구체적인 구현이 아닌 추상화에 의존해야 한다        |

> "하나의 클래스는 오직 하나의 이유로만 변경되어야 한다."

클래스에 대한 얘기지만, React 컴포넌트에도 그대로 통한다.

### 📌 다중 책임 컴포넌트의 문제

아래 `UserProfile` 컴포넌트를 보면 한 컴포넌트가 무려 5가지를 다 하고 있다:

1. 데이터 페칭
2. 에러 핸들링
3. 로딩 상태 관리
4. 폼 핸들링
5. 레이아웃 + 프레젠테이션

**❌ 이렇게 쓰면 안 된다**

```tsx
const UserProfile = () => {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    fetchUser();
  }, []);

  // 데이터 페칭도 여기서
  const fetchUser = async () => {
    try {
      const response = await fetch("/api/user");
      const data = await response.json();
      setUser(data);
    } catch (e) {
      setError(e as Error);
    } finally {
      setLoading(false);
    }
  };

  // 업데이트 로직도 여기서
  const handleUpdateProfile = async (data: Partial<User>) => {
    try {
      await fetch("/api/user", {
        method: "PUT",
        body: JSON.stringify(data),
      });
      fetchUser();
    } catch (e) {
      setError(e as Error);
    }
  };

  // 로딩/에러/빈 상태도 여기서
  if (loading) return <div>Loading...</div>;
  if (error) return <div>Error: {error.message}</div>;
  if (!user) return <div>No user found</div>;

  // UI도 여기서
  return (
    <div>
      <h1>{user.name}</h1>
      <form onSubmit={/* form logic */}>{/* 복잡한 폼 필드들 */}</form>
      <UserStats userId={user.id} />
      <UserPosts userId={user.id} />
    </div>
  );
};
```

이 컴포넌트를 말로 설명하면 이렇게 된다
"유저를 가져오고, 에러를 처리하고, 폼을 핸들링하고, UI를 보여준다." — **"하고"가 네 번** 나왔다. `SRP 위반 신호` 이다.

### 관심사를 분리하는 방법

#### 📌 1단계: 데이터 페칭 → 커스텀 훅으로 분리

```tsx
const useUser = (userId: string) => {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    fetchUser();
  }, [userId]);

  const fetchUser = async () => {
    try {
      const response = await fetch(`/api/user/${userId}`);
      const data = await response.json();
      setUser(data);
    } catch (e) {
      setError(e as Error);
    } finally {
      setLoading(false);
    }
  };

  return { user, loading, error, refetch: fetchUser };
};
```

이 훅의 책임은 딱 하나 — **유저 데이터를 가져오고 상태를 관리하는 것**.

#### 📌 2단계: UI → 프레젠테이션 컴포넌트로 분리

```tsx
const UserProfileView = ({
  user,
  onUpdate,
}: {
  user: User;
  onUpdate: (data: Partial<User>) => void;
}) => (
  <div>
    <h1>{user.name}</h1>
    <UserProfileForm user={user} onSubmit={onUpdate} />
    <UserStats userId={user.id} />
    <UserPosts userId={user.id} />
  </div>
);
```

이 컴포넌트는 **어떻게 보여줄지**만 안다. 데이터가 어디서 왔는지는 모른다.

#### 📌 3단계: 컨테이너 컴포넌트로 조립

```tsx
const UserProfileContainer = ({ userId }: { userId: string }) => {
  const { user, loading, error, refetch } = useUser(userId);

  const handleUpdate = async (data: Partial<User>) => {
    try {
      await fetch(`/api/user/${userId}`, {
        method: "PUT",
        body: JSON.stringify(data),
      });
      refetch();
    } catch (e) {
      // 에러 처리
    }
  };

  if (loading) return <LoadingSpinner />;
  if (error) return <ErrorMessage error={error} />;
  if (!user) return <NotFound message="User not found" />;

  return <UserProfileView user={user} onUpdate={handleUpdate} />;
};
```

컨테이너의 책임 — 데이터와 UI를 **연결하는 것**. 직접 렌더링 로직을 갖지 않는다.

---

### SRP를 지켰을 때 얻는 것

✅ 각 단위가 하나만 바뀌는 이유를 가짐
✅ 커스텀 훅은 다른 컴포넌트에서도 재사용 가능
✅ `UserProfileView`는 Storybook이나 테스트에서 독립적으로 렌더링 가능
✅ API 스펙 바뀌면 훅만, UI 바뀌면 View만 수정하면 됨

### 📌 정리

`컴포넌트를 설명할 때 "그리고"가 나오면 SRP 위반을 의심하라`

#### 핵심

> 데이터, 비즈니스 로직, 프레젠테이션 — 이 세 계층을 명확히 분리하는 것이 React SRP의 본질이다.
> "같은 이유로 변경되는 것"은 함께, "다른 이유로 변경되는 것"은 분리.

#### ⭐️ 앞으로 기억할 것

1. **커스텀 훅 = 데이터/로직 레이어**
   데이터 페칭, 상태, 사이드이펙트는 훅으로 빼는 게 기본값이 되어야 한다.

```tsx
// 이렇게 바로 뽑아 쓸 수 있어야 함
const { user, loading, error, refetch } = useUser(userId);
```

2. **Presentation Component는 순수해야 한다**
   props만 받아서 렌더링. `fetch`, `useState` 없음. 테스트와 스토리북 친화적.

```tsx
// 바깥 세계를 전혀 모르는 컴포넌트
const UserProfileView = ({ user, onUpdate }: Props) => <div>...</div>;
```

3. **"그리고" 테스트**
   컴포넌트 역할을 한 문장으로 설명해봤을 때 "그리고"가 나오면 분리 신호다.

```
❌ "유저를 가져오고, 로딩을 보여주고, 폼을 처리하고, UI를 렌더링한다"
✅ "유저 프로필 UI를 보여준다"
```

4. **Container-View 패턴 습관화**
   모든 컴포넌트가 이 패턴일 필요는 없지만, 복잡해지기 시작하면 먼저 이 구조를 떠올리기.

> **📌 결론**
> ▎ SRP는 "작게 쪼개라"가 아니라 "변경 이유를 하나로 줄여라"는 원칙이다
> ▎ React에서는 훅 / 컨테이너 / 프레젠테이션 세 역할로 자연스럽게 대응된다
> ▎ ⭐️ 컴포넌트가 뚱뚱해지기 시작하면, 훅 분리부터 시도해보자
