# React는 어떻게 UI를 순서 없이(out-of-order) 스트리밍하면서도 순서를 지키나

> 원문: [How React streams UI out of order and still manages to keep order](https://inside-react.vercel.app/blog/how-react-streams-ui-out-of-order) — Sankalpa Acharya (Inside React)

Suspense를 쓰면 느린 컴포넌트가 준비되는 대로 화면에 꽂히는데, "HTML 파싱은 순차적인데 어떻게 순서를 무시하고 꽂아넣지?"가 늘 궁금했다. 이 글은 서버가 보내는 **실제 HTML 스트림**과 `$RC`/`$RB`/`$RV` 인라인 스크립트를 까보면서 그 트릭을 보여준다. 한 줄 요약: **있는 건 즉시 보내고, 없는 건 표시된 placeholder를 남겼다가, 데이터가 준비되면 JS로 바꿔치기.**

### 전제: 스트리밍 자체는 새 게 아니다

서버 컴포넌트 이전에도 React는 스트리밍을 지원했다(React 18 `renderToPipeableStream()`/`renderToReadableStream()`). 브라우저도 원래 HTML을 청크 받는 대로 렌더한다. 보통 스트리밍은 `chunk(1)` → `chunk(2)` … `chunk(N)` 순서. **근데 RSC + Suspense의 흥미로운 점은 이 순서를 안 따른다는 것** — `component(2)`, `component(N)` … `component(1)` 처럼 아무 순서로 꽂힌다.

### Traditional SSR의 한계

```jsx
async function ProductPage() {
  const product = await getProduct();                 // 50ms
  const recommendations = await getRecommendations(); // 800ms
  const reviews = await getReviews();                 // 300ms
  return (<>
    <Navbar /> <ProductDetails product={product} />
    <Reviews reviews={reviews} /> <Recommendations recommendations={recommendations} /> <Footer />
  </>);
}
```

"이건 워터폴이잖아, 병렬로 fetch해" → 맞다, await을 떼서 병렬로 만들 수 있다. 그래도 **문제가 남는다**: 페이지가 세 개 fetch가 다 끝나야 HTML 첫 바이트를 보낸다. **데이터와 무관한 `Navbar`/`Footer`까지 `getRecommendations()`(800ms) 끝날 때까지 막힌다.** 데이터 필요 없는 컴포넌트는 바로 보여주면 좋을 텐데.

### In-order streaming의 한계

스트리밍을 도입하면 `Navbar`는 `ProductDetails` 안 기다리고 볼 수 있다. 근데 **`Footer`는 여전히 `Recommendations`가 로딩 중이라 막힌다.** 이게 **in-order streaming** — 각 컴포넌트가 HTML에 나타난 순서대로 순차 도착. (Next.js에선 일부러 만들어야 보이고 직접은 못 함.)

### Out-of-order streaming

> `Navbar`/`Footer`를 즉시 보내고, 느린 컴포넌트 자리엔 **placeholder(marker)** 를 떨궈두고, 데이터 준비되면 그 자리를 실제 콘텐츠로 **swap**하면? 대기도 블로킹도 없이 완전 독립.

이게 out-of-order streaming. 정해진 순서가 없고, 컴포넌트는 **자기 데이터가 준비되는 대로** 도착. React 18 `renderToPipeableStream`이 가능케 했고 React 19가 RSC를 안정화. 느린 컴포넌트를 `<Suspense fallback>`으로 감싸면 나머진 React가 처리한다.

```jsx
async function ProductDetails()   { await delay(50);  return <section>ProductDetails</section>; }
async function Reviews()          { await delay(800); return <section>Reviews</section>; }
async function Recommendations()  { await delay(300); return <section>Recommendations</section>; }

export default function Page() {
  return (
    <main>
      <Navbar />
      <Suspense fallback={<div>loading...</div>}><ProductDetails /></Suspense>
      <Suspense fallback={<div>loading...</div>}><Reviews /></Suspense>
      <Suspense fallback={<div>loading...</div>}><Recommendations /></Suspense>
      <Footer />
    </main>
  );
}
```

## Internals — 어떻게 동작하나

말로 하면 단순하다: **있는 건 즉시 전송, 없는 건 표시된 placeholder, 서버가 데이터 resolve하면 JS로 swap.** 아래는 전부 이 아이디어의 구현일 뿐.

### 초기 HTML 스트림

서버가 처음 보내는 HTML:

```html
<header>Navbar</header>

<!--$?-->
<template id="B:0"></template>
<div>loading..</div>
<!--/$-->

<!--$?-->
<template id="B:1"></template>
<div>loading...</div>
<!--/$-->

<!--$?-->
<template id="B:2"></template>
<div>loading...</div>
<!--/$-->

<footer>Footer</footer>
```

`Navbar`/`Footer`는 이미 있다. 느린 컴포넌트는 각각 fallback div를 가진 suspense boundary. `ProductDetails` 하나만 보면:

```html
<!--$?-->                         <!-- suspense boundary 시작, ? = pending -->
<template id="B:0"></template>    <!-- 교체될 placeholder -->
<div>loading..</div>              <!-- fallback UI -->
<!--/$-->                         <!-- suspense boundary 끝 -->
```

- `<!--$?-->` / `<!--/$-->` = suspense boundary 마커
- `<template>` = 교체될 placeholder
- `<div>loading..</div>` = fallback UI
- `id="B:0"` = 어느 placeholder를 swap할지 식별하는 키
- 주석의 `$?` = **boundary가 아직 pending**(fallback 표시 중, 실제 데이터 아직)

> 👉 Next.js 프로젝트 열고 DevTools Network 탭에서 hidden div랑 script가 실시간으로 스트리밍되는 걸 직접 보면 훨씬 빨리 이해된다. ([데모](https://oos-lime.vercel.app/check))

### 데이터 resolve → 컴포넌트가 클라이언트로

서버에서 데이터가 resolve되면 React가 컴포넌트를 스트리밍하는데:

```html
<div hidden id="S:0">
  <section>ProductDetails</section>
</div>
```

**`hidden` div**라는 게 포인트. React는 실제 위치에 바로 안 넣고 `id="S:0"`로 화면 밖에 주차해둔다. 그 직후 작은 script를 흘려보낸다:

```html
<script>$RC("B:0", "S:0")</script>
```

여기서 swap이 일어난다. `$RC`는 React가 스트림 앞부분에 이미 보내둔 함수라 클라이언트가 준비된 상태. 핵심 함수 3개: **`$RB` 큐, `$RC` 함수, `$RV` 함수.**

### `$RC` — 쌍을 큐에 넣기
```js
$RC = function(a, b) {
  if (b = document.getElementById(b))
    (a = document.getElementById(a))
      ? (/* replace logic */)
      : b.parentNode.removeChild(b)
}
```
- `a` = template id (`B:0`), `b` = resolved 컴포넌트 id (`S:0`)
- 먼저 resolved div(`b`)를 찾음 → 없으면 제거하고 끝
- 있으면 template(`a`)을 찾음 → 찾으면 **이전 형제 주석 노드를 `$?` → `$~`로 바꿔** boundary를 "queued"로 표시하고, 둘을 `$RB` 큐에 push:
  ```js
  a.previousSibling.data = "$~";
  $RB.push(a, b);
  ```
- 쌍이 완성되면 `requestAnimationFrame`으로 `$RV`를 호출해 실제 DOM swap.

### `$RB` — 큐(배열)
`[template, resolved]` 쌍을 담는 단순 배열. swap이 `$RC` 호출마다 즉시 일어나는 게 아니라, **최소 한 쌍이 쌓이면 다음 프레임에 `$RV` 실행을 예약.**

### `$RV` — 실제 swap
```js
$RV = function(a) {
  for (var b = 0; b < a.length; b += 2) {
    var c = a[b],     // template (B:0)
        e = a[b+1];   // resolved 컴포넌트 (S:0)
    ...
  }
}
```
쌍으로 push했으니 2개씩 순회. 순서대로:
1. resolved 컴포넌트를 hidden div에서 떼어냄(더 이상 hidden 아님)
2. suspense boundary 안 형제 노드들을 하나씩 제거 → **fallback UI(로딩 스피너) 싹 제거**
   ```js
   do { d = c.nextSibling; f.removeChild(c); c = d; } while (c);
   ```
3. resolved 컴포넌트의 자식들을 boundary 닫는 주석 앞에 삽입
   ```js
   for (; e.firstChild;) f.insertBefore(e.firstChild, c);
   ```
4. boundary 주석을 `$~` → `$`로 갱신(suspense resolved). boundary에 `_reactRetry`가 붙어있으면 발화(concurrent mode 재시도 처리).

**boundary 생애주기 = `$?` → `$~` → `$`:**
```
$?  = pending  (fallback 표시 중)
$~  = queued   (resolved 콘텐츠 준비됨, RAF 대기)
$   = complete (실제 콘텐츠가 DOM에)
```

### Suspense 깨뜨리기 (재밌는 실험)
React는 그냥 DOM에서 `<template id="B:0">`을 찾을 뿐이다. 그럼 내가 직접 하나 넣으면?
```jsx
<main>
  <div>hello <template id="B:0">hello testing</template></div>  {/* 가짜 심기 */}
  <Navbar />
  <Suspense fallback={<div>loading..</div>}><ProductDetails /></Suspense>
  ...
</main>
```
React는 이게 가짜인 줄 모른다. `$RC("B:0", "S:0")`이 `document.getElementById("B:0")`를 하면 **내 가짜를 먼저 찾아서**, 실제 ProductDetails placeholder 대신 내 랜덤 div를 swap해버린다. (id 중복의 함정)

## 종합

이게 단순 HTML 청킹과 React 스트리밍의 차이다. **일반 HTML 스트림은 파싱이 순차적이라 순서를 강제당한다.** React는 이를 우회한다 — **DOM을 staging area로 쓰고**, 컴포넌트를 hidden div로 보내고, **JS로 올바른 자리에 올바른 타이밍에 배치.**

---

### 📌 정리
`있는 건 즉시 보내고, 빈 자리엔 <template> 표시, JS(`$RC`/`$RV`)로 swap`

#### 핵심
> 일반 HTML 스트림은 파싱이 순차라 in-order만 가능 — 느린 컴포넌트 하나가 뒤의 모든 걸 막는다. React out-of-order 스트리밍은 **DOM을 staging area로 활용**한다: 초기엔 `<!--$?-->`+`<template id="B:n">`+fallback을 즉시 보내고, 서버에서 데이터가 resolve되면 컴포넌트를 **hidden div(`id="S:n"`)** 로 보낸 뒤 인라인 `<script>$RC("B:n","S:n")</script>`로 placeholder를 실제 콘텐츠와 바꿔치기한다. boundary는 `$?`(pending) → `$~`(queued) → `$`(complete) 생애주기를 거치고, swap은 `requestAnimationFrame`에 배치 처리된다.

#### ⭐️ 앞으로 기억할 것
- 1. **out-of-order의 본질 = DOM을 staging area로 쓰기**
  HTML 순차 파싱의 한계를, "화면 밖 hidden div + JS 재배치"로 우회. 청킹과 다른 점이 바로 이것.
- 2. **초기 HTML의 3종 마커 읽는 법**
  ```html
  <!--$?-->                    <!-- boundary 시작 + pending -->
  <template id="B:0"></template> <!-- 교체될 자리 -->
  <div>loading..</div>           <!-- fallback -->
  <!--/$-->                    <!-- boundary 끝 -->
  ```
- 3. **resolved 컴포넌트는 `<div hidden id="S:n">`로 먼저 주차된다**
  바로 제자리에 안 넣음. 그 다음 `$RC("B:n","S:n")` 스크립트가 swap을 트리거.
- 4. **swap 3단계(`$RV`)**: hidden 해제 → fallback 형제들 제거 → 자식들을 boundary 닫는 주석 앞에 삽입.
- 5. **boundary 생애주기 `$?` → `$~` → `$`**
  pending → queued(RAF 대기) → complete. DevTools에서 주석 노드 보면 상태 추적 가능.
- 6. **swap은 RAF로 배치된다 (`$RB` 큐)**
  매 `$RC`마다 즉시가 아니라, 쌍이 쌓이면 다음 프레임에 한 번에. 레이아웃 thrash 방지.
- 7. **id 식별 기반이라 위조 가능 = 함정**
  `getElementById("B:0")`만 보니 같은 id를 직접 심으면 엉뚱한 곳이 swap됨. id 충돌 주의.

📌 결론
▎ 일반 스트림 = 순차 강제, React 스트림 = DOM staging + JS 재배치로 순서 탈출.
▎ 흐름: `<template>` placeholder 즉시 → resolve되면 hidden div(`S:n`) → `$RC` 스크립트 → `$RV`가 RAF에 swap.
▎ boundary 상태는 `$?`(pending)→`$~`(queued)→`$`(complete) 주석으로 추적된다.
▎ ⭐️ 실무 적용 포인트
▎ → Suspense 경계는 "독립적으로 도착해도 되는 단위"로 설계 — 느린 데이터일수록 별도 boundary로 격리해 나머지 블로킹 방지.
▎ → 데이터 무관 컴포넌트(Navbar/Footer)는 Suspense 밖에 둬서 즉시 스트리밍.
▎ → 직접 확인: Next.js + DevTools Network에서 hidden div/script 스트리밍을 눈으로 보는 게 최고의 학습.
