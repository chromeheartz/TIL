# 자바스크립트 모듈에서 default export는 끔찍하다

> **원문**: https://www.lloydatkinson.net/posts/2022/default-exports-in-javascript-modules-are-terrible/
> **번역 출처**: https://velog.io/@eunbinn/default-exports-in-javascript-modules-are-terrible

`export default` 그냥 습관적으로 써왔는데, 이 글이 "왜 default export가 개발 경험을 망치는가"를 조목조목 까서 정리해둔다. 결론은 단순하다 — **named export만 쓰자.** 처음엔 "그게 그렇게까지 문제인가?" 싶었는데 IDE 자동완성, 리팩토링, 네이밍 일관성 얘기를 확인해보자.

### 두 가지 export 방식

자바스크립트 모듈에는 named export와 default export가 있다.

```javascript
export const add = (a, b) => a + b; // named

export default subtract = (a, b) => a - b; // default
```

이 글은 default export 쪽이 왜 나쁜 관행인지를 다섯 가지로 짚는다.

### 문제 1 — 무엇이 export됐는지 찾기 어렵다

처음 보는 모듈은 default export가 뭔지, 아니 default export가 있긴 한지조차 알기 힘들다. 결국 문서(있으면)나 소스 코드를 직접 뒤져야 한다. named export만 있으면 검색으로 바로 찾을 수 있다.

> 저자는 "JS 문서가 부실한 게 업계 표준"이라며, 진입 장벽이 낮아 경험 적은 개발자도 NPM에 패키지를 올릴 수 있는 환경 탓도 있다고 꼬집는다.

### 문제 2 — IDE 인텔리센스가 제대로 못 잡는다

IDE는 모듈이 export한 멤버들을 보여주지만, default export는 이름이 없어서 의미 있게 표시하지 못한다. 이건 CommonJS의 `require` / `module.exports`가 IDE 지원을 거의 못 받는 것과 같은 한계다.

### 문제 3 — import할 때 아무 이름이나 붙일 수 있다

default export는 import 시 이름을 마음대로 지을 수 있다. 이게 심각한 버그로 이어진다.

```javascript
import multiply from "./math.js";
const result = multiply(2, 2); // 4가 아니라 0이 나온다
```

`subtract`를 `multiply`라는 이름으로 가져왔다. 이름이 강제되지 않으니 이런 사고가 난다.

### 문제 4 — 네이밍 컨벤션이 제각각이 된다

여러 개발자가 같은 default export를 서로 다른 이름으로 가져온다. 저자는 rehype 모듈에서 이걸 겪었는데, 공식 문서 예제조차 네이밍이 들쭉날쭉했다.

```javascript
import rehypeExternalLinks from "rehype-external-links";
import addClasses from "rehype-add-classes"; // 같은 패턴인데 이름 규칙이 다름
```

일관된 컨벤션이 없으니 유지보수가 어려워진다.

### 문제 5 — 리팩토링이 안 된다

named export는 이름을 바꾸면 IDE가 모든 사용처를 자동으로 갱신해준다. default export는 이 자동 리네임 혜택을 못 받는다.

---

### 해법 1 — named export + import 별칭(alias)

"이름이 겹치면 어쩌냐"가 default를 쓰는 흔한 이유인데, named export는 `as`로 깔끔하게 푼다.

```javascript
import { Article } from "./types";
import { Article as ArticleComponent } from "my-design-system";
```

별칭을 붙이는 게, default import마다 매번 새 이름을 발명하는 것보다 인지 부담이 적다.

### 해법 2 — 프레임워크가 default를 강제하면 index 패턴

일부 프레임워크는 default export를 요구한다(예: 페이지 컴포넌트). 그럴 땐 디렉터리 루트에 `index.js`/`index.ts`를 두고 거기서만 default export를 하는 패턴으로 격리한다.

---

### 📌 정리

`default export 버리고 named export로 통일하라`

#### 핵심

> default export는 ① 뭐가 export됐는지 찾기 어렵고 ② IDE 인텔리센스가 못 잡고 ③ import 시 아무 이름이나 붙일 수 있어 버그를 부르고 ④ 네이밍이 제각각이 되며 ⑤ 자동 리팩토링(리네임)이 안 된다. named export는 이 모든 걸 해결하고, 이름 충돌은 `import { X as Y }` 별칭으로 깔끔히 푼다. 프레임워크가 default를 강제하면 `index` 파일에서만 격리해서 쓴다.

#### ⭐️ 앞으로 기억할 것

- 1. 기본은 named export
     검색·자동완성·리팩토링이 다 named에서만 제대로 작동한다.
  ```javascript
  export const subtract = (a, b) => a - b; // ✅
  // export default (a, b) => a - b;       // ❌
  ```
- 2. 이름 충돌은 default가 아니라 alias로
  ```javascript
  import { Article as ArticleComponent } from "my-design-system";
  ```
- 3. default는 "이름 강제"가 안 돼서 버그를 부른다
     `import multiply from "./math"`처럼 실제와 다른 이름으로 가져와도 막아주는 게 없다.
- 4. IDE 리네임이 default에는 안 먹는다
     named면 한 번에 전 사용처 갱신, default면 수작업.
- 5. 프레임워크가 default를 요구하면 index 파일로 격리
     `index.ts`에서만 `export { default } ...` / default 처리, 나머지는 전부 named로.

📌 결론
▎ default export의 "자유로운 이름"은 장점이 아니라 버그·불일치의 원천이다.
▎ named export로 통일하면 발견성·자동완성·리팩토링·일관성을 전부 얻는다.
▎ ⭐️ 실무 적용 포인트

- 새 모듈은 named export만. ESLint `import/no-default-export` 규칙으로 강제 가능.
- 라이브러리/디자인 시스템일수록 named가 필수 — 소비자 IDE 경험이 좋아진다.
- Next.js 페이지처럼 default가 강제되는 곳만 예외로 두고 index 패턴으로 가둔다.
