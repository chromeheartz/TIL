# CommonJS가 자바스크립트를 해치고 있다

> 원문: https://deno.com/blog/commonjs-is-hurting-javascript
> 번역 출처: https://velog.io/@eunbinn/commonjs-is-hurting-javascript

![](https://velog.velcdn.com/images/chromeheartz/post/2b22be33-4634-4523-81c5-d86511dd55b0/image.png)

자바스크립트 모듈 시스템 얘기는 꺼낼 때마다 좀 복잡한데, 이 글은 CommonJS가 왜 지금 시점에서 짐이 됐는지를 역사적 맥락부터 차근차근 설명해준다. 읽으면서 "아, 이래서 ESM 전환이 그렇게 오래 걸렸구나" 싶었다.

### CommonJS는 어떻게 생겨났나

2009년, Mozilla 개발자 Kevin Dangoor가 "서버 사이드 자바스크립트에 필요한 것"이라는 글을 올린다. 브라우저 밖에서 JS를 쓰려면 모듈 시스템이 필요하다는 얘기였는데, 반응이 폭발적이었다. 글 올린 지 일주일 만에 ServerJS 구글 그룹에 224명이 가입했고, npm 창립자 Isaac Schlueter, Node.js 만든 Ryan Dahl도 합류했다.

그렇게 CommonJS가 탄생했다. 문법은 지금도 익숙한 그것:

```js
// 가져오기
const fs = require("fs");

// 내보내기
module.exports = { myFunction };
```

당시엔 서버 사이드 JS 런타임들이 우후죽순 생겨났고 (Flusspferd, Narwhal, RingoJS 등), 이들이 모두 CommonJS를 채택하면서 사실상 표준이 됐다.

### 근데 뭐가 문제야

CommonJS는 **서버 사이드**를 위해 설계됐다. 파일 시스템에서 모듈을 읽는다는 전제로 만들어졌기 때문에 구조적으로 세 가지 한계를 갖는다.

#### 1. 동기식 로딩

`require()`는 동기다. 모듈을 하나씩 순서대로 읽고 실행한다. 로컬 파일 시스템에서는 괜찮지만, 네트워크를 통해 모듈을 가져와야 하는 브라우저에서는 치명적이다. 첫 번째 `require`가 끝날 때까지 다음 게 시작을 못 하니까.

#### 2. 트리 셰이킹이 안 된다

```js
// CJS — 빌드 도구가 이걸 정적으로 분석하기 어렵다
const { readFile } = require("fs");

// ESM — 정적 분석 가능
import { readFile } from "fs";
```

CJS는 `require()`가 런타임에 동적으로 실행되기 때문에 번들러가 "이 모듈에서 실제로 어떤 게 쓰이는지"를 빌드 타임에 파악하기 힘들다. 결과적으로 번들 크기가 커진다.

#### 3. 브라우저용이 아니다

브라우저에서 CJS를 쓰려면 Webpack, Rollup 같은 번들러와 트랜스파일러가 필수다. 빌드 단계가 없으면 그냥 동작하지 않는다.

### ESM이 등장하다

2013년쯤 CommonJS 그룹이 사실상 해체되기 시작했고, TC39 위원회가 ES6 스펙에 모듈 시스템을 직접 넣었다. 이게 ESM(ECMAScript Modules)이다.

```js
// ESM
import { readFile } from "fs";
export const myFunction = () => {};
```

ESM이 CJS보다 나은 점:

| 특징          | CJS              | ESM            |
| ------------- | ---------------- | -------------- |
| 로딩 방식     | 동기             | 비동기         |
| 브라우저 지원 | ❌ (번들러 필요) | ✅ 네이티브    |
| 트리 셰이킹   | 어려움           | 정적 분석 가능 |
| 정적 분석     | ❌               | ✅             |

### Node.js가 두 마리 토끼를 잡으려 하면서 생긴 일

문제는 Node.js였다. CJS가 이미 Node에 너무 깊숙이 박혀 있어서 뺄 수가 없었다. 그래서 ESM을 추가로 지원하기로 했는데, 이 결정이 패키지 작성자들한테는 악몽이 됐다.

CJS와 ESM은 서로 완전히 호환되지 않는다. 그래서 npm 패키지를 두 형식 모두 지원하려면:

```json
// package.json — 이중 지원 설정
{
  "main": "./dist/index.cjs", // CJS용
  "module": "./dist/index.mjs", // ESM용
  "exports": {
    "require": "./dist/index.cjs",
    "import": "./dist/index.mjs"
  }
}
```

![](https://velog.velcdn.com/images/eunbinn/post/e15205a7-1d29-4e5e-bfd6-5de0802cc7da/image.jpeg)

Wes Bos가 트윗에서 "ESM과 CJS를 모두 지원하는 패키지 배포는 악몽"이라고 했는데, 실제로 이 dual package 구조를 올바르게 설정하는 게 만만치 않다.

#### dnt 같은 도구로 돌아가는 식으로

Deno팀에서 만든 [dnt](https://github.com/denoland/dnt) 같은 도구를 쓰면 TypeScript 소스 하나로 CJS/ESM 양쪽 빌드를 뽑을 수 있다. 하지만 이런 도구가 필요하다는 것 자체가 문제의 방증이기도 하다.

### 그래서 결론은

![](https://velog.velcdn.com/images/eunbinn/post/25775107-c998-40fa-ade9-23cc0ae4cc3d/image.jpeg)

2009년의 CommonJS는 당시 상황에서 최선이었다. 하지만 브라우저, 엣지, 서버리스까지 JS가 뻗어나간 지금, 동기식·파일 시스템 기반의 CJS는 맞지 않는 옷이 됐다. ESM은 이미 브라우저, Deno, 최신 Node.js 모두에서 네이티브로 지원된다.

글의 핵심 메시지: **새 프로젝트라면 ESM으로 시작하고, 기존 CJS 패키지도 ESM으로 마이그레이션하는 게 맞다.**

### 📌 정리

`CJS는 2009년의 해답이었고, ESM은 지금의 표준이다`

#### 핵심

> CommonJS는 서버 사이드 파일 시스템을 전제로 설계된 동기식 모듈 시스템이다. 브라우저·엣지·서버리스 환경이 표준이 된 지금, CJS의 동기 로딩과 정적 분석 불가 특성은 번들 크기와 성능 모두에서 손해다. ESM은 비동기·정적 분석·트리 셰이킹을 모두 지원하며, 이미 모든 주요 런타임에서 네이티브로 동작한다.

#### ⭐️ 앞으로 기억할 것

- 1. **`require`는 런타임, `import`는 컴파일 타임**
     CJS `require()`는 코드가 실행될 때 평가되기 때문에 번들러가 정적으로 분석할 수 없다. ESM `import`는 파일 최상단에 선언되고 빌드 타임에 분석 가능하다. 트리 셰이킹이 ESM에서만 제대로 되는 이유.

- 2. **Dual package 지옥**
     Node.js가 CJS와 ESM을 동시에 지원하기로 하면서, 패키지 작성자는 양쪽을 모두 빌드해야 하는 상황이 됐다. `package.json`의 `exports` 필드로 분기하는 게 현재 관례지만, 설정 실수가 잦다.

  ```json
  {
    "exports": {
      "require": "./dist/index.cjs",
      "import": "./dist/index.mjs"
    }
  }
  ```

- 3. **새 프로젝트는 그냥 ESM으로**
     `package.json`에 `"type": "module"` 한 줄 추가하면 `.js` 파일이 전부 ESM으로 처리된다. CJS가 꼭 필요한 레거시 의존성이 없다면 고민할 이유가 없다.

📌 결론
▎ CommonJS는 탄생 배경 자체가 "브라우저 없는 서버"였기 때문에, 지금의 JS 환경과는 구조적으로 맞지 않는다.
▎ ESM 전환이 늦어진 건 기술 문제가 아니라 Node.js의 하위 호환 정책 때문이었다.
▎ ⭐️ 실무 적용 포인트: 라이브러리 만들 때 CJS 지원을 기본으로 생각하는 관성을 버려야 한다. 타겟 환경이 최신 Node.js + 브라우저라면 ESM 단일 출력으로 충분하다.
