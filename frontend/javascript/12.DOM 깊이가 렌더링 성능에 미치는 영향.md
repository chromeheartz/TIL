# DOM 깊이가 렌더링 성능에 미치는 영향

> **원문**: https://frontendatscale.com/blog/how-deep-is-your-dom/
> **번역 출처**: https://velog.io/@eunbinn/how-deep-is-your-dom

![](https://velog.velcdn.com/images/chromeheartz/post/1e1cd127-3ab0-431f-a2ec-86edbe4d9ab4/image.png)

Lighthouse에서 "DOM 크기가 너무 크다"는 경고를 본 적이 있는데, 단순히 요소 개수만의 문제가 아니라 **DOM 깊이**도 렌더링 성능에 영향을 준다는 걸 실험으로 보여주는 글을 읽었다. 수치로 직접 확인할 수 있어서 인상적이었음.

---

### 왜 DOM 깊이가 문제인가

DOM 크기가 크면 메모리 사용량이 늘고 스타일 계산 비용이 올라간다. 특히 저사양 기기에서 두드러진다.

같은 개수의 요소라도 구조에 따라 브라우저가 탐색하는 방식이 달라진다.

**얕은 트리 (depth 2)**

```javascript
body.children[4]; // 바로 접근
```

**깊은 트리 (depth 6)**

```javascript
body.children[0].children[0].children[0].children[0].children[0]; // 계속 내려가야 함
```

---

### 실험: 얕은 구조 vs 깊은 구조

div 100개를 두 가지 방식으로 배치하고 렌더링 시간을 비교했다.

**얕은 구조** — body 바로 아래 100개 나열

```html
<body>
  <div></div>
  <div></div>
  <!-- ... -->
  <div>마지막 100번째 div.</div>
</body>
```

**깊은 구조** — 100단계로 중첩

```html
<body>
  <div>
    <div>
      <div>
        <!-- ... -->
        <div>마지막 100번째 div.</div>
      </div>
    </div>
  </div>
</body>
```

#### 결과

| 요소 수 | 얕은 트리 | 깊은 트리 |
| ------- | --------- | --------- |
| 100개   | 51ms      | 53ms      |
| 500개   | 56ms      | 102ms     |

100개에서는 차이가 거의 없었는데, 500개부터 2배 가까이 벌어졌다. 5,000개에서는 그 차이가 더 극명해진다.

![요소 수에 따른 렌더링 시간 비교](https://velog.velcdn.com/images/eunbinn/post/59ab3c15-7baa-46c4-a48d-70c5dab36fea/image.png)

### CSS가 더하면 훨씬 심해진다

```css
div {
  padding-top: 10px;
}
```

이런 광범위한 셀렉터 하나가 페이지의 거의 모든 요소에 영향을 미치면, 브라우저는 각 div의 위치를 전부 다시 계산해야 한다. 깊은 트리에서는 이 비용이 기하급수적으로 커진다.

![CSS 스타일 추가 후 렌더링 시간 비교](https://frontendatscale.com/blog-assets/how-deep-is-your-dom-style.webp)

---

### DOM 깊이 확인하는 법

```javascript
// 현재 페이지 전체 요소 수 확인
document.querySelectorAll("*").length;
```

Lighthouse나 PageSpeed Insights에서도 DOM 크기 경고를 확인할 수 있다.

---

### 실무 시사점

- 2017년 연구에 따르면 페이지 요소 수가 400개 → 6,000개로 늘어나면 전환율이 95% 감소한다.
- 현대 브라우저는 수천 개 중첩도 밀리초 단위로 처리할 만큼 뛰어나지만, 규모가 커질수록 차이가 나온다.
- CSS 셀렉터의 범위를 줄이는 것도 DOM 최적화만큼 중요하다.

![요소 수와 전환율의 관계](https://frontendatscale.com/blog-assets/how-deep-is-your-dom-stat.webp)

### 📌 정리

`DOM은 얕을수록 좋다 — 같은 요소 수라도 깊이에 따라 성능이 달라진다`

#### 핵심

> DOM 깊이는 요소 개수만큼이나 성능에 영향을 준다.
> CSS 스타일 재계산 비용은 DOM이 깊을수록 더 크게 늘어난다.

#### ⭐️ 앞으로 기억할 것

- 1. **불필요한 래퍼 div 줄이기**
     "레이아웃을 위해 감쌌는데 또 감쌌는데 또 감쌌는" 구조가 쌓이면 depth가 올라간다. 컴포넌트 설계할 때 wrapping 최소화.

- 2. **광범위한 CSS 셀렉터 주의**
     `div { ... }`, `* { ... }` 같은 셀렉터는 모든 요소에 스타일 계산을 트리거한다. 범위를 좁힐수록 재계산 비용이 줄어든다.

- 3. **주기적으로 DOM 크기 확인**

```javascript
document.querySelectorAll("*").length;
```

Lighthouse 경고 나오기 전에 개발 중에도 가끔 체크해보기.

- 4. **100개 이하에서는 깊이 차이가 거의 없다**
     과도하게 플랫하게 만들 필요는 없다. 의미 있는 구조를 유지하되, 중첩이 과하게 쌓이는 것만 경계하면 된다.

> **📌 결론**
> ▎ DOM 깊이는 렌더링 성능에 실질적인 영향을 준다 — 특히 규모가 커질수록
> ▎ CSS 스타일 계산 비용은 DOM 깊이와 셀렉터 범위가 결합될 때 폭발적으로 늘어난다
> ▎ ⭐️ 컴포넌트 설계할 때 불필요한 래퍼 div를 한 번쯤 의심해보자
