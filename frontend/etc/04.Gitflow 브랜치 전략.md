# Gitflow 브랜치 전략

> 원문: [[Gitflow] 깃플로우 브랜치 전략](https://m.blog.naver.com/adamdoha/222712473510) — adamdoha

협업하다 보면 "이 기능은 어느 브랜치에서 짜고, 배포는 어디서 나가고, 급한 버그는 어떻게 끼워넣지?"가 늘 헷갈린다. Gitflow는 Vincent Driessen이 2010년 제시한 방법론인데, 중요한 건 **특별한 기술이 아니라 "협업하는 사람끼리 브랜치를 어떻게 관리할지에 대한 약속"** 이라는 점이다. 약속이니까 팀마다 변형해서 써도 된다.

### 브랜치 6종

| 브랜치 | 역할 |
|---|---|
| **master** | 제품 배포용 (실제 출시된 코드) |
| **develop** | 개발 기본 브랜치. **feature의 기준점이자 중심** |
| **feature** | 단위 기능 개발용 |
| **release** | 배포 전 QA·품질 체크용 |
| **hotfix** | 배포 후 긴급 버그 수정용 |
| **support** | 버전 호환성 관리용 |

> **핵심: `develop`이 깃플로우의 중심.** 개발자들이 각자 역할을 나눠 짠 코드를 처음 모으는 집합소다. master는 "배포된 것"만, develop은 "다음 배포를 향해 쌓이는 것".
> 그리고 **작업 단위(feature)는 작게작게 가져가는 게 좋다** — 작을수록 리뷰·머지·롤백이 쉬움.

### 설치 & 초기화

```bash
$ brew install git
$ brew install git-flow-avh   # git-flow 확장 (avh 에디션)
```
```bash
git flow init
```
초기화 시 묻는 브랜치 네이밍(기본값):
- Production branch → `master`
- Next release development → `develop`
- Feature branches → `feature/`
- Release branches → `release/`
- Hotfix branches → `hotfix/`
- Support branches → `support/`

### feature — 기능 개발

```bash
git flow feature start {name}
# → feature/{name} 브랜치가 develop에서 생성됨
```
기능을 다 짜면 → **코드 리뷰 승인 + PR을 develop에 머지** → origin·local의 feature 브랜치 삭제.
(실무에선 `git flow feature finish` 대신 PR 기반으로 develop에 머지하는 흐름이 흔함.)

### release — 배포 준비/QA

```bash
git flow release start {name}
# → release/{name} 브랜치가 develop에서 생성됨
```
QA 통과 후, 머지 전에 관련 브랜치를 최신화하고 finish:
```bash
git switch master
git pull origin master
git switch develop
git pull origin develop
git switch release/{name}
git pull origin release/{name}
git flow release finish {name}
```
`release finish`가 자동으로 해주는 것:
- **release → master 머지** (배포)
- **버전 태그 생성**
- **release → develop 머지** (배포분을 develop에도 반영)
- local release 브랜치 삭제

마지막에 원격으로 push:
```bash
git push origin master develop 태그명
```

### hotfix — 배포 후 긴급 수정

```bash
git flow hotfix start {tag명}
# → hotfix 브랜치가 master에서 생성됨 (배포된 코드 기준!)
```
PR 승인 후:
```bash
git flow hotfix finish {name}
```
`hotfix finish`도 release처럼 **master·develop 양쪽에 머지 + 태그 생성 + 로컬 브랜치 삭제**를 자동 처리. 그 후 push:
```bash
git push origin master develop 20220426-h1
```

> **release vs hotfix의 출발점 차이:** release는 `develop`에서, hotfix는 `master`에서 분기한다. hotfix는 이미 배포된 코드(master)의 급한 버그를 고치는 거라 master 기준이어야 함. 둘 다 끝나면 master·develop **양쪽에** 반영되는 게 포인트.

### 📌 정리
`develop 중심의 브랜치 6종 약속 — feature/release/hotfix를 git-flow 명령으로 자동 머지·태그`

#### 핵심
> Gitflow는 기술이 아니라 **팀의 브랜치 관리 약속**이다. `master`(배포)·`develop`(개발 중심) 두 상시 브랜치를 두고, 그때그때 `feature`(develop에서 분기)·`release`(develop→QA→master+develop)·`hotfix`(master에서 분기→master+develop)를 만들어 쓴다. `git flow` 명령이 머지·태그·브랜치 삭제를 자동화한다. release/hotfix의 `finish`는 **master와 develop 양쪽에 머지**하고 버전 태그를 단다는 게 핵심.

#### ⭐️ 앞으로 기억할 것
- 1. **두 상시 브랜치: master(배포) / develop(중심)**
  master엔 "출시된 것"만, develop엔 "다음 배포를 향해 쌓이는 것". feature는 항상 develop에서 분기.
- 2. **feature는 작게**
  작업 단위가 작을수록 리뷰·머지·롤백이 쉽다. `git flow feature start {name}` → 보통 PR로 develop 머지 후 브랜치 삭제.
- 3. **release는 develop에서, hotfix는 master에서 분기**
  ```text
  release/* ← develop  (배포 준비 + QA)
  hotfix/*  ← master   (배포된 코드의 긴급 버그)
  ```
- 4. **release/hotfix finish = master + develop 양쪽 머지 + 태그**
  배포분이 develop에도 반영돼 두 브랜치가 어긋나지 않게 한다. 자동 처리되는 게 git-flow의 편의.
- 5. **finish 전엔 관련 브랜치 pull로 최신화**
  master·develop·release를 각각 `git pull` 후 finish → 충돌·꼬임 방지.
- 6. **push는 두 브랜치 + 태그 같이**
  ```bash
  git push origin master develop {태그명}
  ```
- 7. **결국 "약속"이다 — 팀 방식 우선**
  Gitflow가 무거우면 GitHub Flow 등 더 가벼운 전략도 OK. 팀 내 합의가 정답.

📌 결론
▎ Gitflow = master/develop 상시 + feature/release/hotfix/support 임시 브랜치의 협업 약속.
▎ feature→develop, release(develop기준)·hotfix(master기준)→master+develop 양쪽 머지 & 태그.
▎ `git-flow-avh`가 머지·태그·삭제를 자동화해 실수를 줄여준다.
▎ ⭐️ 실무 적용 포인트
▎ → feature는 작게 쪼개 PR 단위로. 리뷰 통과 후 develop 머지 + 브랜치 정리.
▎ → 긴급 버그는 develop이 아니라 master에서 hotfix 분기(배포 코드 기준).
▎ → finish 전 항상 master/develop pull로 최신화. push는 master·develop·태그를 함께.
▎ → 팀이 작거나 배포가 잦으면 Gitflow 대신 더 단순한 전략 고려 — 핵심은 "팀의 약속".
