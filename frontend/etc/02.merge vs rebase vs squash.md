# merge vs rebase vs squash

> **원문**: https://gist.github.com/mitchellh/319019b1b8aac9110fcfb1862e0c97fb
> **번역 출처**: https://velog.io/@eunbinn/merge-vs-rebase-vs-squash

![](https://velog.velcdn.com/images/chromeheartz/post/b4a3480d-8aae-46c8-b8e8-d20cc54a1a4e/image.png)

Vagrant, HashiCorp 창업자인 Mitchell Hashimoto가 git 전략에 대해 쓴 글이다. "항상 squash만 써라" 같은 고집없이 상황에 따라 세 가지를 다 쓴다는 내용

---

### 전제 조건

어떤 전략을 쓰든 **모든 커밋은 빌드 가능해야 한다.** 이 원칙 위에서 세 전략을 선택적으로 사용한다.

---

### Merge — 기본값 (90% 이상)

저자는 대부분의 경우 merge commit을 선호한다. 이유는 **이력을 가장 충실하게 보여주기** 때문이다.

merge commit이 있으면:

- 언제 어떤 브랜치가 합쳐졌는지 명확히 보임
- WIP 단계의 커밋들도 전부 남아 있어서 개발 흐름 추적 가능
- 전체 PR을 통째로 되돌릴 때 간단함 (`git revert -mN`)

#### git bisect와의 궁합

merge 전략이 빛나는 건 버그 추적할 때다.

커밋이 ±50줄 단위로 남아 있으면 `git bisect`로 버그를 유발한 커밋을 정확히 찾을 수 있다. 반면 2,000줄짜리 squash 커밋 하나만 남아 있으면 "이 PR 어딘가에 버그가 있다"는 것만 알 수 있고, 어느 변경사항이 문제인지 알기 어렵다.

---

### Squash — WIP가 많은데 diff가 작을 때만

"WIP", "WIP2", "fix", "fix2" 같은 의미 없는 커밋이 수두룩한데 최종 변경사항 자체는 작은 경우에만 squash를 쓴다.

**주의**: GitHub이 자동으로 만들어주는 squash 메시지는 각 커밋 메시지를 그냥 이어붙인 수준이라 거의 쓸모없다. squash할 때는 커밋 메시지를 직접 새로 써야 의미 있다.

squash의 단점 — git bisect 정보가 사라진다. 2,000줄 변경이 하나의 커밋으로 뭉쳐지면 버그 추적 단위가 너무 커진다.

**squash를 쓸 상황 요약:**

- WIP 커밋이 많음 ✅
- 최종 diff가 작음 ✅
- 커밋 메시지를 직접 다시 쓸 것 ✅

---

### Rebase (interactive) — diff도 크고 WIP도 많을 때

diff가 크면서 WIP 커밋도 많을 때 interactive rebase로 직접 정리한다.

- 의미 없는 커밋끼리만 선택적으로 squash
- 순서 바꾸기, 커밋 쪼개기 등 세밀한 조정 가능

커밋이 50개 이상이면 GUI 클라이언트(Tower 등)를 쓰는 게 낫다. 터미널 interactive rebase는 커밋이 많아지면 관리하기 힘들다.

---

### 세 전략 비교 요약

| 전략       | 언제                  | git bisect        | 이력                  |
| ---------- | --------------------- | ----------------- | --------------------- |
| **Merge**  | 기본값                | ✅ 유리           | 실제 개발 흐름 그대로 |
| **Squash** | WIP 많고 diff 작을 때 | ❌ 정보 손실      | 깔끔하지만 압축됨     |
| **Rebase** | diff 크고 WIP 많을 때 | ✅ 유리 (정리 후) | 선택적으로 정리 가능  |

---

### 📌 정리

`상황에 따라 세 가지를 다 써라 — 하나만 고집하면 어딘가에서 손해본다`

#### 핵심

> merge는 이력과 bisect를 위한 선택, squash는 노이즈 제거용, rebase는 큰 PR 정리용이다.
> 전제는 하나 — 모든 커밋은 빌드 가능해야 한다.

#### ⭐️ 앞으로 기억할 것

- 1. **merge가 기본값이 되어야 하는 이유**
     버그 나면 `git bisect`로 추적하는데, squash 커밋 하나에 2,000줄이 뭉쳐 있으면 범위를 좁힐 수가 없다. 작은 단위 커밋이 남아있을수록 디버깅이 쉽다.

- 2. **squash할 때는 메시지를 직접 써야 한다**
     GitHub 자동 메시지는 WIP 커밋들을 그냥 이어붙인 것 — 아무 의미 없다. squash의 가치는 깔끔한 메시지에서 나온다.

- 3. **"항상 squash"는 bisect를 포기하는 것**
     팀에서 squash를 강제하는 곳들이 있는데, 버그 추적 비용을 높이는 트레이드오프가 있다는 걸 인식하고 써야 한다.

- 4. **커밋 50개 이상 rebase는 GUI 클라이언트로**
     터미널 `git rebase -i`는 커밋이 많아지면 관리하기 힘들다. Tower 같은 GUI 쓰는 게 실수 줄이는 방법.

> **📌 결론**
> ▎ merge = 이력 보존 + bisect 친화적, 90%+ 상황에서 좋은 선택
> ▎ squash = WIP 노이즈 정리용, 단 메시지 직접 작성 필수
> ▎ ⭐️ "항상 squash만 써라"는 말은 절반만 맞다 — bisect 비용을 생각하면 merge가 더 나을 때가 많다
