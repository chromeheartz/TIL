# SSR과 CSR의 차이

> **출처**: https://velog.io/@vagabondms/%EA%B8%B0%EC%88%A0-%EC%8A%A4%ED%84%B0%EB%94%94-SSR%EA%B3%BC-CSR%EC%9D%98-%EC%B0%A8%EC%9D%B4 (velog @vagabondms, 김민석)

렌더링 얘기를 하다 보면 결국 "그래서 CSR이랑 SSR이 정확히 뭐가 다른데?"로 돌아온다. [하이드레이션 글][hydration]을 정리하면서 기본기를 한 번 더 다지고 싶어서 이 기술 스터디 글로 개념을 정돈했다. 두 방식의 흐름을 따라가 보면 차이가 한눈에 들어온다.

### SSR (Server Side Rendering)

서버가 클라이언트로 보내기 **전에** 렌더링 준비를 끝낸다. 서버가 리소스를 확인하고 **완성된 HTML**을 컴파일해서 보낸다.

![SSR process flow diagram](https://velog.velcdn.com/images%2Fvagabondms%2Fpost%2F8c4c7988-c35d-4722-8e78-4c4f4bbb54a5%2Fimage.png)

- 클라이언트에 도착하면 HTML이 **즉시 렌더링**된다 → 화면이 바로 보임
- 단, JavaScript가 로드되기 전까지는 **상호작용 불가**(non-interactive)
- 사용자는 JS 다운로드 중에도 **콘텐츠는 볼 수 있지만** 클릭 등은 안 먹음
- JS가 성공적으로 컴파일되면 그제서야 인터랙션이 동작 → 완전히 interactive

> 즉 SSR은 "**보는 것(view)**"은 빠르지만 "**누르는 것(interactive)**"은 JS 실행 이후. 이 간극이 [하이드레이션 글][hydration]에서 말한 "보이지만 안 눌리는 시간".

### CSR (Client Side Rendering)

렌더링이 서버가 아니라 **클라이언트 쪽에서** 일어난다. 서버는 HTML과 JavaScript 링크를 보내고, 실제 렌더링은 클라이언트가 시작한다.

![CSR process flow diagram](https://velog.velcdn.com/images%2Fvagabondms%2Fpost%2F0289f96e-c34d-48a9-b942-eb9376ab43af%2Fimage.png)

- SSR과 달리 HTML·JS 다운로드 동안 사용자는 **아무것도 못 본다**(빈 화면)
- JS 실행 + API 호출 후 → **placeholder 콘텐츠**가 보임
- API 데이터가 돌아오면 → 페이지가 완성되고 interactive해짐

### 각각의 차이

#### 1) 웹페이지 로딩 시간

- **첫 페이지 로드**: SSR은 **필요한 HTML과 스크립트만** 로드 → 평균적으로 더 빠름. CSR은 스크립트를 **한 번에 다** 받느라 느림.
- **이후 페이지 전환**: CSR은 남은 코드가 이미 로드돼 있어 **더 빠름**. SSR은 매 페이지마다 첫 페이지 과정을 반복.

#### 2) SEO 대응

- CSR은 JavaScript로 콘텐츠를 동적 생성 → 메타데이터 변경에도 **JS 실행이 필요**.
- 과거 크롤러는 JS를 실행하지 않아 SEO 최적화가 필수였다 (다만 **구글은 이 흐름을 바꾸는 중**).
- SSR은 서버에서 **미리 컴파일**해서 보내므로 크롤러 대응이 쉽다.

#### 3) 서버 자원 사용

- SSR은 요청마다 렌더링을 반복해 **서버 자원을 더 많이 쓴다.**
- React의 `renderToString`은 **스택을 블로킹하며 동기적으로** 처리 → 실행 동안 서버가 멈춘다.
- CSR은 작업을 **클라이언트에 떠넘겨** 서버 부담을 최소화.

> 참고: 이 블로킹 문제는 React 18의 스트리밍 SSR(`renderToPipeableStream` 등)로 개선됐다.

### 사용 권장 예시

- **SEO가 중요**하거나 **첫 화면이 빨라야** 하는 페이지(랜딩/블로그/상품 상세) → **SSR**
- **상호작용이 많고** SEO가 덜 중요한 페이지(로그인 후 대시보드 등) → **CSR**

### 📌 정리
`렌더링을 어디서 하느냐 — 서버(SSR)냐 클라(CSR)냐의 트레이드오프`

#### 핵심
> SSR은 서버가 완성 HTML을 만들어 보내 **첫 화면·SEO에 유리**하지만, 인터랙션은 JS 실행 이후이고 서버 자원을 더 쓴다(`renderToString`이 동기 블로킹). CSR은 빈 HTML+JS를 받아 브라우저가 그려 **첫 화면은 느리지만(빈 화면)** 이후 전환이 빠르고 서버 부하가 적다. "무엇이 더 좋다"가 아니라 **SEO 중요도 / 초기속도 / 서버비용**을 저울질하는 선택이다.

#### ⭐️ 앞으로 기억할 것
- 1. **SSR = 첫 화면·SEO 유리 + 서버 자원 ↑**, 단 인터랙션은 하이드레이션 후
- 2. **CSR = 첫 화면 느림(빈 화면) + 이후 전환 빠름 + 서버 부하 ↓**
- 3. **SEO 핵심 차이**: 크롤러가 JS 실행을 잘 못 기다림 → 메타데이터가 HTML에 박혀야 안전(SSR). 단 구글은 점점 JS 실행 지원.
- 4. **`renderToString`은 동기 블로킹** → 서버가 그동안 멈춤. React 18 스트리밍 SSR로 개선.
- 5. **SSR의 "보이지만 안 눌리는 시간"** = 하이드레이션 구간 → [하이드레이션 글][hydration]과 직결.

📌 결론
▎ CSR/SSR은 "렌더링 위치"의 트레이드오프 — 초기속도·SEO·서버비용의 균형 문제.
▎ SSR이 첫 화면·SEO에서 이기지만 공짜가 아니다(하이드레이션 + 서버 자원 + 블로킹).
▎ ⭐️ 실무 적용 포인트
▎ - 검색 노출 중요 페이지(랜딩/블로그/상품) → SSR(or SSG).
▎ - 로그인 후 인터랙션 위주 화면 → CSR로 충분.
▎ - Next.js라면 → 페이지 성격별로 SSR/SSG/CSR을 섞는 게 정석.
