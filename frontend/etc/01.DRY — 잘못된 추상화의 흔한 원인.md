# DRY — 잘못된 추상화의 흔한 원인

> **원문**: https://swizec.com/blog/dry-the-common-source-of-bad-abstractions/
> **번역 출처**: https://velog.io/@eunbinn/dry-the-common-source-of-bad-abstractions

![](https://velog.velcdn.com/images/chromeheartz/post/4fed1251-b7e9-4be5-b79a-a7bc8358dd1e/image.png)

DRY(Don't Repeat Yourself)가 무조건 좋은 원칙인 줄 알았는데, 잘못 적용하면 오히려 유지보수를 더 어렵게 만드는 추상화를 낳는다는 걸 잘 정리한 글을 읽었다. 내가 쓴 코드 중에도 비슷한 게 있었던 것 같아서 제대로 정리해둠.

---

### DRY가 잘 작동하는 경우

반복되는 코드를 제거하는 건 맞다.

❌ 그냥 쓰면

```javascript
console.log(1);
console.log(2);
console.log(3);
console.log(4);
```

✅ DRY 적용

```javascript
for (let i = 1; i < 5; i++) {
  console.log(i);
}
```

이건 명확히 이득이다. 변경 포인트가 하나로 줄어들고, 의도도 더 잘 드러난다.

---

### DRY가 잘못되는 과정

내비게이션 메뉴 예시. 처음엔 이렇게 시작한다:

```tsx
const NavigationMenu = () => (
  <ul>
    <li>
      <a href="/about">
        <img src="question-icon.png" />
        About
      </a>
    </li>
    <li>
      <a href="/contact">
        <img src="person-icon.png" />
        Contact
      </a>
    </li>
    <li>
      <a href="/buy">
        <img src="cash-icon.png" />
        Buy
      </a>
    </li>
  </ul>
);
```

반복이 눈에 보이니까 DRY를 적용하고 싶어진다.

#### 📌 1단계: 배열로 데이터 분리

```tsx
const NavigationMenu = () => {
  const items = [
    { url: "/about", icon: "question-icon.png", label: "About" },
    { url: "/contact", icon: "person-icon.png", label: "Contact" },
    { url: "/buy", icon: "cash-icon.png", label: "Buy" },
  ];

  return (
    <ul>
      {items.map((item) => (
        <li>
          <a href={item.url}>
            <img src={item.icon} />
            {item.label}
          </a>
        </li>
      ))}
    </ul>
  );
};
```

#### 📌 2단계: 팩토리 함수까지 추가

```tsx
function makeNavItem(url, icon, label) {
  return { url, icon, label };
}

const items = [
  makeNavItem("/about", "question-icon.png", "About"),
  makeNavItem("/contact", "person-icon.png", "Contact"),
  makeNavItem("/buy", "cash-icon.png", "Buy"),
];
```

코드가 짧아졌다. 얼핏 보면 좋아 보인다.

---

### 이것이 나쁜 추상화인 이유

마케팅팀이 요청을 넣는다: **"Buy 버튼에만 빨간 테두리 추가해주세요."**

지금 구조에서는 이걸 처리하기가 어렵다. `items.map()`이 모든 항목을 동일하게 렌더링하니까, Buy만 다르게 만들려면 `if` 분기를 넣거나 데이터 스키마를 건드려야 한다.

```tsx
// 억지로 끼워 맞추면 이렇게 된다
{
  items.map((item) => (
    <li>
      <a
        href={item.url}
        style={item.url === "/buy" ? { border: "1px solid red" } : {}}
      >
        <img src={item.icon} />
        {item.label}
      </a>
    </li>
  ));
}
```

추상화가 오히려 변경을 막고 있다.

**핵심 통찰**: About과 Contact는 탐색 링크다. Buy는 행동을 유도하는 CTA(Call to Action)다. **겉모습이 같다고 같은 것이 아니다.**

---

### 더 나은 추상화

데이터 드리빙 대신, 재사용 가능한 컴포넌트로 추출한다.

```tsx
const MenuItem = ({ href, icon, style, children }) => (
  <li>
    <a href={href} style={style}>
      <img src={icon} />
      {children}
    </a>
  </li>
);

const NavigationMenu = () => (
  <ul>
    <MenuItem href="/about" icon="question-icon.png">
      About
    </MenuItem>
    <MenuItem href="/contact" icon="person-icon.png">
      Contact
    </MenuItem>
    <MenuItem
      href="/buy"
      icon="cash-icon.png"
      style={{ border: "1px solid red" }}
    >
      Buy
    </MenuItem>
  </ul>
);
```

이제 Buy만 다르게 만드는 게 자연스럽다. 각 항목이 명시적으로 선언되어 있으니까.

**책임 분리:**

- `NavigationMenu` — 메뉴 전체 구조
- `MenuItem` — 개별 항목의 구조
- 각 `<MenuItem>` 호출부 — 실제 값과 스타일

---

### 📌 정리

`"반복처럼 보이는 것"이 "같은 것"은 아니다`

#### 핵심

> DRY의 목적은 코드를 짧게 만드는 게 아니라 변경 이유가 같은 코드를 하나로 묶는 것이다.
> 변경 이유가 다른 코드를 억지로 묶으면, 나중에 하나를 바꿀 때 다른 것도 영향을 받는다.

#### ⭐️ 앞으로 기억할 것

- 1. **DRY 전에 "왜 반복되는가"를 먼저 물어보기**
     같은 이유로 변경되는 반복이면 → 묶어도 된다.
     다른 이유로 변경될 수 있는 반복이면 → 분리해두는 게 낫다.

- 2. **데이터 드리빙이 항상 좋은 건 아니다**
     `items.map()`은 모든 항목을 동일하게 취급한다. 항목마다 다른 동작이 필요해지는 순간 추상화가 짐이 된다.

- 3. **컴포넌트 추출 = 더 나은 DRY**
     반복 로직을 배열+맵으로 없애는 것보다, 재사용 가능한 컴포넌트를 만들고 각 호출부에서 다름을 표현하는 게 유연하다.

```tsx
// ❌ 모든 항목을 똑같이 처리
items.map(item => <li>...</li>)

// ✅ 공통 구조는 컴포넌트로, 다름은 props로
<MenuItem href="/buy" style={{ border: "1px solid red" }}>Buy</MenuItem>
```

- 4. **WET(Write Everything Twice) 원칙도 있다**
     두 번 이상 반복되기 전까지는 추상화하지 않는다. 세 번째 반복이 보일 때 진짜 공통점이 드러난다.

> **📌 결론**
> ▎ 지금까지 본 최악의 코드 중 상당수는 DRY를 너무 열심히 적용한 코드였다
> ▎ 코드 길이를 줄이는 게 목적이 아니라, 변경 이유를 기준으로 묶어야 한다
> ▎ ⭐️ "이 두 코드가 같아 보이는가"가 아니라 "이 두 코드가 같은 이유로 바뀌는가"를 물어보자
