# 현실 프론트엔드에서 쓰는 자바스크립트 자료구조 (React 예시)

> 원문: [JavaScript Data Structures in real Frontend code](https://profy.dev/article/javascript-data-structures) — Johannes Kettmann (profy.dev)
> 번역/정리 출처: [번역글 — eunbinn](https://velog.io/@eunbinn/javascript-data-structures)

![](https://velog.velcdn.com/images/chromeheartz/post/e3075256-5fb4-4846-9c9e-557094779b60/image.png)

자료구조 공부하면 늘 `foo`, `bar`에 추상적인 다이어그램만 나와서 "이걸 프론트에서 진짜 쓰나?" 싶었다. 이 글은 그 반대로, **실제 React 코드에서 자료구조를 잘못 고르면 생기는 문제**를 보여주고 Map / Set / Stack / Queue / Tree를 실전 예제로 풀어준다. 화려한 커스텀 구현 없이 JS 내장(객체, 배열, Map, Set)만으로 충분하다는 게 포인트.

### 프론트엔드 개발자도 자료구조를 알아야 할까

흔히 "프론트는 데이터셋이 작아서 고급 자료구조가 필요 없다"고 한다. 실제로:
- 프론트 데이터셋은 대개 너무 작아서 고급 구조의 이득이 작다
- JS는 많은 자료구조를 내장으로 지원하지 않는다
- React의 불변성 요구가 커스텀 구조의 성능 이점을 깎아먹기도 한다

그럼에도 **"잘못된 자료구조 선택"은 코드 문제의 주요 원인**이다. 성능보다도 코드 품질·가독성·유지보수 측면에서 알 가치가 있다(면접 대비는 덤).

---

## 1. Map (키-값 저장)

JS엔 일반 객체와 내장 `Map` 둘 다 있다. 일반 객체:

```javascript
{
  key1: "value 1",
  key2: "value 2",
  key3: "value 3",
}
```

내장 `Map`의 이점: `Map.size`, 최적화된 키-값 조회, **어떤 타입이든 키로 사용 가능**(객체도 OK). 일반 객체는 키를 문자열로 강제 변환한다.

```javascript
const numericKeys = {
  1: "value-1",
  2: "value-2",
};

console.log(Object.keys(numericKeys));
// ["1", "2"]  ← 숫자 1, 2가 문자열로 바뀜
```

### 실전: 채팅 메시지에 작성자 이름 붙이기

메시지 배열과 유저 배열이 따로 있고, 메시지마다 `userId`가 있다. 순진하게 가면 메시지마다 유저 배열을 `find`로 전부 훑는다 → **O(n×m)**.

```javascript
// ❌ 메시지마다 users 전체를 검색
messages.map(({ id, text, userId }) => {
  const name = users.find((user) => user.id === userId).name;
  return ... // JSX
})
```

**해법 A — 일반 객체로 id→name 맵 만들기**

```javascript
const messages = [
  { id: "message-1", text: "Hey folks!", userId: "user-1" },
  // ...
];

const users = [
  { id: "user-1", name: "Paul" },
  // ...
];

function ChatUsingMap() {
  const namesById = users.reduce(
    (prev, user) => ({ ...prev, [user.id]: user.name }),
    {}
  );

  return messages.map(({ id, text, userId }) => (
    <div key={id}>
      <div>{text}</div>
      <div>{namesById[userId]}</div>
    </div>
  ));
}
```

**해법 B — 내장 Map**

```javascript
function ChatUsingMap() {
  const namesById = new Map(users.map(({ id, name }) => [id, name]));

  return messages.map(({ id, text, userId }) => (
    <div key={id}>
      <div>{text}</div>
      <div>{namesById.get(userId)}</div>
    </div>
  ));
}
```

`Map` 생성자는 `[키, 값]` 쌍의 배열을 받는다:

```javascript
[
  ["user-1", "Paul"],
  ["user-2", "Lisa"],
  // ...
]
```

> 한 번 맵을 만들어두면 조회는 O(1). find 반복(O(n))이 사라진다.

## 2. Set (중복 없는 집합)

모든 값이 유일한 내장 컬렉션. 배열 중복 제거에 흔히 쓴다.

```javascript
const uniqueValues = [...new Set([1, 1, 2, 3, 3])];
// [1, 2, 3]
```

배열 대비 핵심 이점: **포함 여부(has) 체크가 빠르다.**

### 실전: 선택된 행 추적

표에서 여러 행을 체크박스로 선택. index 기반 배열로 관리하면:

```javascript
// ❌ index에 의존 → 행 정렬/순서 바뀌면 깨짐
const [selectedRows, setSelectedRows] = useState(
  rows.map((row) => ({ selected: false }))
);

const handleOnChange = (index) => {
  const updatedSelectedRows = [...selectedRows];
  updatedSelectedRows[index] = !updatedSelectedRows[index];
  setSelectedRows(updatedSelectedRows);
};
```

행 순서가 바뀌면 인덱스와 선택 상태가 어긋난다. **id를 Set에 담으면** 순서와 무관해진다.

```javascript
import { useState } from "react";

const rows = [
  { id: "id-1", name: "Row 1" },
  { id: "id-2", name: "Row 2" },
  // ...
];

function TableUsingSet() {
  const [selectedIds, setSelectedIds] = useState(new Set());

  const handleOnChange = (id) => {
    const updatedIdToSelected = new Set(selectedIds); // 불변성: 새 Set 복제
    if (updatedIdToSelected.has(id)) {
      updatedIdToSelected.delete(id);
    } else {
      updatedIdToSelected.add(id);
    }
    setSelectedIds(updatedIdToSelected);
  };

  return (
    <table>
      <tbody>
        {rows.map(({ id, name }) => (
          <tr key={id}>
            <td>
              <input
                type="checkbox"
                checked={selectedIds.has(id)}
                onChange={() => handleOnChange(id)}
              />
            </td>
            <td>{id}</td>
            <td>{name}</td>
          </tr>
        ))}
      </tbody>
    </table>
  );
}
```

## 3. Stack (LIFO, 후입선출)

맨 위에서 넣고 빼는 구조.

```javascript
const stack = [];

stack.push("value");        // 맨 위에 추가
const topItem = stack.pop(); // 맨 위에서 제거
```

### 실전: Undo(되돌리기)

행 삭제를 되돌린다. 삭제할 때 히스토리 스택에 액션+행을 push, undo 누르면 pop해서 복원.

> React 불변성 때문에 `push` 대신 `Array.concat()`, `pop` 대신 `Array.slice()`를 쓴다.

```javascript
import { useReducer } from "react";

const rows = [
  { id: "id-1", name: "Row 1" },
  { id: "id-2", name: "Row 2" },
  // ...
];

const initialState = { rows, history: [] };

function reducer(state, action) {
  switch (action.type) {
    case "remove":
      return {
        rows: removeRow(state, action),
        history: state.history.concat({   // push 대신 concat
          action,
          row: state.rows[action.index],
        }),
      };
    case "undo":
      return {
        rows: addRowAtOriginalIndex(state),
        history: state.history.slice(0, -1), // pop 대신 slice
      };
    default:
      throw new Error();
  }
}

function TableUsingStack() {
  const [state, dispatch] = useReducer(reducer, initialState);

  return (
    <>
      <button
        onClick={() => dispatch({ type: "undo" })}
        disabled={state.history.length === 0}
      >
        Undo Last Action
      </button>

      <table>
        <thead>
          <tr><th>ID</th><th>Name</th><th></th></tr>
        </thead>
        <tbody>
          {state.rows.map(({ id, name }, index) => (
            <tr key={id}>
              <td>{id}</td>
              <td>{name}</td>
              <td>
                <button onClick={() => dispatch({ type: "remove", id, index })}>
                  Delete
                </button>
              </td>
            </tr>
          ))}
        </tbody>
      </table>
    </>
  );
}
```

## 4. Queue (FIFO, 선입선출)

뒤에서 넣고 앞에서 빼는 구조.

```javascript
const queue = [];

queue.push("value");          // 뒤에 추가
const firstItem = queue.shift(); // 앞에서 제거
```

### 실전: 잠깐 떴다 사라지는 알림

알림을 큐에 추가하고, 일정 시간 뒤 가장 오래된 것을 제거.

```javascript
import { faker } from "@faker-js/faker";
import { useState } from "react";

function ButtonAddingNotifications() {
  const [notifications, setNotifications] = useState([]);

  const addNotification = () => {
    setNotifications((previous) => {
      return previous.concat(`${faker.name.firstName()} joined!`); // 뒤에 추가
    });
    setTimeout(() => {
      setNotifications((previous) => previous.slice(1)); // 앞에서 제거
    }, 1000);
  };

  return (
    <>
      <button onClick={() => addNotification()}>Invite User To Community</button>
      <aside>
        {notifications.map((message, index) => (
          <p key={index}>{message}</p>
        ))}
      </aside>
    </>
  );
}
```

> 여기서도 불변성을 위해 `push` 대신 `concat`, `shift` 대신 `slice(1)`.

## 5. Tree (트리, 계층 구조)

부모-자식 관계의 중첩 구조. 자식이 또 자식을 가질 수 있어 보통 재귀로 다룬다.

**중첩 객체 표현:**

```javascript
{
  name: "Parent",
  children: [
    { name: "Child 1" },
    {
      name: "Child 2",
      children: [{ name: "Grandchild 21" }],
    },
  ],
}
```

**평탄화(flat) + id 참조 표현:**

```javascript
[
  { id: "parent-1", name: "Parent", children: ["child-1", "child-2"] },
  { id: "child-1", name: "Child 1", parent: "parent-1" },
  { id: "child-2", name: "Child 2", parent: "parent-1", children: ["grandchild-2"] },
  { id: "grandchild-21", name: "Grandchild 21", parent: "child-2" },
]
```

### 실전: 중첩 메뉴 재귀 렌더링

**해법 A — 중첩 객체**

```javascript
const menuItems = [
  {
    text: "Menu 1",
    children: [
      { text: "Menu 1 1", href: "#11" },
      { text: "Menu 1 2", href: "#12" },
    ],
  },
  { text: "Menu 2", href: "#2" },
  {
    text: "Menu 3",
    children: [
      {
        text: "Menu 3 1",
        children: [{ id: "311", text: "Menu 3 1 1", href: "#311" }],
      },
    ],
  },
];

function Menu({ items }) {
  return (
    <ul>
      {items.map((item, index) => (
        <MenuItem key={index} {...item} />
      ))}
    </ul>
  );
}

function MenuItem({ text, href, children }) {
  if (!children) {
    return (
      <li>
        <a href={href}>{text}</a>
      </li>
    );
  }
  return (
    <li>
      {text}
      <Menu items={children} /> {/* Menu ↔ MenuItem 상호 재귀 */}
    </li>
  );
}

function NestedMenu() {
  return <Menu items={menuItems} />;
}
```

> 단점: 데이터를 **업데이트**할 때 깊은 중첩 + 불변성 유지가 까다롭다.

**해법 B — 평탄화 + id 참조 (업데이트가 쉬움)**

```javascript
const menuItems = [
  { id: "1", text: "Menu 1", children: ["11", "12"], isRoot: true },
  { id: "11", text: "Menu 1 1", href: "#11" },
  { id: "12", text: "Menu 1 2", href: "#12" },
  { id: "2", text: "Menu 2", href: "#2", isRoot: true },
  { id: "3", text: "Menu 3", children: ["31"], isRoot: true },
  { id: "31", text: "Menu 3 1", children: ["311"] },
  { id: "311", text: "Menu 3 1 1", href: "#311" },
];

function Menu({ itemIds, itemsById }) {
  return (
    <ul>
      {itemIds.map((id) => (
        <MenuItem key={id} itemId={id} itemsById={itemsById} />
      ))}
    </ul>
  );
}

function MenuItem({ itemId, itemsById }) {
  const item = itemsById[itemId];
  if (!item.children) {
    return (
      <li>
        <a href={item.href}>{item.text}</a>
      </li>
    );
  }
  return (
    <li>
      {item.text}
      <Menu itemIds={item.children} itemsById={itemsById} />
    </li>
  );
}

function NestedMenu() {
  const itemsById = menuItems.reduce(
    (prev, item) => ({ ...prev, [item.id]: item }),
    {}
  );
  const rootIds = menuItems.filter(({ isRoot }) => isRoot).map(({ id }) => id);
  return <Menu itemIds={rootIds} itemsById={itemsById} />;
}
```

> 재귀 깊이가 10,000 단계를 넘으면 콜스택 한도를 초과할 수 있다. 하지만 일반적인 UI 계층에선 거의 마주칠 일 없다.

### 📌 정리
`자료구조는 성능보다 "올바른 선택 = 깔끔한 코드"가 핵심`

#### 핵심
> 프론트에서도 자료구조는 쓸모 있다. 커스텀 구현이 아니라 JS 내장(객체/배열/Map/Set)을 상황에 맞게 고르는 게 핵심. 반복 조회는 Map, 유일성·포함체크는 Set, 되돌리기는 Stack, 순차 처리(알림)는 Queue, 계층 UI는 Tree(업데이트가 잦으면 평탄화+id 참조). React에선 불변성 때문에 push/pop/shift 대신 concat/slice를 쓴다.

#### ⭐️ 앞으로 기억할 것
- 1. `find` 반복이 보이면 Map으로
  배열 안에서 매번 find하면 O(n×m). id→값 맵을 한 번 만들면 조회 O(1).
  ```js
  const namesById = new Map(users.map(({id, name}) => [id, name]));
  namesById.get(userId);
  ```
- 2. 선택/포함 상태는 index 배열 말고 id Set
  index 기반은 정렬·순서 변경에 깨진다. id를 Set에 담으면 순서 무관 + `has` 빠름.
- 3. React 불변성: push/pop/shift 금지
  스택/큐를 흉내 낼 때 `concat`(추가), `slice(0,-1)`(pop), `slice(1)`(shift)로 새 배열 반환.
  ```js
  history.concat(item);   // push
  history.slice(0, -1);   // pop
  notis.slice(1);         // shift
  ```
- 4. Undo는 히스토리 스택 + useReducer
  액션과 직전 상태를 스택에 쌓고, undo 때 pop해서 복원. reducer에 모으면 깔끔.
- 5. 트리 업데이트가 잦으면 평탄화 + id 참조
  중첩 객체는 렌더는 쉬워도 깊은 불변 업데이트가 지옥. `itemsById` + `children: [id...]`로 펴면 수정이 쉬워진다.

📌 결론
▎ 커스텀 자료구조 구현이 아니라 "내장 도구를 맞게 고르는 능력"이 프론트에서의 자료구조 실력.
▎ Map=빠른 조회 / Set=유일성·포함 / Stack=undo / Queue=순차 알림 / Tree=계층 UI.
▎ React면 항상 불변성: concat/slice, new Set(prev) 복제.
▎ ⭐️ 실무 적용 포인트
  - 코드에서 중첩 `find`/`includes`가 반복되면 Map/Set 전환 신호.
  - 선택·즐겨찾기·체크 상태는 id 기반 Set이 거의 정답.
  - 메뉴/댓글/카테고리 같은 계층 데이터는 편집 빈도 보고 중첩 vs 평탄화 선택.
