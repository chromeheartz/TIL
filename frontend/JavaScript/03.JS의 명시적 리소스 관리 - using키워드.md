# JS의 명시적 리소스 관리 — `using` 키워드

  파일, 락, DB 연결 같은 리소스 정리를 깜빡한 적 있지 않나? 사실 `try/finally`로 감싸는 게 맞는 건데, 이게 중첩되기 시작하면 진짜 지저분해진다. JS에
   드디어 이걸 언어 차원에서 깔끔하게 해결하는 문법이 생겼다.

  ---

  ### 기존 방식의 문제

  `try/finally` 패턴 자체가 틀린 건 아닌데, 현실에서는 이렇게 된다:

  ```js
  const file = await openFile("data.txt");
  const lock = await acquireLock();

  try {
    // 작업
  } finally {
    await lock.release(); // 순서 틀리면 버그
    await file.close();
  }
  ```

  리소스가 늘어날수록 정리 순서, 에러 경로, 보일러플레이트가 같이 늘어난다. 솔직히 피곤한 패턴이다.

  ---

  ### `using` — 수명 기반 정리

  ```js
  await using file = await openFile("data.txt");

  // 작업

  // 스코프 끝나면 알아서 file.close() 호출됨
  ```

  핵심은 **정리가 제어 흐름이 아니라 변수의 수명에 묶인다**는 것. 스코프를 벗어나는 순간 런타임이 알아서 처리한다.

  #### 동기 vs 비동기

  | 키워드 | 심볼 | 언제 |
  |--------|------|------|
  | `using` | `Symbol.dispose` | 정리가 동기일 때 |
  | `await using` | `Symbol.asyncDispose` | 정리가 비동기일 때 |

  ---

  ### 리소스에 `[Symbol.asyncDispose]` 달아주기

  마법이 아니다. 라이브러리/클래스가 이 심볼을 구현해야 `using`이 동작한다:

  ```js
  class FileHandle {
    async write(data) { /* ... */ }

    async [Symbol.asyncDispose]() {
      await this.close(); // 이게 스코프 끝날 때 호출됨
    }
  }
  ```

  `using`은 각 라이브러리의 정리 방식을 **표준화**하는 것이지, 파일을 알아서 닫아주는 마법이 아니다.

  ---

  ### 여러 리소스 쌓기 — 역순 정리 보장

  ```js
  // ❌ 기존
  const file = await openFile("data.txt");
  const lock = await acquireLock();
  try {
    // 작업
  } finally {
    await lock.release();
    await file.close();
  }

  // ✅ 신규
  await using file = await openFile("data.txt");
  using lock = await acquireLock();
  // 작업
  // → lock 해제 후 file 닫힘 (선언 역순으로 자동 정리)
  ```

  에러가 나도 정리가 중단되지 않고, 순서도 명확하게 정의된다.

  ---

  ### 스코프로 수명을 명시

  ```js
  {
    await using file = await openFile("data.txt");
    // file 유효
  }
  // 여기서 정리됨
  ```

  `const`/`let`처럼 블록 스코프를 따른다. 자연스럽게 리소스 수명을 코드에 드러내게 된다.

  ---

  ### 블록에 안 맞는 케이스 — `AsyncDisposableStack`

  리소스가 블록 구조에 깔끔하게 들어맞지 않는 경우:

  ```js
  const stack = new AsyncDisposableStack();

  const file = stack.use(await openFile("data.txt"));
  const lock = stack.use(await acquireLock());

  // 작업

  await stack.disposeAsync(); // 한 번에 역순 정리
  ```

  `using`과 동일한 안전성을 유지하면서 더 유연하게 쓸 수 있다.

  ---

  ### 프론트엔드에서도 쓸 수 있다

  백엔드 전용이 아니다. 다음 케이스에도 적용 가능:

  - Web Streams 정리
  - `navigator.locks` 해제
  - 옵저버/구독 해제
  - IndexedDB 트랜잭션

  `subscribe()`/`unsubscribe()` 쌍을 손으로 관리해봤다면 이게 얼마나 반가운지 알 것이다.

  ---

  ### 지원 현황 (2026년 초 기준)

  - Chrome 123+, Firefox 119+, Node.js 20.9+ ✅
  - Safari — 아직 미지원, 로드맵엔 있음 ⚠️

  지금 당장 전면 도입은 부담스럽고, 라이브러리나 플랫폼 추상화 레이어 설계할 때 API 차원에서 고려하는 게 현실적이다.

  ---

  ### 📌 정리
  `using/await using으로 리소스 수명을 변수에 묶는다`

  #### 핵심
  > `try/finally`의 보일러플레이트 없이, 스코프 기반으로 리소스 정리를 자동화하는 문법. 정리 순서는 선언 역순으로 보장되며, 에러 경로에서도
  안전하다.

  #### ⭐️ 앞으로 기억할 것
  - 1. `Symbol.dispose` / `Symbol.asyncDispose` 직접 구현
    라이브러리 만들거나 클래스 설계할 때, 정리 로직이 있으면 이 심볼부터 달아두자.
    ```js
    class MyResource {
      [Symbol.dispose]() {
        // 동기 정리
      }
    }
    ```

  - 2. 여러 리소스 선언 순서 = 정리 역순
    선언 순서가 곧 정리 순서의 반대. `lock`을 먼저 해제해야 한다면 `file` 다음에 선언하면 된다.
    ```js
    await using file = openFile();
    using lock = acquireLock();
    // 정리: lock → file
    ```

  - 3. 블록 구조에 안 맞으면 `DisposableStack`
    클래스 메서드나 조건부 리소스 등 블록으로 못 감싸는 케이스에서 꺼내는 카드.

  📌 결론
  ▎ `using`은 `try/finally`를 대체하는 게 아니라 더 나은 기본값을 제공한다
  ▎ 핵심은 "정리를 잊지 않는 것"이 아니라 "잊을 필요가 없게 만드는 것"
  ▎ ⭐️ Safari 미지원이 발목을 잡지만, 라이브러리 레벨 API 설계엔 지금부터 반영할 만하다
  ▎ 프론트에서도 구독 해제, Web Streams, IndexedDB 트랜잭션에 바로 적용 가능
