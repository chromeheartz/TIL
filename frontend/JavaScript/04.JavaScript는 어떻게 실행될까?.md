# JavaScript는 어떻게 실행될까?

![](https://velog.velcdn.com/images/chromeheartz/post/831bcb4f-aaf6-458a-8379-72eda83905a6/image.png)

자바스크립트가 "인터프리터 언어"라는 말은 많이 들었는데, 실제로 브라우저
안에서 어떤 과정을 거쳐 실행되는지 제대로 몰랐다. 이 글 읽고 나서야 V8 같은
JS 엔진이 그냥 코드 한 줄씩 읽는 게 아니라는 걸 알았다.

---

### JS 실행, 생각보다 꽤 복잡하다

웹 브라우저는 자바스크립트 엔진을 내장하고 있다. 크롬이면 V8, 파이어폭스면
SpiderMonkey. 그리고 코드를 실행할 때 단순히 "위에서 아래로 읽는다"는 표현은
반만 맞는 말이다.

실제론 총 5단계를 거친다.

---

### 📌 1단계: 소스코드 파싱 → AST 생성

엔진이 가장 먼저 하는 일은 우리가 쓴 소스코드를 **파싱**하는 것이다.
파싱 결과물이 바로 `AST (Abstract Syntax Tree)` — 추상 구문 트리.

```js
const x = 1 + 2;
```

이 한 줄도 엔진 눈에는 그냥 텍스트가 아니라 트리 구조로 분해된다.
`VariableDeclaration` → `BinaryExpression` → `Literal(1)`, `Literal(2)` 이런
식으로.

![AST 구조 이미지](https://velog.velcdn.com/images/hihijin/post/dbf57592-2c2d
-4035-8d44-7d7ca2816997/image.png)

---

### 📌 2단계: 인터프리터 → 바이트코드 생성

AST가 만들어지면 **인터프리터**가 이걸 기반으로 `바이트코드(bytecode)`를
생성한다.

바이트코드는 기계어도 아니고 원본 JS도 아닌 **중간 단계 코드**다.
CPU가 바로 이해할 수 있는 건 아니지만, 원본 텍스트보다는 훨씬 실행에 가깝다.

자바스크립트가 "인터프리터 언어"라고 불리는 이유가 여기 있다.
컴파일러 언어처럼 미리 전부 기계어로 변환하지 않고, 런타임에 단계적으로
처리한다.

> 참고로 Java도 바이트코드를 쓰는데, JS 바이트코드는 JVM 바이트코드와는
다르다. 각 엔진마다 포맷이 다름.

---

### 📌 3단계: 프로파일링 데이터 수집

바이트코드를 실행하면서 엔진은 조용히 데이터를 모은다.

- 어떤 함수가 자주 호출되는가?
- 특정 변수에 어떤 타입이 주로 들어오는가?

이 정보가 **프로파일링 데이터**다.
"이 함수는 항상 숫자만 들어오네?" 하면 최적화 컴파일러한테 넘긴다.

---

### 📌 4단계: 최적화 컴파일러 → 최적화된 기계어 생성

수집된 프로파일링 데이터를 바탕으로 **최적화 컴파일러**가 빠른 기계어를
만든다.

V8에서는 이 역할을 `TurboFan`이 한다.
자주 쓰이는 코드(= **핫 코드**) 를 기계어로 변환해서 바이트코드보다 훨씬
빠르게 실행한다.

이 과정 덕분에 JS가 인터프리터 언어임에도 불구하고 꽤 빠른 성능을 낼 수 있는
거다.

```js
// 이런 함수가 수천 번 호출되면 → TurboFan이 최적화
function add(a, b) {
  return a + b;
}
```

---

### 📌 5단계: 실행 & 역최적화(Deoptimization)

최적화된 코드가 실행되다가 문제가 생기면 어떻게 될까?

예를 들어 3단계에서 "이 변수엔 항상 숫자가 들어온다"고 가정하고 최적화했는데,

갑자기 문자열이 들어오면 그 가정이 깨진다.

이때 엔진은 **역최적화(Deoptimization)** 를 수행한다.
최적화된 기계어를 버리고, 다시 2단계 바이트코드로 돌아가서 재실행.

```js
function add(a, b) {
  return a + b;
}

add(1, 2);       // 수천 번 → 숫자용으로 최적화됨
add("hello", 2); // 갑자기 문자열 → 역최적화 발생
```

그래서 타입을 일관성 있게 쓰는 게 성능에 실제로 영향을 준다.

---

### 📌 정리

`JS 엔진은 파싱 → 바이트코드 → 최적화 → 실행의 파이프라인을 돈다`

#### 핵심
> 자바스크립트는 단순한 인터프리터가 아니라, 인터프리터 + JIT 컴파일러의
조합으로 동작한다.
> 소스코드는 AST → 바이트코드 → 최적화 기계어 순서로 변환되며,
> 가정이 깨지면 역최적화가 일어나 다시 바이트코드로 돌아간다.

#### ⭐️ 앞으로 기억할 것

- 1. **타입 일관성이 성능이다**
  최적화 컴파일러는 타입을 기반으로 최적화한다. 같은 함수에 숫자/문자열을
섞어 넣으면 역최적화가 일어나서 오히려 느려질 수 있다.
  ```js
  // ❌ 타입이 섞이면 역최적화
  process(1);
  process("hello");

  // ✅ 타입 일관성 유지
  processNumber(1);
  processString("hello");
  ```

- 2. **AST는 코드 분석 도구의 기반**
  Babel, ESLint, Prettier 같은 툴이 전부 AST를 기반으로 동작한다. 코드를
"변환"한다는 게 결국 AST를 조작하는 것.

- 3. **JIT(Just-In-Time) 컴파일이 JS를 빠르게 만든다**
  초기 JS는 순수 인터프리터라 느렸다. V8이 JIT를 도입하면서 성능이 크게
올라갔다. "JS는 느리다"는 말은 이제 옛날 얘기.

- 4. **핫 코드(Hot Code)를 의식하자**
  루프 안이나 자주 호출되는 함수는 최적화 대상이 된다. 여기서 타입이 흔들리면
  성능 저하가 크다.

📌 결론
▎ JS 실행은 AST 파싱 → 바이트코드 → 프로파일링 → 최적화 기계어 → 실행의
5단계다
▎ 최적화는 "가정"을 기반으로 하기 때문에, 가정이 깨지면 역최적화가 발생한다
▎ ⭐️ 타입을 일관되게 쓰는 것 자체가 JS 성능 최적화의 첫 걸음
