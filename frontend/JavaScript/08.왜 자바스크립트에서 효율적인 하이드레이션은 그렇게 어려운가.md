# 왜 자바스크립트에서 효율적인 하이드레이션은 그렇게 어려운가

> **원문**: https://dev.to/this-is-learning/why-efficient-hydration-in-javascript-frameworks-is-so-challenging-1ca3
> **번역 출처**: https://velog.io/@eunbinn/%EB%B2%88%EC%97%AD-why-efficient-hydration-in-javascript-is-so-challenging

![post-thumbnail](https://velog.velcdn.com/images/eunbinn/post/f87c7853-7881-47bd-973a-13db51818f49/lqisgld0xxx7djak3367.jpeg)

SSR이 만능인 줄 알았는데, 사실 하이드레이션이 붙는 순간 얘기가 완전히 달라진다. "서버에서 미리 그려주면 빠르겠지" 하고 끝나는 게 아니라, 오히려 번들이 커지고 인터랙션이 늦어질 수도 있다. 이 글은 *왜* 효율적인 하이드레이션이 그렇게 어려운지, 그리고 프레임워크들이 이걸 어떻게 깨보려고 발버둥치는지를 쭉 훑는다. 읽으면서 "아, 이래서 Astro/Qwik/RSC 같은 게 나왔구나"가 한 번에 연결됐다.

### 서버 렌더링은 완성된 해법이 아니다

솔직히 이 부분이 제일 충격이었다. SSR을 하면 프레임워크는 **서버 렌더링 코드 + 클라이언트 동작 코드**를 둘 다 실어 보내야 한다. 그래서 순수 CSR보다 자바스크립트 번들이 오히려 더 커진다.

HTML은 빨리 그려진다. 근데 그게 "보이는 것"일 뿐이고, 실제로 클릭이 먹으려면 JS를 받아서 하이드레이션이 끝나야 한다. 그 사이 구간이 문제다.

> 화면은 다 그려졌는데 버튼을 눌러도 아무 반응이 없는 그 어색한 시간 — 이게 하이드레이션이 만들어내는 함정이다.

#### 근본 원인: 데이터와 템플릿을 두 번 보낸다

왜 이렇게 비효율적일까? 핵심은 **중복 전송**이다.

- **템플릿**: 번들된 JS 안에도 있고, 서버가 그려준 HTML 안에도 있다
- **데이터**: `<script>` 태그(직렬화된 state)에도 있고, 최종 HTML 마크업에도 박혀 있다

```html
<!-- 같은 데이터가 마크업으로 한 번... -->
<h1>Hello, Bibi</h1>

<!-- 직렬화된 state로 또 한 번 -->
<script>window.__INITIAL_STATE__ = { name: "Bibi" }</script>
```

전부 HTML로 한 번, JS로 한 번. 대역폭도 낭비고, 서버/클라 두 표현을 동기화하는 것도 복잡하다. 효율적인 하이드레이션 = "이 중복을 어떻게 줄이느냐"의 싸움이라고 보면 된다.

---

### 깨보려는 시도들

#### ① 정적 라우팅 (하이드레이션 없음)

가장 단순한 발상. 페이지가 완전히 정적이면 JS가 필요 없으니 `<script>` 태그를 아예 빼버린다. Remix, SvelteKit, SolidStart 같은 프레임워크가 정적 페이지에서 이걸 한다.

- ✅ 정적인 페이지는 JS 0KB
- ❌ 동적 요소 하나만 들어가도 무너진다 → 결국 프레임워크 전체를 끌고 와야 함

세밀하게 "이 부분만 인터랙티브"가 안 된다. all or nothing에 가깝다.

#### ② JS 지연 로딩 (Progressive Hydration)

"필요할 때 로드하면 되잖아?" 인터랙션 시점까지 스크립트 로딩을 미루는 전략.

근데 대부분의 프레임워크는 **공통 루트에서 top-down으로 하이드레이션**한다. 그래서 페이지 한가운데 버튼을 누르면, 그 버튼만 살아나는 게 아니라 **위에 있는 조상 컴포넌트 코드까지 다 끌려온다.**

```
App (root)
└─ Layout
   └─ Section
      └─ Button  ← 이거 하나 누르려고
                    위 전부를 하이드레이트해야 함
```

> 지연 로딩의 취지가 무색해진다. 정작 유저가 누르고 싶을 때 "기다리세요"가 되니까. 체감 성능이 오히려 나빠질 수 있다.

#### ③ HTML에서 state 거꾸로 뽑아내기

"이미 HTML에 데이터가 다 있는데, 거기서 state를 역추출하면 두 번 안 보내도 되잖아?" — 좋은 아이디어처럼 보이는데 **안 된다.**

이유: **모델과 뷰가 항상 1:1로 대응하지 않는다.**

- 타임스탬프 `1717459200`을 화면엔 `"3시간 전"`으로 그렸다면 → HTML만 보고 원래 숫자를 복원 불가
- 표시용으로 만들어낸(made-up) 데이터는 원본을 되살릴 수 없다
- UI가 바뀌면서 데이터 표현이 달라지면 원래 state는 마크업만으로 복구 불가능

```js
// model → view 는 되는데
const view = formatRelative(timestamp); // "3시간 전"

// view → model 은 안 됨 (정보 손실)
const timestamp = parse("3시간 전"); // ❌ 원본 복원 불가
```

#### ④ 아일랜드 (Partial Hydration) — Marko, Astro

발상의 전환. **HTML 대부분을 정적으로 취급하고, 인터랙티브한 "섬(island)"만 하이드레이트**한다.

- 섬 바깥 영역은 JS를 아예 안 보냄 → 전송량 급감
- 각 섬은 독립적이라 top-down 끌려오기 문제도 없음

```astro
---
// Astro: 정적 HTML 안에 인터랙티브 섬만 심는다
---
<StaticHeader />
<Counter client:load />   <!-- 이 컴포넌트만 하이드레이트 -->
<StaticFooter />
```

- ✅ 콘텐츠 위주 사이트(블로그, 문서, 마케팅)에 강력
- ❌ 컴포넌트 트리가 얕을 때 잘 맞고, 보통 네비게이션 시 풀 페이지 리로드

#### ⑤ Resumable / Reverse Hydration — Qwik

top-down을 뒤집는다. **부모 state나 라우팅 로직을 로드하지 않고도 특정 컴포넌트를 독립적으로 살릴 수 있다.**

각 컴포넌트가 필요한 의존성을 주입(injection)받는 구조. 대신 컴포넌트로 들어가는 모든 입력을 빠짐없이 직렬화(serialize)해 둬야 한다. 그 대가로 깊은 트리 순회 없이 "재개(resume)"가 된다.

> 일반 프레임워크가 "처음부터 다시 실행(replay)"한다면, Qwik은 "멈춘 지점부터 이어서(resume)"한다. 그래서 *resumable*.

- ✅ 중간 버튼만 정확히 살릴 수 있음
- ❌ 전체 직렬화 비용 + 컴포넌트 관계에 제약

#### ⑥ 서버 컴포넌트 — React Server Components

정적인 부분은 서버에서 렌더링한 채로 두고, **네비게이션을 넘어서도 클라이언트 state는 유지**한다. SPA의 장점은 살리면서 진짜 동적인 부분에만 JS를 쓴다.

- ✅ 정말 동적인 곳에만 JS → 번들 절약
- ❌ 브라우저에 더 큰 런타임이 필요하고, 빌드 오케스트레이션이 복잡함

### 📌 정리
`하이드레이션 = "두 번 보낸 걸 어떻게 한 번으로 줄이냐"의 전쟁`

#### 핵심
> SSR은 공짜가 아니다. 서버 코드 + 클라 코드를 둘 다 실어야 해서 번들이 커지고, HTML은 빨리 그려지지만 JS 하이드레이션이 끝나기 전까지는 "보이지만 안 눌리는" 페이지가 된다. 근본 원인은 **데이터·템플릿을 HTML로 한 번, JS로 한 번 — 두 번 보내는 중복**이다. 프레임워크들의 모든 시도(아일랜드, resumable, 서버 컴포넌트)는 결국 이 중복과 top-down 하이드레이션을 어떻게 깨느냐로 수렴한다.

#### ⭐️ 앞으로 기억할 것
- 1. **하이드레이션의 본질은 "중복 전송"**
  템플릿도 데이터도 HTML·JS 양쪽에 들어간다. 최적화 = 이 중복 줄이기.
- 2. **top-down 하이드레이션이 progressive loading의 발목을 잡는다**
  가운데 버튼 하나 누르려고 조상 트리 전체를 끌고 온다. 그래서 "지연 로딩"이 오히려 체감 성능을 깎을 수 있다.
- 3. **HTML에서 state 역추출은 불가능 (model ↔ view 비대칭)**
  ```js
  formatRelative(ts) // model → view ✅
  parse("3시간 전")  // view → model ❌ 정보 손실
  ```
- 4. **아일랜드 = 정적 우선, 섬만 하이드레이트** (Astro/Marko)
  콘텐츠 사이트에 최적. `client:load` 같은 디렉티브로 섬을 지정.
- 5. **Qwik = resume(재개), 일반 프레임워크 = replay(재실행)**
  부모 없이도 특정 컴포넌트만 살림. 대신 입력 전체 직렬화 비용.
- 6. **RSC = 정적은 서버에, 동적에만 JS**, 네비게이션 넘어 state 유지. SPA 장점 + 번들 절약, 대신 런타임·빌드 복잡.

📌 결론
▎ SSR + 하이드레이션은 "빠른 첫 화면"과 "빠른 인터랙션"을 동시에 만족시키기가 구조적으로 어렵다.
▎ 정답은 아직 없고, 아일랜드 · resumable · 서버 컴포넌트를 **상황에 맞게 조합**하는 게 현재의 best.
▎ ⭐️ 실무 적용 포인트
▎ - 콘텐츠/마케팅/문서 → **Astro(아일랜드)**. JS 거의 0에서 시작.
▎ - 인터랙션 풍부한 앱에서 첫 클릭 지연이 문제 → **Qwik(resumable)** 또는 **RSC** 검토.
▎ - "SSR 켰는데 왜 더 느리지?" 싶으면 → 번들에 서버+클라 코드가 둘 다 실리고 있는지, top-down 하이드레이션으로 큰 트리를 통째로 살리고 있는지부터 의심.
